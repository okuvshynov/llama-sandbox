## 🐞 Soundness Issues (Correctness & Safety)

### 1. Inconsistent State in `Metric::trim_older_than` (High Severity)
The method `trim_older_than` attempts to prune old data points. However, the logic for finding the cutoff point is flawed and potentially incorrect under concurrency.

*   **The issue:**
    ```rust
    let cutoff = self.timestamps.iter()
        .position(|t| now.duration_since(*t) < window);
    ```
    This code finds the *first* timestamp that is *within* the window. If the vector is chronologically ordered (which it should be, since data is appended via `record`), this correctly identifies the start index of valid data. However, if the vector were *not* guaranteed to be sorted or if multiple threads called `trim_older_than` concurrently (which they won't if they hold the lock, but imagine future complexity), this could lead to unpredictable behavior.
*   **The biggest flaw:** The method relies on the assumption that `self.timestamps` is always chronologically sorted from oldest to newest. While `Metric::record` enforces this by pushing to the end, relying on sequence order for truncation can sometimes be brittle. If the metric were ever updated or re-indexed, this would break.
*   **Recommendation:** While functional in the current setup, if performance were critical, you might want to use binary search (`binary_search_by`) if the data became massive, although for simple draining, `position()` is fine given the current invariant.

### 2. Potential Panic in `Metric::percentile` (Medium Severity)
The percentile calculation relies on `partial_cmp` and `unwrap()`.

*   **The issue:**
    ```rust
    sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
    ```
    Floating-point numbers (`f64`) implement `Ord` only if they are guaranteed not to be `NaN`. If any value recorded into the metric (`value: f64` in `record`) is `NaN`, the `unwrap()` call will panic, crashing the application.
*   **Recommendation:** Instead of `.unwrap()`, you should handle the `None` case (which occurs if one of the values is `NaN`). A robust implementation might filter out `NaN` values before sorting, or handle the panic gracefully.

## 🐌 Performance Problems (Efficiency & Overhead)

### 1. Excessive Locking Overhead (Critical Performance Issue)
The structure uses a multi-layered locking mechanism: `RwLock<HashMap<...>>` -> `Arc<Mutex<Metric>>`. Every single `record` operation requires two levels of locking acquisition/release.

*   **`get_or_create`:** Requires a read lock (`RwLock::read().unwrap()`), followed by potentially escalating to a write lock (`RwLock::write().unwrap()`). This is a classic, but slow, pattern for lazy initialization in a concurrent map.
*   **`record`:** Requires acquiring the `RwLock` to get the `Arc`, then acquiring the `Mutex` inside the `Arc`.
*   **Implication:** For high-frequency recording (which is the primary use case of a metrics system), the synchronization cost of acquiring and releasing these locks will dominate the actual work (a simple `Vec::push`). This system will scale poorly under extreme load.

**Recommendation (Major Refactoring):**
Instead of holding the metric data behind a `Mutex`, consider using lock-free structures or partitioning/sharding the data.

*   **Option A: Fine-Grained Locking:** If you stick to the map structure, the `Metric` itself could be protected by something faster, but the overhead of managing many small mutexes remains.
*   **Option B: Concurrent Map:** Replace `HashMap` with a concurrent hash map (e.g., from the `dashmap` crate). This significantly reduces contention when multiple threads are accessing *different* metrics.
*   **Option C: Pre-allocation/Sharding:** If the metrics are known beforehand, pre-allocate the metrics and avoid the runtime `get_or_create` synchronization entirely.

### 2. Copying Data in `Metric::percentile` (Minor Performance Issue)
The percentile calculation creates a full clone of the values vector just to sort it.

```rust
let mut sorted = self.values.clone(); // Expensive clone
sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
```
If the `Metric` holds millions of data points, cloning the entire vector just to calculate a percentile will be extremely slow and memory intensive.

**Recommendation:**
If you need high-performance percentile/quantile calculations on streaming data, do not store all raw values. Use specialized streaming algorithms like T-Digest or DDSketch, which approximate quantiles with a small, fixed amount of memory, avoiding the need to store and sort the full history.

### 3. Redundant Computation in Snapshot (Minor Performance Issue)
In `MetricsRegistry::snapshot`, multiple derived statistics (mean, p50, p95, p99) are calculated from the raw `Metric` data.

*   **The issue:** `mean()` requires a full iteration/sum. `percentile()` requires a full clone and sort (if the underlying data is large).
*   **Recommendation:** If performance is critical, the `Metric` structure should maintain running statistics (e.g., `total_sum`, `count`) atomically, allowing `mean()` to be $O(1)$ instead of $O(N)$.

## 📐 API Design Concerns (Usability & Idioms)

### 1. Missing `Drop` Implementation for `MetricsRegistry` (Minor API)
While not strictly necessary, a robust registry might benefit from a `Drop` implementation to ensure all underlying threads or resources (if they existed) are gracefully shut down or flushed before the program exits.

### 2. Histogram Implementation is Non-Standard (Design/Usability)
The `Histogram` structure is a custom implementation. In production systems, you would typically rely on battle-tested, standardized libraries (like those that implement HdrHistogram), which are designed specifically to handle floating-point distributions and extreme counts with guaranteed statistical integrity.

*   **Issue with the current implementation:** The `quantile` logic looks complex and fragile, especially handling edge cases where bucket counts might be identical or where the loop terminates prematurely. Furthermore, the definition of what constitutes a "bucket" boundary vs. the final infinite bucket can lead to off-by-one errors in the quantile calculation.
*   **Recommendation:** Unless this is a learning exercise, use a mature library for histograms.

### 3. Mixing Time Granularity (Design Inconsistency)
The `rate` calculation is highly context-dependent and confusingly implemented:

```rust
fn rate(&self, window: Duration) -> f64 {
    let now = Instant::now();
    let count = self.timestamps.iter()
        .filter(|t| now.duration_since(**t) < window)
        .count();
    count as f64 / window.as_secs_f64()
}
```
This function calculates the *average rate over the window*. If you measure requests every 10ms, but calculate the rate over 1 hour, the result is mathematically correct (`count / 3600`), but it's often misleading in metrics.

Furthermore, the `snapshot` method hardcodes `rate_1m`, while the `Metric::rate` method takes a parameter. This mixing of hardcoding and parameterization can lead to confusion.

**Recommendation:**
If you want a true rate, the metric should track total count and total duration, and the rate calculation should happen externally or leverage more advanced sampling to prevent the metric storage from ballooning indefinitely.

### 4. Implicit Errors (Error Handling)
The use of `.unwrap()` throughout the codebase (on mutex and RwLock locks) assumes that the locks will never be poisoned.

*   **The issue:** A poisoned lock occurs if a thread holding a lock panics. If this happens, `.unwrap()` will cause *another* panic in the current thread, which is rarely desirable in critical infrastructure code.
*   **Recommendation:** Use `lock().expect("Metric registry mutex poisoned")
