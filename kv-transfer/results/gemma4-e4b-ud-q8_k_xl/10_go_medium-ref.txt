## ⚠️ Critical Bugs & Race Conditions

These issues could lead to incorrect state, crashes, or unpredictable behavior.

### 1. Race Condition in `WorkerPool.Submit` (Data Visibility)

When a job is submitted, you update the `wp.results` map *before* pushing the job onto the queue. This is generally fine, but the job *itself* is an instance being shared.

**The Problem:** The job structure passed to `Submit` is a value/pointer that is then manipulated. If the caller (the HTTP handler) retains a reference to the original `job` struct while the pool is processing it, and the pool modifies its internal copy via the channel, consistency can become an issue, though this is less severe than the next point.

More critically, the state updates (`job.Status = "queued"`, `job.CreatedAt = time.Now()`) happen *outside* of the lock protecting `wp.results`. While the lock is only used to write to `wp.results`, the fields on the `job` struct itself are mutable and will be updated by the worker later. It is best practice to ensure all initial state changes are atomic relative to the structure's lifecycle.

**The Fix (Design Improvement):** Ensure that the job object stored in `wp.results` is the *exact* job object that will be processed.

### 2. Failure in Retry Logic (Deadlock/Data Race during Requeue)

The most severe functional bug is in `WorkerPool.processJob` during the retry mechanism.

```go
// Inside processJob, if failure occurs and retries are available:
// ...
		time.Sleep(time.Duration(job.Retries) * time.Second)
		wp.jobQueue <- job // PROBLEM HERE
		return
// ...
```

**The Problem:**
1. **Data Race:** When a job fails and is requeued, the worker is currently holding the *same* reference (`job`) that is stored in `wp.results`. The worker continues processing the current failure loop, modifies the job, and then requeues it.
2. **Incorrect State Management:** The retry logic only updates the `Retries` count. If the job fails, its status is temporarily set to `failed` inside the retry block (implicitly, as it hasn't reached completion/final failure). When it's requeued, it returns to the queue, but its metadata needs to be treated correctly.
3. **Blocking:** The `time.Sleep` blocks the worker goroutine. While this is intended for backoff, it prevents the worker from handling other jobs for that duration.

**The Fix:**
1. **Update State Safely:** Before requeueing, you must update the job's status to reflect the failed state *before* the retry, and ensure this state update is visible to all components (though for job state, relying on the `results` map is standard).
2. **Remove Blocking Wait:** Do not block the worker thread. Use a separate mechanism (e.g., a time-based scheduler or a dedicated delay channel) if you need an exponential backoff. For a simple implementation, relying on the nature of the queue (and potentially adding logic to prevent immediate requeueing) is better.
3. **Atomicity:** The transition (update state $\rightarrow$ requeue) must be handled carefully.

### 3. Incorrect Statistics Calculation (Race Condition)

The `/stats` handler reads shared state without sufficient protection.

```go
// Inside /stats handler:
	pool.mu.Lock()
	stats := map[string]interface{}{
		"total_jobs": len(pool.results),
		"queue_size": len(pool.jobQueue), // <-- Potential Read Race
	}
	// ... processing results map
	pool.mu.Unlock()
```

**The Problem:** `len(pool.jobQueue)` is read while the lock is held on `pool.mu`. However, the `wp.jobQueue` channel is **not** protected by `wp.mu`. A worker goroutine (`wp.worker`) can simultaneously read from or write to the channel, leading to a race condition when calculating its length.

**The Fix:** Channels are designed for safe communication and do not generally have a mutex guard around their operations. To get an accurate, atomic count of items in a channel, you often have to rely on external metrics or accept that the count might be slightly inaccurate (which is acceptable for stats). *However, since the queue length is highly volatile, the current implementation is a classic concurrency error.*

If you must read it while holding the lock, you would need to wrap the channel access logic inside the mutex, which is generally discouraged. For this specific case, the count is inherently non-atomic, but we must at least acknowledge the design weakness.

## ⚙️ Design Issues & Improvements

These are not crashes, but areas where the code could be more robust, scalable, or Go-idiomatic.

### 4. Requeueing Strategy (Efficiency)

**The Problem:** When a job fails and is retried, it is immediately placed back at the end of the `wp.jobQueue`. If the failure rate is high, a single "problematic" job can perpetually clog the queue, causing a "busy loop" or livelock situation, as workers endlessly pick it up, fail it, and requeue it.

**The Fix (Robustness):** Implement **Circuit Breaking** or **Delayed Retries**.
1. **Circuit Breaker:** After $N$ failed retries, stop reprocessing the job and mark it as permanently failed/dead-lettered.
2. **Delayed Retries:** Instead of pushing it back immediately, push it to a separate, time-delayed queue (or use a background goroutine/timer) which will push it back to the main `jobQueue` only after the backoff time has elapsed.

### 5. Handling Job Submission Concurrency

**The Problem:** The `Submit` method assumes the caller (the HTTP handler) is stable. If the HTTP request succeeds, but the pool's internal queue is full (meaning the channel send blocks indefinitely), the HTTP server will hang until the channel accepts the job.

```go
// Inside Submit:
wp.jobQueue <- job // This blocks if the channel is full!
```

**The Fix (API Robustness):** The `Submit` function should handle queue fullness gracefully.
```go
func (wp *WorkerPool) Submit(job *Job) error {
    // ... setup job fields ...

    // Use a select statement to attempt submission with a timeout
    select {
    case wp.jobQueue <- job:
        return nil
    case <-time.After(5 * time.Second): // Or use wp.ctx.Done() for shutdown awareness
        return fmt.Errorf("failed to submit job: queue is full or timed out")
    }
}
```
*Note: If using a timeout, ensure the caller (the HTTP handler) understands that a timeout means queue saturation, not failure.*

### 6. Middleware Signature (Design)

**The Problem:** The `WorkerPool` holds a slice of `func(*Job) error` for middleware. The `RateLimiter` middleware requires the `RateLimiter` instance to be passed into a factory function, resulting in a nested closure structure:

```go
// Outside pool:
func rateLimitMiddleware(limiter *RateLimiter) func(*Job) error { ... }

// Inside pool.Use:
pool.Use(rateLimitMiddleware(limiter)) // This is fine, but a bit messy
```
This works, but it limits the pool's ability to manage middleware dependencies cleanly.

**The Fix (Refactoring):** For more complex middleware systems, you could define a standard `Middleware` interface that accepts context/dependencies, allowing the pool to register structured components rather than just closures. For this scope, the current implementation is acceptable, but worth noting.
