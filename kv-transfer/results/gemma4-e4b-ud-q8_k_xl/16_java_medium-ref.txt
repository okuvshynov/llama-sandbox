If the cache is accessed, the entry's *logical* age should reset (similar to how Redis implements TTL on access), but the current implementation does not. Furthermore, the `getValue()` method modifies `lastAccessedAt`, but this variable is unused by the primary expiration logic.

**Problem:** The entry expires based purely on its initial creation time (`createdAt`). Accessing an entry (which should *keep* it alive in a simple TTL model) does not extend its life.

### 2. Bug: Lack of Atomicity in `get(K key)` (Double Loading)
The `get(key)` method uses the pattern: Check cache $\rightarrow$ If miss, execute `loader.apply(key)` $\rightarrow$ Put result. This is highly prone to a "thundering herd" problem or duplicate loading if two threads simultaneously try to fetch the same missing key.

**Problem:** Thread A misses the cache and starts loading. Thread B also misses the cache (before A finishes and puts the value) and also starts loading.
**Fix:** Implement a mechanism for concurrent fetching, usually using `CompletableFuture` or a specialized map structure (like `ConcurrentMap` combined with `computeIfAbsent` or similar locking logic) to ensure only one thread calculates the value while others wait for it.

**Suggested Approach (Using `ConcurrentHashMap` internally for concurrency control):**

If you switch your internal `store` to a `ConcurrentHashMap`, you can use `computeIfAbsent` logic, but because you need to manage eviction and LRU, a pure `ConcurrentHashMap` is insufficient.

The minimal fix using your existing structure would involve:

1.  Using a concurrent map wrapper that tracks loading futures for each key.
2.  When `get(key)` finds a miss, it checks the futures map. If a future exists, it waits on it. If not, it creates and executes the loading Future, then places it in the map.

Since modifying the structure is complex, a practical fix is to ensure the *entire* `get` operation (including loading) is atomic for a specific key, perhaps by introducing a specific per-key lock or utilizing a `LoadingCache` pattern.

### 3. Design Flaw: `CacheEntry` State Management
The `CacheEntry` uses `final K key` but allows `V value` to be reassigned (implicitly, as it's not marked `final` in the provided snippet, though in your constructor it's set once). If you want to update the value of an existing entry without recalculating TTL, you need to ensure thread safety around the `value` field itself, which is currently reliant solely on the external lock (`lock`).

**Problem:** The `CacheEntry` should ideally be immutable or managed atomically if it is meant to be accessed outside the main lock, but currently, the modification of `lastAccessedAt` (if implemented) or the replacement of the entire entry upon `put` requires locking.

**Fix:** Ensure that any method that mutates the internal state of `CacheEntry` (like updating access time or value) is explicitly protected or uses atomic variables (`AtomicReference<V>`). Since you are already locking the entire cache structure, this is acceptable for now, but it is a fragile design point.

## ­¤Éī Performance & Concurrency Issues

### 4. Performance Bottleneck: Global Lock (`ReentrantLock`)
The use of a single `ReentrantLock` protects almost every operation (`get`, `put`, `remove`, `evictExpired`, `snapshot`).

**Problem:** Any read operation (`get`) or write operation (`put`) forces serialization, meaning only one thread can interact with the cache at any given time. This severely limits throughput under high concurrency.
**Impact:** Even simple lookups become highly contested points.

**Fix:**

*   **Fine-Grained Locking:** Use a concurrent structure (like `ConcurrentHashMap`) for the primary store.
*   **Per-Key Locking:** For operations that modify the cache structure (LRU update, eviction), consider sharding the cache into segments, each protected by its own lock, rather than using one global lock. This allows concurrent operations on different parts of the cache.

### 5. Performance Issue: LRU Implementation (`LinkedList`)
The LRU policy is implemented using `java.util.LinkedList` (`accessOrder`).

**Problem:** When an entry is accessed (`get` or `put`), the function `touchEntry(key)` performs: `accessOrder.remove(key)` followed by `accessOrder.addFirst(key)`. Removing an element from the middle of a `LinkedList` requires traversing the list from the head/tail, resulting in $O(N)$ time complexity, where $N$ is the cache size.
**Impact:** Every single successful cache hit becomes an $O(N)$ operation, destroying the expected $O(1)$ performance of a cache.

**Fix:** The standard pattern for implementing $O(1)$ LRU is to combine a `HashMap` (for $O(1)$ lookups) with a Doubly Linked List (for $O(1)$ insertion and removal from arbitrary nodes).

**Suggested Fix:**

1.  Replace `private final LinkedList<K> accessOrder` with a custom internal linked list implementation (e.g., using a `Node` class) that allows direct node manipulation.
2.  Store the reference to the corresponding `Node` within the `CacheEntry` itself.
3.  When `touchEntry` is called, retrieve the node reference from the entry and relink it to the head of the list in $O(1)$ time.
