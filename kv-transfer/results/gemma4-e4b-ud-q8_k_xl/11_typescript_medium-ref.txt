## 🐞 Bugs and Runtime Issues

### 1. Incorrect Loop Control in `run()`
The `run()` method uses a `while (true)` loop combined with `Promise.all(batch.map(...))` which immediately resolves when the `batch` finishes. If the pipeline processes a batch and finishes it, the loop continues, attempts to find *more* ready tasks, but if none are ready (and everything is completed or failed), it hits the check at the top:

```typescript
// Inside run():
while (true) {
    const ready = this.getReadyTasks();
    if (ready.length === 0) {
        // ... deadlock/completion checks
        if (pending.length === 0 && running.length === 0) break; // Only breaks if both are zero
        // ... wait
        continue;
    }
    // When tasks are running, they complete asynchronously. 
    // The loop immediately continues to the next iteration, potentially starting a new batch 
    // before the previous batch has truly finished its side effects (like updating task status).
    const batch = ready.slice(0, this.concurrency);
    await Promise.all(batch.map(task => this.executeTask(task))); // <-- This waits for the batch to finish
}
```

**The Problem:** By `await`ing `Promise.all(batch.map(...))` inside the loop, the loop blocks until the current batch is finished. However, the dependency resolution logic (`getReadyTasks`) only checks the current state (`task.status`). If a batch is running, the subsequent iterations of the `while` loop will only see new tasks become ready *after* the current batch resolves and the internal state updates complete. This pattern is brittle for a reactive, continuous scheduler.

**Improvement Suggestion:**
Instead of a tight `while(true)` loop that polls, a more robust scheduler should react to state changes. Use the event system (`task:complete`, `task:fail`) to trigger the next scheduling cycle, or use a mechanism that waits until the *next* task becomes available before proceeding, rather than blindly polling every 100ms.

### 2. Deadlock Detection Simplification (Potential)
Your current deadlock detection logic is:
> If no tasks are ready (`ready.length === 0`), and there are still pending tasks (`pending.length > 0`), but nothing is running (`running.length === 0`), then we have a deadlock.

This is generally correct, but the logic relies on the polling interval (100ms) being small enough to catch the state before an infinite wait begins. If the pipeline halts in a state where tasks are pending but are waiting for a task that will never run (e.g., circular dependency), the polling mechanism handles it, but it's not an immediate, provable detection.

**Improvement Suggestion:**
While complex, a production-grade scheduler might check for circular dependencies during `addTask` or after all tasks have been defined. For this implementation, the current polling deadlock detection is acceptable but should be documented as such.

## 🔒 Type Safety and Generics Issues

### 3. Inconsistent Task Typing in `addTask`
When using `addTask`, the configuration uses `Promise<T>`, but the `Task` interface assumes the result type `T` is carried through.

```typescript
// In addTask:
addTask<T>(config: {
    // ...
    execute: () => Promise<T>;
    // ...
}): this {
    const task: Task<T> = {
        // ...
        status: 'pending',
    };
    // ...
}

// In PipelineEvents:
'task:complete': (task: Task, result: any) => void; // <-- 'any' here
```

**The Problem:** The `task:complete` event handler specifies `result: any`. If a user defines a task returning `Promise<User[]>` and another returning `Promise<string>`, the event handler loses the type information about the successful result.

**Improvement Suggestion:**
The `PipelineEvents` interface should be parameterized based on the tasks defined, or the event system needs a more complex signature, potentially requiring users to define custom event handlers based on task ID or event type to retain type safety. Since a single `Pipeline` instance manages heterogeneous tasks, forcing a single generic type for the event is impossible without losing specificity. Using `any` for events is a pragmatic choice here, but it defeats the purpose of strict typing for that specific event.

**Mitigation:** Change `PipelineEvents` to use a union type or use `unknown` for `result` to at least enforce checking/narrowing, although `any` is often used for heterogeneous event payloads.

## 📐 Architectural and Design Issues

### 6. Redundant Listener Clearing in `off`
In `TypedEventEmitter.off`:

```typescript
// In off:
off<K extends keyof T>(event: K, listener: T[K]): void {
    this.listeners.get(event)?.delete(listener);
    this.onceListeners.get(event)?.delete(listener); // <-- Issue here
}
```

**The Problem:** Once a listener is successfully executed and cleared from `onceListeners` (usually via `emit` logic), there is no need for `off` to also manually try to remove it from `onceListeners`. If a consumer manually calls `off` on a listener that was originally set via `once`, this logic is fine, but it might indicate a confusion of state management. More importantly, the listener set should track *all* listeners regardless of whether they are `once` or not, if the goal is a unified removal mechanism.

**Improvement Suggestion:**
Maintain the separation of concerns (listeners vs. onceListeners) but ensure the primary job of `off` is simply removal. If a listener is *only* meant to be cleared once, it should ideally be handled internally by the emission cycle, not rely on external manual cleanup via `off`.
