## 1. SlabAllocator (Memory Pool)

The slab allocator is a standard, thread-safe memory pool implementation.

### 🐞 Bugs & Issues

1. **Incorrect Initialization of `object_size_`:**
   The constructor logic for `object_size_` is flawed:
   ```cpp
   object_size_(object_size < sizeof(SlabBlock) ? sizeof(SlabBlock) : object_size)
   ```
   If a user requests a small object size (e.g., 16 bytes), the allocator correctly ensures the allocated size is at least `sizeof(SlabBlock)`. However, if the user requests a large object size (e.g., 512 bytes), `object_size_` is correctly set. The core issue is that **when calculating chunk alignment**, the code relies on `std::aligned_alloc(CACHE_LINE_SIZE, alloc_size)`. This only ensures the *start* of the chunk is aligned, not that the subsequent blocks within the chunk are aligned based on `object_size_` if `object_size_` is larger than the cache line size. Since `std::aligned_alloc` returns a raw memory chunk, the block alignment might be implicitly handled by the cast, but this implementation is fragile.

2. **Potential Integer Overflow in `grow()`:**
   ```cpp
   size_t alloc_size = object_size_ * count;
   char* chunk = static_cast<char*>(std::aligned_alloc(CACHE_LINE_SIZE, alloc_size));
   ```
   If `object_size_` and `count` are large (e.g., near `SIZE_MAX / 2`), `alloc_size` could wrap around, leading to a small allocation request that crashes later when blocks are accessed.

### 🚀 Performance & Design Fixes

1. **Allocation Overhead:** The use of `std::mutex` on *every* `allocate()` and `deallocate()` call is a severe performance bottleneck. For a high-throughput pool, this mutex will be a major contention point.
   * **Fix:** Implement thread-local caches (per-thread pools) that lock only when the local cache runs empty or needs synchronization with the global pool. The central mutex should only guard the global free list refill.
2. **Standard Library Usage:** Using `std::free` in the destructor is correct for memory allocated by `std::aligned_alloc`, but this complexity is often best handled by wrapping the allocation/deallocation logic entirely or using a custom deallocator/allocator pair.

## 2. HazardPointerDomain / HazardPointerGuard

The Hazard Pointer implementation aims to enable safe memory reclamation (SMR) in the absence of garbage collection, particularly needed for lock-free data structures like `LockFreeHashMap`.

### 🐞 Bugs & Issues (Critical)

1. **Hazard Pointer Ownership/Reclamation (CRITICAL DESIGN FLAW):**
   The `HazardPointerDomain` currently *only* provides a mechanism to *retire* a pointer and checks it against active hazard pointers. **It lacks the actual memory reclamation step (i.e., the safe destruction/deallocation of the retired object).**
   When `scan` finishes, objects that are not protected are retired. The `retire` function passes a `deleter` lambda, which is invoked in `scan`. However, the objects being retired (`void* ptr`) originate from the `LockFreeHashMap` and are managed by the `SlabAllocator` pool. The domain has no knowledge of how to safely return memory to the pool without potentially double-freeing or leaking.

2. **ABA Problem Vulnerability (HazardPointerGuard):**
   The `HazardPointerGuard::protect` function reads the `source` atomic pointer, stores it, and then returns it. It does **not** perform a check against concurrent modifications.
   ```cpp
   // In HazardPointerGuard::protect
   do {
       ptr = source.load(std::memory_order_relaxed);
       record_->pointers[index_].store(ptr, std::memory_order_relaxed);
   } while (ptr != source.load(std::memory_order_relaxed)); // <-- This loop is faulty
   return ptr;
   ```
   This `do-while` loop condition is fundamentally broken for protection. If a thread reads `ptr`, stores it, and then another thread modifies `source` *after* the read but *before* the store, the CAS check in the loop condition might succeed spuriously or might not catch the concurrent change, but more importantly, the goal of hazard pointers is to ensure the *pointed-to* object remains valid while *used*. The current design only ensures the pointer was stable at the time of the read, which is insufficient if the subsequent usage involves non-atomic operations or if the pointer could change and cycle (ABA).

3. **Thread ID Equality (Subtle Bug):**
   Comparing `std::thread::id` for emptiness (`std::thread::id empty{};`) is generally fine, but using `compare_exchange_strong` on a volatile system resource like thread IDs can lead to tricky races, although the implementation here uses relaxed ordering which is typical for optimistic checks.

### 🚀 Performance & Design Fixes

1. **Implement True SMR:** The `HazardPointerDomain` must be integrated with the memory allocator. If the pointed-to objects are pooled (like in the Hash Map), the domain needs a mechanism to track object ownership and guarantee that the pool is only modified (deallocated) after the pointer is confirmed safe.
2. **Refactor Protection:** Hazard pointer usage should ideally be scoped around the *use* of the pointer, not just its initial read. If the structure being pointed to is complex, its safety must be guaranteed until the guard destructs.
3. **Memory Ordering:** Using `std::memory_order_relaxed` extensively in the HZ domain risks breaking the fundamental guarantees of SMR, especially around the visibility of updates to the pointer array itself. Stronger ordering (like `acquire`/`release`) might be necessary depending on how pointer state changes are synchronized.

## 3. LockFreeHashMap

This is the most complex component. Implementing a robust lock-free data structure requires extremely meticulous adherence to memory models and CAS loops.

### 🐞 Bugs & Issues (Critical)

1. **Resize Contention (CRITICAL DESIGN FLAW):**
   The `resize()` function is executed *without* any external synchronization or lock-free protocol. If `resize()` is called while other threads are running `insert()`, `find()`, or `erase()`, the following occurs:
    *   The entire map structure (`table_`, `capacity_`) is replaced.
    *   Threads might be traversing the old `table_` while it is being deleted, or they might suddenly see the new `table_` and try to probe using old or incorrect size/capacity parameters.
    *   The process of re-hashing and moving values is inherently sequential and non-atomic relative to user operations.
   *   **Fix:** Lock-free resizing is extraordinarily difficult. This operation requires epoch-based memory management (like RCU) or a comprehensive CAS-based migration protocol (e.g., using multi-word CAS or linearization points) to ensure all threads see a consistent state transition.

2. **Missing ABA Check during CAS (Performance/Safety):**
   When performing CAS operations, especially on the `key` or `value` pointers, the code relies purely on the value matching (`existing == k`). In high-contention scenarios, especially with tombstones, this simple check does not guarantee that the state hasn't been modified and reverted (ABA problem), which could lead to corrupted state or missed updates.
