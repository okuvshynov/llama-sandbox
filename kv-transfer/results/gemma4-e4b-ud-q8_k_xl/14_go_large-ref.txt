## 📂 1. `scheduler.go` (Core Scheduler)

### 🛑 Concurrency & Race Conditions

**1. Issue: Race Condition in `Task.Retry` Logic**
In `CompleteTask`, when a task fails and is scheduled for retry:

```go
// scheduler.go (inside CompleteTask)
		if taskRetryCount < task.MaxRetries {
			// ...
			go func() { // <-- UNPROTECTED ASYNC EXECUTION
				delay := task.RetryDelay * time.Duration(task.RetryCount)
				jitter := time.Duration(rand.Int63n(int64(delay / 4)))
				time.Sleep(delay + jitter)
				s.mu.Lock() // Locking only during the push
				task.State = TaskPending
				heap.Push(&s.queue, task)
				s.mu.Unlock()
			}() // <-- The task state (TaskRetrying -> TaskPending) update is outside the main lock
			// ...
		}
```

The `go func()` runs asynchronously. While it eventually locks to push the task, the state transition from `TaskRetrying` back to `TaskPending` happens *uncontrolled* relative to the main scheduler state. More critically, multiple retrying tasks might execute this goroutine, potentially leading to race conditions if they try to modify shared state outside the mutex scope (though here, it mostly modifies the task itself, which is shared).

**Fix:** The retry logic should use a dedicated channel or a managed retry loop rather than relying on an uncontrolled, detached goroutine. The entire scheduling and state change should ideally be managed by the main scheduler loop or a dedicated retry manager.

**2. Issue: Unsafe Access to Worker Load/Stats (Minor)**
In `CompleteTask`, you use `atomic.AddInt64(&worker.TasksFailed, 1)` and `atomic.AddInt64(&worker.TasksCompleted, 1)` *while the main scheduler mutex (`s.mu`) is held*.

```go
// scheduler.go (inside CompleteTask)
		if worker != nil {
			atomic.AddInt64(&worker.TasksFailed, 1) // Atomic operation inside main lock
		}
```
While atomic operations are safe, holding the global lock (`s.mu`) while potentially manipulating concurrent counters inside a loop (`for _, worker := range s.workers`) is unnecessary contention and slightly degrades performance.

**Fix:** Only lock `s.mu` to perform state transitions (`task.State = TaskCompleted`, `worker.Load--`, etc.). Move the `atomic.AddInt64` calls *outside* the critical section, after releasing `s.mu`, or manage them carefully.

### 🐛 Bugs & Logic

**3. Issue: Dependency Resolution During Submission is Incomplete**
In `SubmitTask`, dependency checking is flawed:

```go
// scheduler.go (inside SubmitTask)
		allResolved := true
		for _, depID := range task.Dependencies {
			dep, exists := s.completed[depID]
			if !exists || dep.State != TaskCompleted {
				allResolved = false
				break
			}
		}
		if !allResolved {
			s.tasks[task.ID] = task
			return nil // <-- Task is stored in s.tasks but is NOT pushed to the queue
		}
```
If a task has dependencies that are not yet completed, it is correctly marked as unready and stored in `s.tasks`. However, the scheduler relies on `CompleteTask` to *re-evaluate* dependents and push them. If a dependent task is never re-evaluated (e.g., if the `s.depGraph` population was faulty), it will remain stuck in `s.tasks` but never run.

The logic here relies entirely on the dependent task being re-evaluated *only* when one of its dependencies completes. This is the correct dependency pattern, but the `s.depGraph` population needs to be checked carefully.

**4. Issue: Worker Status Update Race in `CompleteTask`**
In `CompleteTask`, if a worker is successfully assigned a task, its load is decremented. If that worker dies *just* before the task completes, the worker might not exist in `s.workers` when `CompleteTask` runs, which is fine. However, the tracking of `worker.Load` is manual and could be inconsistent if there is a period where `AssignTask` is running and `CompleteTask` is running.

**5. Issue: `CleanupCompletedTasks` Logic is Destructive**
```go
// scheduler.go (inside cleanupCompletedTasks)
	cutoff := time.Now().Add(-1 * time.Hour)
	for id, task := range s.completed {
		if task.CompletedAt.Before(cutoff) {
			delete(s.completed, id)
			delete(s.tasks, id) // <-- Deleting from s.tasks
		}
	}
```
If a task is in `s.completed`, it *must* still reside in `s.tasks` because `s.completed` is a map *subset* of the complete tasks tracked in `s.tasks`. Deleting it from `s.tasks` breaks the authoritative record of the task, which is dangerous for auditing or state checks.

**Fix:** `s.completed` should simply be a subset index of `s.tasks`, or `s.tasks` should hold all tasks, and `s.completed` should be removed, relying on a state check (`task.State == TaskCompleted`) against the main `s.tasks` map.

### ⚡ Performance Issues

**6. Issue: High Lock Contention on `s.mu`**
The global scheduler lock (`s.mu`) is used for almost every public method (`SubmitTask`, `CancelTask`, `AssignTask`, `CompleteTask`, `RegisterWorker`, etc.). Since `CompleteTask` is likely the most frequent call in a high-throughput system (every task finish), holding this lock for state transitions, dependency checks, *and* queue operations creates a massive bottleneck.

**Fix:** Refactor critical sections. Use finer-grained locks. For instance, use a separate lock for the `TaskQueue` operations (heap access), and potentially separate locks for `s.tasks` and `s.workers`. However, due to the complex cross-mapping (Tasks linked to Workers, Tasks linked to Dependencies), a full micro-locking solution is hard; focus on reducing the *time* spent under the lock (e.g., perform I/O or blocking calls outside the lock).

**7. Issue: Repeated Dependency Checks (O(N*M) Complexity)**
When a task completes, you iterate over all its dependents and then, for each dependent, you iterate over *all* of that dependent's dependencies to check if *all* are completed.

```go
// scheduler.go (inside CompleteTask)
			allResolved := true
			for _, reqID := range depTask.Dependencies {
				req, done := s.completed[reqID]
				if !done || req.State != TaskCompleted { // <-- This check is redundant/expensive
					allResolved = false
					break
				}
			}
			if allResolved {
				heap.Push(&s.queue, depTask)
			}
```
If `s.completed` is maintained correctly, checking `req.State == TaskCompleted` is sufficient.
