## 🐞 Bugs and Correctness Issues

### 1. Inconsistent Key Handling (Hashing vs. Storage)
**Issue:** The cache uses `_hash_key(key)` internally to store keys in `self.entries` (e.g., `hashed`). However, when logging operations (`_log_operation`), it passes the original, unhashed key.
*   **The Bug:** The `cache_log` table stores the original key. If the original key was very long, and the cache intended to use its *hashed* version for internal integrity/efficiency, the log may become inconsistent or misleading relative to internal operations. More critically, consistency is lost if the user assumes the cache is keyed by the original string, but all internal checks use the hash.
*   **Suggestion:** Ensure that when logging, if the internal cache logic is based on the hash, the key logged should ideally be the hash, or clearly documented that the log uses the original key for human readability. Since the hash is just an internal lookup mechanism, logging the original key is acceptable, but the logic should be clear. **(Minor concern, but worth noting for architectural clarity).**

### 2. Potential SQLite Locking Contention (Performance/Integrity)
**Issue:** The `_log_operation` method calls `self.db.commit()` *inside* the critical path of `get` and `put`.
*   **The Problem:** Every single cache hit or miss requires a database transaction (write) and commit. SQLite is a file-based database, and committing frequently forces file synchronization, which is an extremely slow I/O operation, especially under high load. This will become a major bottleneck, turning a fast in-memory cache into an I/O-bound service.
*   **Suggestion:** Decouple logging from the cache operation. Instead of committing immediately:
    1.  Log operations to an in-memory queue (`collections.deque`).
    2.  Run a separate, dedicated background thread (or use the `PeriodicCleaner` logic) that drains this queue and batches inserts/commits to the SQLite database every few seconds.

### 3. Access Policy Complexity and Potential Race Condition during Eviction
**Issue:** When eviction occurs in `put`, the code relies solely on `LRUEvictionPolicy.get_eviction_candidate()`. While this is mostly correct, the interaction between the policies is fragile.
*   **The Problem:** LRU and TTL policies are maintained separately. When `_evict(candidate)` is called, it must manually remove the key from *both* `self.entries`, `self.lru`, and `self.ttl`. If the removal fails or is missed in one policy, the policies become desynchronized, leading to incorrect future evictions or stale data.
*   **Suggestion:** Centralize the entry management. When an item is placed in `self.entries`, it should become the single source of truth. The LRU and TTL policies should operate on the keys/hashes managed by `self.entries`. The `_evict` method is currently performing necessary cleanup, but this complexity highlights that relying on three independent, manually synchronized data structures (`entries`, `lru.access_order`, `ttl.entries`) is high maintenance and bug-prone.

## 🐌 Performance Issues

### 4. $O(N)$ Operations in LRUEvictionPolicy
**Issue:** The `LRUEvictionPolicy` uses a standard Python `list` (`self.access_order`) to track access order.
*   **The Problem:** When `record_access(key)` is called, if the key is already present, `self.access_order.remove(key)` is executed. Removing an item from the middle of a list requires shifting all subsequent elements, resulting in $O(N)$ time complexity, where $N$ is the number of entries.
*   **Suggestion:** Implement LRU using a combination of a **dictionary** (for $O(1)$ key lookup) and a **doubly linked list** (for $O(1)$ item movement and removal). Python's `collections.OrderedDict` or `collections.deque` (if customized) can provide this functionality efficiently, or simply using `dict` and manually managing the order via move-to-end logic (as Python 3.7+ dicts maintain insertion order).

### 5. Repeated Deep Serialization for Size Calculation
**Issue:** The method `_compute_size(self, value)` calls `len(json.dumps(value))`.
*   **The Problem:** This forces serialization of the entire Python object into JSON *every single time* an item is put into the cache. If `put` is called frequently, this repeated serialization adds significant CPU overhead.
*   **Suggestion:** If the input data structure (`value`) is complex, consider having the caller pre-calculate and pass the size/serialized form along with the value, or ensure the size calculation is only performed once during the initial `put` call.

### 6. Global Lock Contention
**Issue:** The entire `CacheStore` uses a single `threading.Lock()` (`self.lock`) protecting all read (`get`), write (`put`, `delete`), and cleanup operations.
*   **The Problem:** This creates a global bottleneck. Even if one thread is performing a simple read (`get`), it blocks all other threads, including those performing reads.
*   **Suggestion:** Use a **Read-Write Lock** (or `threading.RLock` combined with careful scope management, though a true RW lock is better). This allows multiple readers to access the cache concurrently while still ensuring exclusive access for writers (puts, deletes, cleanup).

## 📐 Design and Maintainability Problems

### 7. Monolithic CacheStore Responsibility
**Issue:** The `CacheStore` is currently responsible for:
1. In-memory storage and retrieval.
2. Implementing LRU and TTL policies.
3. Managing thread synchronization.
4. Calculating object size (serialization).
5. Logging all operations to a persistent database.
*   **The Problem:** This violates the Single Responsibility Principle (SRP). The cache is too large and complex. If we wanted to swap the database (e.g., from SQLite to Redis), we would have to modify the core `CacheStore` class.
*   **Suggestion:** Introduce dependency injection/separation:
    *   Extract logging into a dedicated `CacheLogger` class (which could handle asynchronous writing).
    *   Extract persistence/storage into an interface if you wanted to add disk storage or external services later.

### 8. Inefficient TTL Policy Scanning
**Issue:** The `TTLEvictionPolicy.get_expired_keys()` iterates over *every single entry* (`self.entries.items()`) in the cache on every cleanup cycle.
*   **The Problem:** If the cache grows very large (millions of entries), this scan becomes an $O(N)$ operation, regardless of how many keys are actually expired.
*   **Suggestion:** Instead of iterating over the whole dictionary, maintain a sorted structure (like a min-heap or a specialized structure optimized for time) of expiration timestamps. When cleaning up, you only need to check the top entry in the heap/sorted structure. This reduces the scan time from $O(N)$ to $O(\log N)$ or even $O(1)$ to find the oldest entry.

### 9. Error Handling for Serialization
**Issue:** The `_compute_size` relies on `json.dumps(value)`.
*   **The Problem:** If `value` contains non-serializable objects (e.g., `datetime` objects without a custom encoder, or file handles), `json.dumps` will raise a `TypeError`, crashing the `put` method.
*   **Suggestion:** Wrap serialization in a `try...except` block and either log the error and reject the put, or attempt to serialize the object using a custom JSON encoder.
