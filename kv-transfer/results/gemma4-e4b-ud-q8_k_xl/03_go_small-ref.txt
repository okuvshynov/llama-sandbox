## 🐛 Critical Bugs (Data Races & Crashes)

These bugs lead to runtime panics or unpredictable concurrent behavior.

### 1. Nil Map Panic (In `Counter.Increment`)

The `Counter` struct initializes `counts` to `nil` because maps are reference types and are not initialized by default in Go. The first time `Increment` runs, it attempts to write to this nil map, causing a panic (`panic: assignment to entry in nil map`).

**Code Section Affected:**
```go
func (c *Counter) Increment(key string) int {
	c.mu.Lock()
	c.counts[key]++ // <-- CRASHES HERE on the very first call
	// ...
}
```

**Explanation:**
Before any operation can occur, the map must be initialized using `make(map[string]int)`.

**Proposed Fix:**
Initialize the map in the `NewCounter` constructor:
```go
func NewCounter() *Counter {
	return &Counter{
		counts: make(map[string]int), // <-- Add this initialization
	}
}
```

### 2. Data Race on Read (In `Counter.Get`)

The `Get` method reads from the shared `counts` map without acquiring the mutex lock. If another goroutine is simultaneously calling `Increment` (which is holding the lock and modifying the map), this creates a classic data race, which can lead to crashes or incorrect values.

**Code Section Affected:**
```go
func (c *Counter) Get(key string) int {
	return c.counts[key] // <-- UNLOCKED READ - DATA RACE!
}
```

**Explanation:**
Any read operation on shared, mutable data must be protected by the same synchronization primitive (the mutex) used for writes.

**Proposed Fix:**
Acquire the lock before reading:
```go
func (c *Counter) Get(key string) int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts[key]
}
```

### 3. Exposed Internal State / Mutable Reference (In `Counter.GetAll`)

The `GetAll` method returns the direct internal reference to the `c.counts` map.

**Code Section Affected:**
```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts // <-- Returns a pointer/reference to the internal map
}
```

**Explanation:**
While the lock is held when the function executes, the calling function (`handleGetAll`) receives a reference to the *actual* internal map. If a client or another part of the application modified this returned map (e.g., `results["hacked"] = 999`), it would modify the server's internal state (`c.counts`) without acquiring the lock, leading to a severe data race and state corruption.

**Proposed Fix:**
The method must return a *copy* of the map:
```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()

	// Create a deep copy of the internal map
	copyMap := make(map[string]int)
	for k, v := range c.counts {
		copyMap[k] = v
	}
	return copyMap // Return the copy
}
```

## 🔬 Minor Bugs / Best Practice Issues

These bugs won't crash the server but violate standard HTTP protocol expectations or lead to poor performance/design.

### 4. Missing HTTP Content Type Header (In `handleGetAll`)

When serving data structured as JSON, the HTTP response *must* include the `Content-Type: application/json` header. Without this header, clients (like web browsers or other APIs) cannot reliably parse the response.

**Code Section Affected:**
```go
func handleGetAll(w http.ResponseWriter, r *http.Request) {
	data := counter.GetAll()
	json.NewEncoder(w).Encode(data) // <-- Does not set Content-Type
}
```

**Explanation:**
This is a fundamental requirement for any API endpoint returning JSON.

**Proposed Fix:**
Set the header before encoding the response:
```go
func handleGetAll(w http.ResponseWriter, r *http.Request) {
	w.Header().Set("Content-Type", "application/json") // <-- Fix
	data := counter.GetAll()
	json.NewEncoder(w).Encode(data)
}
```
