## 🐞 The Bug: Time Complexity

The bug lies in the `running_median` function, specifically the repeated use of the `find_median` function within the loop.

### Explanation

1.  **Redundant Sorting:** In every iteration of the `running_median` loop, the code calls `find_median(window)`.
2.  **Inefficiency:** The `find_median` function immediately calls `sorted(nums)` on the current `window`. This means that for every single data point processed, the entire window of up to 5 elements is fully sorted from scratch.
3.  **Time Complexity:** If $N$ is the total number of elements in the stream, and $W$ is the fixed window size (here $W=5$), the loop runs $N$ times. Since sorting a list of size $W$ takes $O(W \log W)$ time, the total time complexity of the running median function is $O(N \cdot W \log W)$.

While $W$ is very small (5), leading to a fast execution time for this specific example, if the window size were large (e.g., $W=10,000$), this method would be prohibitively slow because of the repeated sorting operation. A proper solution for running medians must maintain the sorted structure incrementally.

## ✅ The Fix (Algorithmic Improvement)

To fix this performance bug, we must avoid sorting the entire window in every step. We need a data structure that allows insertion and deletion while maintaining the order or providing quick access to the median.

The standard, highly efficient way to solve the running median problem is using **Two Heaps** (a Min-Heap and a Max-Heap).

### How the Fix Works (Conceptual)

1.  **Two Heaps:** Maintain two heaps:

3.  **Median Access:** The median is always the root of one of the heaps (or the average of the two roots).
