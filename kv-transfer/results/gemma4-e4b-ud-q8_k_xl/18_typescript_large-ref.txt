## 🛡️ Security Vulnerabilities (CRITICAL)

These issues expose the system to direct attack vectors and should be addressed immediately.

### 1. SQL Injection Vulnerabilities
**Location:** `UserRepository.findByEmail`, `DocumentRepository.search`, `SearchService.getSuggestions`, `AuditLogRepository.findByUser`.

**Problem:** Several repository methods construct SQL queries by directly concatenating user input strings. This allows an attacker to inject malicious SQL commands (e.g., `'; DROP TABLE users; --`).

**Example (UserRepository):**
```typescript
// Vulnerable line:
const result = await pool.query("SELECT * FROM users WHERE email = '" + email + "'");
```

**Fix:** **Always use parameterized queries (`$1`, `$2`, etc.)** for all user-supplied data.

**Example Fix (UserRepository):**
```typescript
async findByEmail(email: string): Promise<User | null> {
  // Safe:
  const result = await pool.query("SELECT * FROM users WHERE email = $1", [email]);
  return result.rows[0] || null;
}
```

### 2. Weak Password Hashing
**Location:** `AuthService.register`, `AuthService.login`.

**Problem:** The service uses SHA256 with a self-generated salt. While this incorporates a salt, SHA256 is a fast hashing algorithm, making it highly susceptible to brute-force attacks using modern GPU hardware. This is completely inadequate for password storage.

**Fix:** **Use a robust, slow, key-derivation function (KDF).** The industry standard is **Bcrypt** or **Argon2**. These libraries are designed to be intentionally slow, mitigating brute-force attacks.

**Suggestion:** Replace `crypto.createHash("sha256")` with a library like `bcrypt`.

### 3. Exposure of Internal Errors (Production Risk)
**Location:** Multiple route handlers, especially `app.post("/api/auth/register")`, `app.use` error handler.

**Problem:** In the catch blocks, the code often sends `(err as Error).message` and sometimes the entire `stack` trace to the client. This leaks valuable internal information about the database schema, file paths, and application logic, aiding attackers in reconnaissance.

**Fix:** **Standardize generic error responses.** In production, catch errors and only return a non-specific message ("Internal Server Error") to the client. Log the detailed error internally.

**Example Fix (Route Handler):**
```typescript
// Instead of:
res.status(500).json({ error: (err as Error).message, stack: (err as Error).stack });

// Use this:
res.status(500).json({ error: "An unexpected server error occurred." });
console.error("Error detail:", err); // Log only internally
```

### 4. Missing Authorization Checks on Operations
**Location:** Document and Folder routes.

**Problem:** While some routes use `roleMiddleware("admin")`, many critical operations rely only on `authMiddleware`. For instance, the document update logic has some permission checks, but many services perform implicit actions (like deletion or version history modification) without a clear check that the user has *any* required permissions (read/write/admin) for the resource they are touching.

**Fix:** Implement granular access control
