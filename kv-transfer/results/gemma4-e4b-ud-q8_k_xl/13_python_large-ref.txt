## 🐞 Bugs and Data Integrity Issues

### 1. Floating Point Arithmetic in Financial Models (`models.py`)
**Issue:** All core financial calculations (`Product.price`, `OrderItem.subtotal`, `Order.total`, `Order.shipping_cost`, `Order.tax`) rely on standard Python `float` types. Floats are binary approximations and cannot accurately represent currency, leading to subtle rounding errors and financial inconsistencies (e.g., $0.10 + $0.20 might result in $0.30000000000000004).
**Impact:** Critical data integrity failure. Incorrect billing, revenue reporting, and customer trust loss.
**Fix:** Use Python's `decimal` module for all monetary values. Replace `float` with `Decimal` in `Product.price`, `OrderItem.unit_price`, and all methods that calculate totals/discounts.

### 2. Incomplete Reservation/Commit Atomicity (`order_processor.py` / `inventory.py`)
**Issue:** The transition from reserving stock to committing stock is not atomic. The critical path is:
1. Inventory reserves stock (`reserve_stock`).
2. Payment processes (`payment.charge`).
3. Inventory commits stock (`commit_reservation`).
If the system crashes or fails between Step 2 and Step 3, the order is paid for, but the stock is neither reserved nor permanently deducted. Furthermore, if the payment succeeds but the commit fails, the inventory is in an inconsistent state.
**Impact:** Lost inventory (phantom sales) or unfulfilled orders (inventory deadlock).
**Fix:** Implement a robust two-phase commit pattern or, more practically, use a transactional database where the reservation and stock decrement are within a single database transaction managed by the `OrderProcessor`.

### 3. Missing Cleanup Trigger in Inventory (`inventory.py`)
**Issue:** The `cleanup_expired_reservations` method exists but is never called. Reservations will accumulate indefinitely if the corresponding orders are not processed, leading to memory exhaustion and incorrect availability checks (`_get_reserved_quantity`).
**Impact:** Resource leak, inaccurate stock levels.
**Fix:** In a real microservice, this function must be scheduled to run periodically (e.g., every 5 minutes) using a dedicated scheduler (like Celery Beat or a cron job).

## 🚀 Performance and Scalability Issues

### 4. Inventory Locking and Scaling Constraint (`inventory.py`)
**Issue:** The `InventoryManager` uses `threading.Lock()`. This lock only protects the state *within a single Python process*. If you deploy this microservice horizontally (e.g., running three instances on three different servers for load balancing), each instance will maintain its own copy of `self.products` and `self.reservations`. Race conditions are guaranteed because all instances operate on divergent, unsynchronized data.
**Impact:** Massive race conditions, data corruption (overselling), and total system failure under load.
**Fix:** Refactor `InventoryManager` to be stateless regarding its storage and delegate all state changes (check availability, reserve, commit) to a centralized, transactional data store (e.g., PostgreSQL, Redis with Lua scripts, or DynamoDB
