## 🚨 Critical Bugs & Undefined Behavior

These issues introduce severe runtime risks, such as crashes or data corruption, and **must be fixed** before the system is considered stable.

### 1. TaskDeque Race Condition (Stealing)
**Location:** `queue.rs`, `TaskDeque::steal`
**Issue:** This function contains a direct data race and leads to Undefined Behavior (UB).
```rust
// Data race: buffer is not atomic, owner can modify buffer[idx]
// while we read it. This needs unsafe + atomic reads or a
// different design.
let task_ptr = &self.buffer[idx] as *const Option<Task>;
let task = unsafe { std::ptr::read(task_ptr) };

// ...
if self.head
    .compare_exchange(head, head + 1, Ordering::Relaxed, Ordering::Relaxed)
    .is_ok()
{
    task // We assume ownership based on CAS success
} else {
    // ...
    std::mem::forget(task); // UB here if task was read prematurely
    None
}
```
When a stealer reads `self.buffer[idx]`, the owner thread (which might be running `pop`) could simultaneously write to or take ownership of that slot. Using `std::ptr::read` on a mutable `Vec` element without proper synchronization is unsafe. If the CAS fails, you have already copied the data (which might be in a transient state), leading to double-ownership or memory corruption.
**Fix:** A lock-free queue designed for multi-producer/multi-consumer (MPMC) access requires using atomic pointers (`Arc` or `AtomicPtr`) to manage the buffer entries, or using a complete non-lock-free wrapper (like `Mutex`) for stealing. Since the design attempts lock-free, the internal buffer structure must be changed to be inherently thread-safe, usually by using specialized crates like `crossbeam-queue` or implementing a sophisticated atomic ring buffer.

### 2. Waker Implementation Flaw (Memory Leaks / Double-Free Risk)
**Location:** `worker.rs`, `futures_waker`
**Issue:** The custom `Waker` implementation relies on manual memory management (`Arc::from_raw`/`std::mem::forget`) and has flaws that risk both leaks and double-frees.
Specifically, in `wake` and `wake_by_ref`:
```rust
unsafe fn wake(ptr: *const ()) {
    let arc = Arc::from_raw(ptr as *const TaskInner); // Takes ownership
    if let Some(w) = arc.waker.lock().unwrap().take() {
        w.wake();
    }
    // Arc is consumed here but the waker might be referenced elsewhere.
}
// ...
unsafe fn wake_by_ref(ptr: *const ()) {
    let arc = Arc::from_raw(ptr as *const TaskInner); // Takes ownership
    // ...
    std::mem::forget(arc); // Forgets ownership
}
```
If the waker is successfully retrieved and used, the pointer handling is complex. Crucially, in `wake`, you consume the `Arc`, but the caller doesn't explicitly manage the lifetime of the raw pointer. If the `TaskInner` object is dropped before the waker is fully retired, this setup is vulnerable. The standard pattern for custom `Waker`s is often simplified by using `Arc` or `Weak` reference counting within the task structure itself, rather than raw pointers for management.
**Fix:** Use `Arc` directly as the identity for the task state and wrap the logic carefully to ensure that reference counts are correctly managed, avoiding `std::mem::forget` unless absolutely necessary for the raw pointer interface. A safer approach is to pass the `Arc<TaskInner>` into the waker creation and use internal `tokio::sync::Notify` or similar mechanisms to signal readiness instead of relying on raw waker mechanics for state changes.

### 3. TaskResult Sender Dropped Too Early (TaskHandle)
**Location:** `scheduler.rs`, `Scheduler::submit`
**Issue:** The `oneshot::channel` sender (`result_tx`) is created locally within `submit` but is never passed to the `TaskInner` or the worker thread.
```rust
let (result_tx, result_rx) = oneshot::channel();
// ...
let handle = TaskHandle { inner: task.inner.clone(), result_rx: Some(result_rx) };
// result_tx is dropped here implicitly.
```
When `submit` returns, `result_tx` goes out of scope and is dropped. This means `result_rx` is already poisoned (`Err(RecvError)`). Any caller trying to `await_result` will immediately fail or receive `None`, regardless of whether the task succeeds or fails.
**Fix:** The `TaskInner` structure must hold the `oneshot::Sender<TaskResult>`. When the worker completes the task, it must access this stored sender to transmit the result to the handle's receiver.

## 🚀 Performance & Concurrency Issues

These issues degrade the system's performance or create unnecessary bottlenecks.

### 4. Blocking the Async Runtime (Worker Loop)
**Location:** `worker.rs`, `worker_loop`
**Issue:** The worker loop mixes synchronous blocking logic (`std::condvar::wait_timeout`) with the asynchronous `tokio` runtime.
```rust
// Mixing std Condvar blocking with async tokio — this
// blocks the tokio runtime thread.
if let Some(task) = ctx.overflow.pop_blocking(
    Duration::from_millis(ctx.scheduler.config.steal_interval_ms),
) {
    // ...
}
```
When `pop_blocking` is called, it locks the `Mutex` inside `OverflowQueue` and then calls `self.notify.wait_timeout(...)`. If the queue is empty, this function blocks the entire thread that is running the `worker_loop` async task, preventing it from polling other tasks and starving the runtime.
**Fix:** The `OverflowQueue` should be refactored to use `tokio::sync::Notify` or `tokio::sync::Mutex` instead of `std::sync::Mutex` and `std::sync::Condvar`. The worker should await on the `tokio` primitive.

### 5. Priority Queue Inefficiency (Priority Inversion)
**Location:** `queue.rs`, `PriorityTaskQueue::pop_highest`
**Issue:** The current implementation of `pop_highest` suffers from poor lock contention and priority inversion. It iterates through all four buckets sequentially, requiring four separate lock attempts.
```rust
// Priority inversion: iterates from Low(0) to Critical(3). If the Low
// bucket lock is contended, we block before checking Critical.
for i in 0..4 {
    if let Some(task) = self.buckets[i].lock().unwrap().pop_front() {
        // ... found task
        return Some(task);
    }
}
```
