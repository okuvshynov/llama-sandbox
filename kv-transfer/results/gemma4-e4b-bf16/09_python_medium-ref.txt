## 🐞 Critical Bugs and Correctness Issues

### 1. Race Condition in `_log_operation` (Concurrency/Database)
**The Issue:** The `_log_operation` method performs a database write (`self.db.execute` followed by `self.db.commit()`) *without* holding the main thread lock (`self.lock`). When multiple threads call `get` or `put` concurrently, they might attempt to log operations simultaneously. Although SQLite uses internal locks, constantly opening and committing single-row transactions in a high-concurrency system is extremely inefficient and can lead to contention or even deadlocks if connections are managed improperly under load.

**The Fix:**
The database interaction should either:
1. **Be delegated to a single-threaded logging queue/consumer:** This is the best practice for decoupling high-speed cache operations from slow I/O (disk logging).
2. **Be executed while holding the lock (if the log speed is acceptable):** Since the logger is fast, for simplicity, you could wrap the logging in a `with self.lock:` block, but this severely limits cache throughput.

*Recommendation: Implement a dedicated logging thread/queue.*

### 2. Inconsistent Key Handling (Logic/Bugs)
**The Issue:** The `CacheStore` uses hashing (`_hash_key`) to store entries (`self.entries`), but the log operation (`_log_operation`) and the eviction policies (`LRUEvictionPolicy`, `TTLEvictionPolicy`) operate on the *hashed* key, while the public methods (`get`, `put`, `delete`) use the *original* user key.

In `_log_operation`, you pass the original `key`, but the cache state relies on the hash. This is generally okay, but it means the stored logs might not reflect the internal hash structure of the cache efficiently.

More critically, the public interface relies on the user providing the original key, but the internal state uses the hash. This separation is maintained, but it increases cognitive load and risk of error.

**The Fix:**
Ensure all internal state management (LRU, TTL, `self.entries`, size tracking) *strictly* uses the hashed key. The public methods must handle the mapping cleanly. Your implementation already does this correctly, but the lack of a central mechanism for key mapping (like a dictionary mapping original key -> hashed key) adds complexity.

### 3. Potential Threading Issue in `get` (Minor Lock Scope)
**The Issue:** In the `get` method, when an entry is found and expired:
```python
        if entry.expires_at < time.time():
            del self.entries[hashed]
            # ... other cleanup
            self.lock.release()
            return None
```
The code calls `del self.entries[hashed]` while holding the lock, which is correct. However, the subsequent calls to `self.lru.remove(hashed)` and `self.ttl.remove(hashed)` rely on the internal state of those policies. Since these policies are designed to operate on the shared state, holding the lock during their operation is crucial, which you are doing.

**Refinement (Style):** Use `with self.lock:` instead of manual `acquire()`/`release()`. This is safer and prevents accidental deadlocks if exceptions are raised.

## 🐌 Performance and Scalability Issues

### 4. LRU Eviction Policy (Performance)
**The Issue:** The `LRUEvictionPolicy` uses a standard Python `list` (`self.access_order`) to track the order.
*   `record_access(key)`: If the key is found, `self.access_order.remove(key)` is $O(N)$, where $N$ is the number of cached items. Appending is $O(1)$.
*   `get_eviction_candidate()`: This is $O(1)$.

When the cache is large, $O(N)$ removal on every access (hit) is a major performance bottleneck. A true, performant LRU implementation must use a **doubly linked list** combined with a **hash map (dictionary)** for $O(1)$ lookup and removal.

**The Fix:**
Replace `LRUEvictionPolicy` with a structure similar to `collections.OrderedDict` (or implement a custom linked list structure).

### 5. TTL Eviction Policy (Performance)
**The Issue:** The `TTLEvictionPolicy` iterates over *all* stored entries (`self.entries`) every time `get_expired_keys()` is called, performing a linear scan ($O(N)$). In a large cache, this periodic cleanup becomes very expensive.

**The Fix:**
Do not use a dictionary for TTL tracking if you intend to scan it linearly. Instead, use a **Min-Heap (Priority Queue)**, keyed by `expires_at`.
1. When putting an item, insert `(expires_at, key)` into the heap ($O(\log N)$).
2. During cleanup, repeatedly pop items from the heap until the top item's expiry time is in the future ($O(k \log N)$, where $k$ is the number of expired items).

### 6. Serialization Cost (`_compute_size`) (Performance)
**The Issue:** The size calculation uses `len(json.dumps(value))`. This requires serializing the entire object *every single time* a `put` operation is called, just to determine its size. For large objects, this is a CPU-intensive operation that adds unnecessary overhead.

**The Fix:**
If the cache implementation requires size tracking, the object size should ideally be provided by the caller or, if it must be computed, the serialization should happen *once* during the `put` process, and the resulting byte size should be stored directly in the `CacheEntry` metadata.

### 7. Database I/O (Performance)
**The Issue:** The `_log_operation` method performs a synchronous `db.execute()` and `db.commit()` inside the critical path of `get` and `put`. Database I/O is orders of magnitude slower than memory access. This guarantees that your cache throughput will be limited by disk write speed.

**The Fix:** (As noted in point 1) **Decouple logging.** Use a background thread and a queue (`queue.Queue`) to send log entries for asynchronous persistence.

## 📐 Design and Maintainability Issues

### 8. Tight Coupling (Design)
**The Issue:** `CacheStore` is highly coupled to the internal implementations of `LRUEvictionPolicy` and `TTLEvictionPolicy`. The cache relies on two separate eviction policies (`lru` and `ttl`) working simultaneously, which is complex and difficult to reason about. When an item expires via TTL, the cache must manually remove it from the LRU structure, which is fragile.

**The Fix:**
Design the cache to use a unified entry structure and a single coordination mechanism. The eviction mechanism should check multiple criteria simultaneously (TTL check, Size check, LRU score) during the eviction process.

A common pattern is to use a *single* structure (e.g., an OrderedDict) that combines the ordering and size metadata, allowing combined LRU/TTL/Size checks simultaneously.

### 9. Missing Thread Safety in Helper Classes (Design)
**The Issue:** `LRUEvictionPolicy` and `TTLEvictionPolicy` are not thread-safe. While `CacheStore` holds the primary lock, if any helper class were to be accessed directly (e.g., for debugging, or if the lock granularity were changed), these classes would fail immediately under concurrent access.

**The Fix:**
If these classes must remain separate, they should either implement their own internal locks, or all interaction with them *must* occur strictly under `CacheStore`'s lock.

### 10. Lack of State Management for Periodic Cleaner (Robustness)
**The Issue:** `PeriodicCleaner` relies on `time.sleep(self.interval)`. If the `cache.cleanup_expired()` method takes longer than `self.interval` (e.g., during heavy load or a large cleanup operation), the cleaner will start the next cycle late, potentially leading to unpredictable cleaning intervals
