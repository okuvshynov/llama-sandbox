## 🧠 Program Explanation

The purpose of an LRU Cache is to store a limited number of data items (key-value pairs) in a way that, when the cache reaches its maximum capacity, the item that has gone the longest time without being accessed (the "Least Recently Used" item) is automatically removed to make room for a new item.

The program manages this cache to provide two primary functionalities:

1.  **`get(K key)`:** Retrieves a value associated with a key. If the key exists, it is retrieved, and critically, its usage is marked as recent by moving it to the "most recently used" position within the cache.
2.  **`put(K key, V value)`:** Inserts or updates a key-value pair.
    *   If the key already exists, its value is updated, and it is moved to the MRU position.
    *   If the key is new and the cache is full, the item at the LRU end is evicted before the new item is inserted.
    *   If the key is new and the cache has space, it is simply added to the MRU position.

### How the Eviction Policy Works:

The cache relies entirely on ordering. The structure is set up so that the nodes nearest the `head` are the Most Recently Used (MRU), and the nodes nearest the `tail` are the Least Recently Used (LRU).

*   **Access/Update (`get` or `put` on existing key):** The item is removed from its current spot and re-inserted right after the `head` (`addToFront`).
*   **Eviction (`put` on new key when full):** The item immediately preceding the `tail` (`tail.prev`) is identified, removed from the linked list, and subsequently deleted from the `HashMap`.

## 🏗️ Data Structure Description

This implementation uses a sophisticated hybrid data structure composed of two parts working together: a **HashMap** and a **Doubly Linked List (DLL)**.

### 1. The Doubly Linked List (DLL)

The DLL is responsible for tracking the **order of usage**.

*   **Nodes:** Each `Node` object holds a `key`, a `value`, and pointers (`prev` and `next`) to its neighbors.
*   **Ordering:** The list is circular and managed by dummy `head` and `tail` nodes.
    *   **Head $\rightarrow$ MRU** (Most Recently Used)
    *   **Tail $\leftarrow$ LRU** (Least Recently Used)
*   **Function:** When an item is accessed, the list allows us to efficiently remove it from the middle (via the pointers) and place it at the front (MRU). This ensures $O(1)$ time complexity for moving items.

### 2. The HashMap (`java.util.HashMap`)

The HashMap is responsible for **fast lookups**.

*   **Mapping:** It stores the mapping from the `key` to the actual `Node` object.
*   **Function:** When a `get(key)` request comes in, the HashMap allows the program to instantly locate the specific `Node` object associated with that key in $O(1)$ time, without having to traverse the entire linked list.

### Synergy (Why both are needed)

The combination of these two structures achieves the necessary performance profile for an LRU cache:

| Operation | Time Complexity | How it's achieved |
| :--- | :--- | :--- |
| **Lookup (`get`)** | $O(1)$ | The `HashMap` instantly finds the node. |
| **Update/Move Node** | $O(1)$ | The DLL allows instant removal and re-insertion using pointer manipulation. |
| **Eviction (`put` on full cache)** | $O(1)$ | The DLL ensures the LRU node (at `tail.prev`) is found immediately, and the `HashMap` is updated instantly. |
