## 🐞 Critical Bugs (Runtime & Concurrency)

### 1. Uninitialized Map (Panic)

**Location:** `NewCounter()` and `Counter.Increment()`
**The Bug:** The `Counter` struct initializes its `counts` field as `nil` (the zero value for a map). In `Increment`, the line `c.counts[key]++` attempts to modify the map. When you write to a `nil` map, Go will trigger a runtime panic (`panic: assignment to entry in nil map`).

**Consequence:** The server will crash immediately upon the first request to `/inc`.

**How to Fix:** Initialize the map in `NewCounter`.

```go
func NewCounter() *Counter {
	return &Counter{
		counts: make(map[string]int), // <-- FIX
	}
}
```

### 2. Exposing Internal State (Severe Data Race/Violation of Encapsulation)

**Location:** `Counter.GetAll()`
**The Bug:** The `GetAll` method acquires the lock, but then it returns a direct reference (a copy of the map header, pointing to the underlying data structure) to the private internal map (`c.counts`).

```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts // <-- Returns the reference
}
```

**Consequence:** A caller (like `handleGetAll`) receives this reference. Although the lock is released, the caller now holds a pointer to the sensitive, shared data. If the caller modifies this returned map (e.g., `data["new_key"] = 100`), they are directly modifying `c.counts` *without* holding the mutex, leading to a classic, undetectable data race and corruption of the internal state.

**How to Fix:** Return a *copy* of the map, ensuring external modifications don't affect the internal state.

```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
	
	// Create a safe copy of the map
	copiedMap := make(map[string]int)
	for k, v := range c.counts {
		copiedMap[k] = v
	}
	return copiedMap // <-- Return the copy
}
```

### 3. Data Race in Read Operation (Concurrency Bug)

**Location:** `handleGetAll()`
**The Bug:** Even if `GetAll()` were corrected to return a copy, the standard Go HTTP handler execution might still cause issues depending on how the data is retrieved. However, when using the buggy `GetAll()`, the JSON encoder is reading the shared map *after* the lock has been released.

```go
// In handleGetAll:
data := counter.GetAll() // Lock is released here (due to defer)
json.NewEncoder(w).Encode(data) // Reading the map *outside* the lock
```

**Consequence:** If two goroutines call `/all` and `/inc` concurrently, the reading goroutine might encounter a situation where the map is being actively resized or modified by the writing goroutine, leading to inconsistent reads or potential crashes.

**How to Fix:** Ensure that all access to the underlying map structure is protected by the mutex. (Fixing Bug #2 solves this by returning a safe copy.)

## ⚠️ Design and Robustness Issues (Code Quality)

### 4. Inconsistent Response Handling (Design Flaw)

**Location:** `handleIncrement()`
**The Bug:** The handler prints the resulting count directly to the response writer using `fmt.Fprintf`.

```go
// In handleIncrement:
val := counter.Increment(key)
fmt.Fprintf(w, "%d", val) // <-- Raw text response
```

**Consequence:** This is poor API design. A modern API endpoint should return a structured response (like JSON) and explicitly set the HTTP status code. While the current logic works, if the client expects JSON, this response will fail parsing.

**How to Improve:** Respond with a JSON object indicating success and the new count.

```go
func handleIncrement(w http.ResponseWriter, r *http.Request) {
    // ... validation ...
	val := counter.Increment(key)

    // Set Content-Type header
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)

    // Respond with JSON
	json.NewEncoder(w).Encode(map[string]int{"key_count": val})
}
```
