## 🐞 General & Scheduler Core (`scheduler.go`)

### 1. Race Condition in Task Retry (Critical)
The task retry logic uses a `go func()` inside `CompleteTask` to handle the delay and re-queueing. This function attempts to modify shared state (`task.State` and `heap.Push(&s.queue, task)`) *outside* of the main `s.mu.Lock()` scope.

**Issue:**
While the anonymous function *does* acquire the lock before modifying state (`s.mu.Lock()` inside the retry goroutine), the process of updating the task state and pushing to the heap is an asynchronous side effect of a synchronous lock held by `CompleteTask`. This asynchronous push could race with other ongoing operations (like `checkWorkerHealth` or another `CompleteTask` call) if the state transitions are not fully protected across the entire retry cycle.

**Fix Suggestion:**
Instead of pushing the task immediately to the queue in a goroutine after a delay, the scheduler should maintain a separate, guarded data structure (e.g., a priority queue or a map) for "Pending Retries." A dedicated background goroutine should monitor this retry queue and, upon timeout, acquire the lock and push the task into the main heap.

### 2. Dependency Graph Update Logic is Inefficient/Flawed
The dependency checking in `SubmitTask` and `CompleteTask` is inefficient and potentially incorrect.

**Issue:**
In `SubmitTask`, you check if *all* dependencies are in `s.completed`. If they are not, you store the task in `s.tasks` but do **not** add it to the dependency graph (`s.depGraph`). This is inconsistent. The dependency graph is meant to track who depends on whom, but here, the logic is only partially handled.

In `CompleteTask`, when a task completes, you iterate over `s.depGraph[taskID]`. This is correct, but the subsequent re-evaluation of readiness involves iterating over *all* dependencies of the dependent task (`reqID` loop), which is $O(D^2)$ where $D$ is the number of dependencies for the dependent task, if dependencies are complex.

**Fix Suggestion:**
1. **In `SubmitTask`:** When adding a dependency relationship (A depends on B), record this connection: `s.depGraph[B] = append(s.depGraph[B], A.ID)`. If the task is not immediately runnable, keep it in a separate `pendingDependencies` map, keyed by Task ID, tracking how many prerequisites it still needs.
2. **In `CompleteTask`:** When task $T$ completes, check its dependents $D$ listed in `s.depGraph[T]`. For each dependent $D$, decrement a counter that tracks unresolved dependencies for $D$. If this counter hits zero, $D$ is ready and should be pushed to the queue. This changes the complexity from $O(D^2)$ to $O(D)$.

### 3. Worker Health Check Risk (Design/Performance)
In `checkWorkerHealth`, when a worker is deemed dead, its task is immediately requeued (`heap.Push(&s.queue, task)`).

**Issue:**
If the scheduler is under high load, `checkWorkerHealth` might trigger repeatedly, causing a cascade of requeue operations, potentially leading to queue flooding or inconsistent state updates across multiple health checks before the system stabilizes. Furthermore, there is no handling for **which** worker should pick up the failed task (the system currently just throws it back into the general queue).

**Fix Suggestion:**
1. **Introduce a Backoff/Deduplication:** When a worker is marked dead, implement a short cooldown period before requeuing the task, preventing immediate thrashing.
2. **Task State Transition:** Instead of immediately pushing to `s.queue`, set the task state to `TaskPending` (if it wasn't already) and perhaps move it to a specific "failed/unassigned" state map, which the main dispatch loop can pick up, allowing for more controlled requeueing logic (e.g., retries, escalation).

### 4. Task ID Generation (Minor Bug/Performance)
The ID generation in `SubmitTask` relies on `time.Now().UnixNano()` combined with an atomic counter.

**Issue:**
Relying solely on `UnixNano()` guarantees uniqueness only across extremely long time periods or processes. If two tasks are submitted in the exact same nanosecond (which can happen under high load), the counter might not increment fast enough, leading to collisions, though the `atomic.AddInt64` helps mitigate this significantly. It's a small risk, but for critical distributed IDs, relying on a robust sequence generator or UUIDs is safer.

**Fix Suggestion:**
Use a UUID library (e.g., `github.com/google/uuid`) for task IDs to guarantee global uniqueness regardless of timing or clock skew.

## 🔐 Concurrency and Thread Safety

### 5. Worker Status Update (Race Condition)
In `CompleteTask`, you modify `worker.Load` and `worker.CurrentTask` based on the `worker` reference derived from `s.workers[task.WorkerID]`.

**Issue:**
Although `CompleteTask` holds `s.mu.Lock()`, the `WorkerStatus` fields like `Load` and `CurrentTask` are simple structs. If another operation were to access or modify these fields (e.g., a hypothetical external status monitor accessing `s.workers` outside of the scheduler lock), a data race could occur. More importantly, the logic relies on the assumption that `s.workers` remains stable during the execution of the task completion handler, which is generally true because `s.mu` is held.

However, the use of `atomic.AddInt64` for `TasksCompleted` and `TasksFailed` *simultaneously* with synchronous map updates (`worker.Load--`) is inconsistent. You are mixing synchronized mutex protection (for Load/CurrentTask) and atomic operations (for counters).

**Fix Suggestion:**
**Consistency:** Since you hold `s.mu.Lock()`, all modifications to `WorkerStatus` (including the counters: `TasksCompleted` and `TasksFailed`) should be done normally, without `atomic.AddInt64`, for consistency and simpler reasoning. Only use `atomic` for counters if you absolutely must access them outside the main scheduler lock.

### 6. Task Queue Mutation in Retry Goroutine (Design)
As noted in #1, the asynchronous retry re-pushes to the queue. While the lock is acquired, relying on external, delayed goroutines to manage the core state of the priority queue is fragile.

**Design Improvement:**
The Scheduler's primary job should be state machine transitions, not asynchronous delays. State management (e.g., `TaskRetrying`) should occur under the lock, and the delay should be managed by a dedicated timer/watcher process, not a raw `time.Sleep` in a detached goroutine.

## ⏱ Performance and Scalability

### 7. Lock Granularity in Workerpool (Performance)
The `WorkerPool` uses a single `sync.RWMutex` (`p.mu`) to protect the entire `p.workers` slice.

**Issue:**
Any operation that touches the list of workers—`addWorker`, `removeWorker`, `autoScale`, `logMetrics`—requires acquiring and releasing this lock, even if the operation only targets a single worker's atomic stats. In a highly scaled system with many workers frequently monitoring/logging/scaling, this single mutex can become a contention point.

**Fix Suggestion:**
If the pool grows very large, consider partitioning the worker list (e.g., sharding by worker ID prefix) or using `sync.Map` if lookups are the primary access pattern. For this current scale, the mutex is probably acceptable, but be aware it will be the bottleneck for scaling logic.
