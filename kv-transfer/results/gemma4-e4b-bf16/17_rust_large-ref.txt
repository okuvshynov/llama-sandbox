## 🐛 Task Module (`task.rs`)

### 1. Critical Bug: Circular Arc and Memory Leak (Dependency Handling)
The mechanism for managing task dependencies creates an immediate and non-recoverable memory leak.

*   **The Issue:** When using `Task::with_dependencies`, you clone the `Arc<TaskInner>` of each dependency (`dep_inners`). You then modify `dep.dependents` to hold a `clone()` of the new task (`inner`). This creates a strong reference cycle: `TaskInner_A` holds a strong reference to `TaskInner_B` (via `dependencies`), and `TaskInner_B` holds a strong reference back to `TaskInner_A` (via `dependents`). Since neither is wrapped in a `Weak` reference, the strong reference count will never reach zero, leading to a permanent memory leak.
*   **Suggestion/Fix:** The `dependents` field in `TaskInner` **must** be changed from `Mutex<Vec<Arc<TaskInner>>>` to `Mutex<Vec<Weak<TaskInner>>>`. This allows the dependency relationship to exist without preventing the tasks from being dropped when all external references (like those held by the scheduler) cease to exist.

### 2. Design Flaw: Task State Management and TaskHandle (Result Retrieval)
The `TaskHandle` setup is fundamentally flawed regarding result retrieval.

*   **The Issue:** In `Task::new`, the `TaskHandle` is initialized with `result_rx: None`. The `TaskInner` does not have a place to store the `oneshot::Sender` (`result_tx`) that corresponds to the `result_rx`. When the task completes, there is no mechanism in `TaskInner` to send the `TaskResult` back to the waiting `TaskHandle`. Furthermore, `TaskHandle::wait_result` relies on `result_rx` being present, which it never is.
*   **Suggestion/Fix:**
    1.  Modify `TaskInner` to include an optional `Sender<TaskResult>`.
    2.  In `Task::new`, create the `oneshot` channel, store the sender in `TaskInner`, and pass the receiver to the `TaskHandle`.
    3.  When the task completes, the worker must access the stored sender and use it to send the final result, updating the `TaskState` accordingly.

### 3. Performance/Design: Global ID Counter
*   **The Issue:** While safe, using a global `static` atomic counter (`TASK_ID_COUNTER`) means all tasks, regardless of whether they are high-priority or local to a worker, rely on a single contention point for ID generation.
*   **Suggestion/Fix:** For a high-performance scheduler, consider delegating ID generation to the `SchedulerInner` or using thread-local counters (though this adds complexity). Given the scale, this is a minor point, but globally contested atomics are bottlenecks.

## ⚙️ Queue Module (`queue.rs`)

### 4. Critical Bug: Lock-Free Deque Implementation (Data Race / UB)
The `TaskDeque::steal` implementation is critically unsafe and prone to data races and Undefined Behavior (UB).

*   **The Issue:**
    1.  `let task_ptr = &self.buffer[idx] as *const Option<Task>;`
    2.  `let task = unsafe { std::ptr::read(task_ptr) };`
    3.  The entire `Vec<Option<Task>>` is *not* atomic. The owner (the local worker/popper) could be in the middle of writing to `buffer[idx]` concurrently with the stealer reading it, leading to torn reads or access to partially initialized data.
    4.  The cleanup logic in the `else` branch of `steal` (`std::mem::forget(task);`) is required because `ptr::read` took ownership, but since the CAS failed, the task was not successfully moved out, leading to memory leakage or UB if not handled perfectly.

*   **Suggestion/Fix:** **Do not attempt to implement a custom lock-free ring buffer using raw pointers and `Vec` in Rust without heavy use of `MaybeUninit` and memory ordering fences.** The complexity is too high and dangerous.
    *   **Primary Fix:** Use a robust, tested external library for lock-free queues (e.g., crossbeam or a dedicated implementation if extreme performance is needed).
    *   **Practical Fix:** Since the `SharedTaskDeque` wrapper already uses `Mutex`, eliminate the buggy `TaskDeque` entirely and refactor all queue interactions to go through the mutex, which is much safer and provides acceptable performance unless profiling proves otherwise.

### 5. Design Flaw: Redundant Queue Wrapper
*   **The Issue:** The `SharedTaskDeque` wrapper uses a `Mutex` around the entire `TaskDeque`. If the goal is maximum concurrency (the whole point of work stealing), locking the entire deque defeats the purpose. However, given the bugs in `TaskDeque`, this is currently a necessary evil.
*   **Suggestion/Fix:** If you stick with the mutex wrapper (`SharedTaskDeque`), rename it to emphasize that it is a *thread-safe* wrapper, not a *lock-free* queue. If you pursue true lock-free performance, you must fix the implementation of `TaskDeque`.

### 6. Design Flaw: Priority Queue Inefficiency (Priority Inversion)
*   **The Issue:** The `PriorityTaskQueue::pop_highest` function iterates through all four priority buckets (`for i in 0..4`). While small (4 buckets), this structure is susceptible to priority inversion: if the `Low` bucket is heavily contended and locking it takes a long time, a high-priority task in the `Critical` bucket must wait for the `Low` bucket lock to be released, even if the Critical bucket is immediately available.
*   **Suggestion/Fix:** If the number of priorities is small (like 4), a single synchronized priority queue (e.g., `BinaryHeap` protected by a single `Mutex`) might offer better predictable latency, sacrificing fine-grained contention for correctness and simplicity. If you must keep buckets, use non-blocking locks (like `tokio::sync::Mutex`) or ensure the lock acquisition logic checks all buckets atomically if possible, though this is complex.

## 🧵 Worker Module (`worker.rs`)

### 7. Critical Bug: Dropping Pending Futures (State Loss)
This is the single most severe logical bug regarding async execution.

*   **The Issue:** In `execute_task`, if `pinned.poll(&mut cx)` returns `Poll::Pending`, the future is immediately dropped:
    ```rust
    // Future is dropped here — if the task yielded (e.g.,
    // awaiting I/O), we lose it entirely. Should store it
    // back in task.inner.future.
    *task.inner.state.lock().unwrap() = TaskState::Pending;
    ```
    If the future yields (which is how all I/O tasks work), the `Future` object is destroyed, its internal state (where it was awaiting) is lost, and it can never be resumed. The task becomes permanently stalled.
*   **Suggestion/Fix:** When `Poll::Pending` is returned, the future **must** be stored back into `task.inner.future`, and the `Waker` that triggered the poll must be used to ensure the scheduler/task management system can re-poll it later (e.g., when the I/O completes). This requires replacing the `Mutex<Option<BoxFuture<TaskResult>>>` with a structure that handles re-insertion and tracking the yielded state.

### 8. Performance Issue: Immediate Re-enqueueing of Unready Tasks
*   **The Issue:** If `task.is_ready()` is false (due to dependencies), the task is immediately pushed back onto the local queue:
    ```rust

    // unfinished dependencies.

    return;
    ```
