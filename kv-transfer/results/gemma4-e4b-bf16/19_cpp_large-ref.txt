## 1. Slab Allocator / Memory Pool (`SlabAllocator`)

The slab allocator appears reasonably well-formed for a simple memory pool, but there are specific correctness and efficiency concerns.

### âš ď¸Ź Critical Bug: Alignment and Usage Mismatch
**Issue:** The `SlabBlock` is aligned to `CACHE_LINE_SIZE` (64 bytes). However, when a block is used to store a user object (`object_size_`), the allocator relies on `std::aligned_alloc(CACHE_LINE_SIZE, alloc_size)`. The allocated block size is `object_size_`. If `object_size_` is smaller than `CACHE_LINE_SIZE`, the memory allocation and subsequent casting are fine, but the `SlabBlock` structure itself (which contains `next` and padding) is only used when the block is **free**.

If `object_size_` is less than `sizeof(SlabBlock)`, the internal structure of `SlabBlock` (which uses the initial bytes to store `next`) will be overwritten or inaccessible when the memory is in use by the user.

**Fix:** Ensure that when a block is allocated, it is large enough to hold the necessary metadata (`SlabBlock::next`) *or* adjust the usage logic.
A common pattern for slab allocators is to always size the memory chunks to accommodate the maximum required metadata size, which is `sizeof(SlabBlock)`.

```cpp
// In SlabAllocator constructor:
object_size_ = std::max(object_size, sizeof(SlabBlock)); 
// This guarantees that even when the block is in use, we haven't lost the potential for
// using the block structure if we ever needed to track it further.
// However, since the user is supposed to write their own object,
// if they write a small object, we might lose the pointer if we rely on casting.
// A better solution is ensuring the object size is always large enough to hold the pointer
// when it's recycled, but since the allocator only uses the block metadata when FREE,
// let's adjust the object size calculation:

SlabAllocator(size_t object_size, size_t initial_capacity = 1024)
    : object_size_(std::max(object_size, sizeof(SlabBlock*) + 1)), // Ensure at least room for a pointer
      // ... rest of constructor
```
*Self-Correction/Refinement: Since the allocated memory block is only used for `next` when FREE, and used for user data when IN USE, the current structure is *only* safe if the user data structure is guaranteed to be placed starting at the beginning of the block.*

### âŹ± Performance Issue: Lock Contention
**Issue:** The entire `allocate()` and `deallocate()` logic is wrapped in a single `std::mutex`. This is a highly contended global lock. If the system uses many threads (as in the benchmark), this mutex will become a major performance bottleneck, negating the benefits of a memory pool for high-concurrency scenarios.

**Fix:** Implement thread-local caching (TLC). Each thread should maintain a small, lock-free or locally locked list of recently freed blocks (`per_thread_cache`). Allocation/deallocation first checks the TLC, and only if the TLC is empty/full does it interact with the global `free_list_` protected by the mutex.

## 2. Hazard Pointer Manager (`HazardPointerDomain`, `HazardPointerGuard`)

Hazard Pointers are inherently complex. The implementation attempts a basic lock-free design, but there are crucial memory ordering and design issues.

### âš ď¸Ź Critical Bug: Lifetime Management of Records (Memory Leak Risk)
**Issue:** When `acquire_for_current_thread()` fails to find a reusable record, it calls `new HazardPointerRecord()`. However, there is no mechanism in the `HazardPointerDomain` destructor or in the usage pattern to ensure these dynamically allocated records are properly cleaned up if a thread dies or fails to release its reference.

**Fix:** Use smart pointers (`std::unique_ptr` or a custom pool) for `HazardPointerRecord`s, or ensure a dedicated thread shutdown routine explicitly cleans up the record associated with that thread ID.

### âš ď¸Ź Critical Bug: Concurrent Access During Reuse Attempt
**Issue:** In `acquire_for_current_thread()`, the reuse loop checks if `p->owner.load(...) == empty`. If it is empty, it attempts a `compare_exchange_strong` to claim the record. This logic is sound *if* the record was truly released completely (i.e., all pointers were set to `nullptr`). The code does set pointers to `nullptr` in `release_record`, but relying on `std::memory_order_relaxed` throughout the entire state machine (owner check, pointer check, CAS) is dangerous without a stronger synchronization barrier to guarantee visibility of the pointer nullification across all cores before the CAS succeeds.

**Fix:** While using `relaxed` often works in HP implementations if the state transitions are carefully modeled, for safety, the transition logic (owner -> claims -> use) should use stronger ordering, typically `release` on setting the new owner and `acquire` when loading it.

### âŹ± Performance Issue: Record Scanning and Overhead
**Issue:** The current design keeps a linked list (`head_`) of all allocated records. In `scan()`, you must iterate over the entire list (`for (HazardPointerRecord* p = head_.load(...)`). If many threads are active and generating many retired pointers, this list grows, making the scan operation $O(N_{threads} \times N_{pointers})$, which is poor performance.

**Fix:** A more advanced implementation might use a specialized data structure (e.g., a dynamically sized array or a thread-local, lazily merged structure) to track active records, minimizing the scan overhead.

## 3. Lock-Free Hash Map (`LockFreeHashMap`)

This is the most complex component, involving linear probing, atomics, and reclamation (Hazard Pointers). This area contains multiple serious bugs related to memory visibility and correctness in concurrent algorithms.

### âš ď¸Ź Critical Bug: Hash Collision Handling in Resize (Race Condition)
**Issue:** The `resize()` function iterates over the old table. For each entry, it tries to insert it into the `new_table` using a simple CAS loop: `if (new_table[probe].key.compare_exchange_strong(expected, k, ...))`.
Since `resize()` is called *without* any external locking, multiple threads might simultaneously trigger `resize()` or attempt to write to the old table slots (`table_[i]`). While the main pointers (`table_`) are swapped *after* rehashing, the read operations by concurrent `insert` or `find` operations are still happening on the old `table_`.

More importantly, if multiple threads call `resize()` concurrently, they will race to acquire and delete the old table, leading to use-after-free and corruption.

**Fix:** Resize operations in a high-performance concurrent map require either:
1. **Global Lock:** (Sacrifices lock-free property).
2. **Non-Blocking Resize Protocol:** The map needs a mechanism (like a version counter or a global CAS on the table pointer) to ensure only one thread performs the swap and cleanup at a time, and concurrent operations must handle the possibility that the table they are reading is outdated.
