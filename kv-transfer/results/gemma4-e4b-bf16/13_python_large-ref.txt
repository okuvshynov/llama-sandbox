## 🐞 Bugs & Data Integrity Issues

### 1. Floating Point Arithmetic Errors (`models.py`)
**Issue:** Financial calculations in Python using `float` types (`price`, `total`, `subtotal`) are inherently prone to floating-point precision errors, which is unacceptable for financial microservices.
**Example:** `Order.total()`, `Product.effective_price()`.
**Fix:** Use Python's `decimal` module for all monetary calculations. This provides arbitrary precision required for handling currency accurately.

**Code Suggestion (in `models.py`):**
```python
from decimal import Decimal, getcontext
getcontext().prec = 28 # Set high precision

# Change types from float to Decimal where money is involved
@dataclass
class Product:
    # ...
    price: Decimal # Change here
    # ...

@dataclass
class OrderItem:
    # ...
    unit_price: Decimal # Change here
    # ...

# Update methods to handle Decimal objects accordingly.
```

### 2. Incomplete Data Persistence (`inventory.py`)
**Issue:** The `_dict_to_product` method only saves a minimal subset of the `Product` attributes (ID, name, price, stock_count, category). Any complex data fields like `description`, `metadata`, `tags`, or `discount_percent` are silently dropped when loading the inventory from the JSON file.
**Fix:** Ensure that `_dict_to_product` maps *all* relevant fields from the persisted dictionary back into the `Product` dataclass.

### 3. Race Condition in Reservation Checking (`inventory.py`)
**Issue:** The `check_availability` method calls `_get_reserved_quantity`. While `reserve_stock` is wrapped in a `self.lock`, the logic inside `check_availability` reads the state *before* it acquires the necessary lock for the reservation check. This is fine, but the repeated calculation of reserved quantity is inefficient (see Performance). More critically, the check relies on reading reserved items globally, which is complex and error-prone.
**Fix:** When checking availability, ensure you are accessing and evaluating the reserved quantity *within* the protection of the primary lock to ensure the state read is consistent.

### 4. Analytics Funnel/Cohort Logic (`analytics.py`)
**Issue:** The `analytics.py` methods (`analyze_funnel`, `cohort_analysis`) assume perfect, sequential, and immediate logging. For a real microservice, events are asynchronous and might arrive out of order or with delays. The current logic doesn't robustly handle delayed or re-ordered events, leading to incorrect funnel conversions or cohort grouping.
**Fix:** Implement strict time-window checks and require robust sequencing metadata (e.g., a monotonic event ID or sequence number) when recording events to allow for correct re-sorting and analysis.

## 🐌 Performance Issues & Scalability

### 5. Synchronous Disk I/O and Locking (`inventory.py`)
**Issue:** The `_save_to_disk` method, which writes the entire inventory state to JSON, is called inside `add_product`, `commit_reservation`, and `restock`. Because these methods are protected by `self.lock`, **any single inventory transaction forces the entire service to halt and perform synchronous disk I/O.** This is a major bottleneck and makes the system non-scalable.
**Fix:**
1.  **Decouple Persistence:** Inventory should use a proper database (SQL, NoSQL) instead of flat files.
2.  **Asynchronous Writing:** If file persistence is strictly required for proof-of-concept, implement a background worker thread or queue that periodically flushes changes to disk, releasing the main lock quickly.
