## 🚨 Critical Bugs & Race Conditions

### 1. Race Condition in `WorkerPool.results` Access (The Most Critical Issue)

The `WorkerPool.results` map stores the state of all jobs. While some methods like `GetJob` and `Submit` use `wp.mu` to protect access, the `WorkerPool.stats` handler is inherently flawed.

**Problem:**
In `main`'s `/stats` handler, you correctly lock `wp.mu` to access `pool.results` and `pool.jobQueue`. However, within the `worker.processJob` function, jobs are modified (Status, Result, DoneAt) while potentially being held in the job queue or being processed. Critically, the `results` map is only modified *during* `Submit` (when it's created). When a job transitions state inside `processJob`, **the status update happens on the pointer inside the map, but other parts of the system (like the `/stats` handler) need to read the map structure safely.**

While the workers are modifying the *contents* of the `Job` struct (which is safe if the worker is the only one writing to that specific `Job` pointer), the `stats` handler is trying to iterate over this map *while* jobs are being transitioned, and more importantly, if the map itself were being modified (e.g., deleting finished jobs), this would be a major issue.

**The real issue here is the scope and granularity of the lock.** The lock in `WorkerPool.mu` is global, protecting the map, but accessing it while processing means you are exposing potential data inconsistencies if the map structure itself were changed.

**Fix:**
The current usage is *minimally* safe regarding map structure (since jobs are never deleted), but for robust concurrency, **all reads and writes to `wp.results` must be synchronized.**

Furthermore, when iterating over `wp.results` in `/stats`, you should create a copy of the results and statuses under the lock to prevent iteration while external mutations happen, although since `Job` structs are mutable, locking around the entire map iteration is necessary.

### 2. Logic Bug: Job Retries and Re-submission (Potential Deadlock/Queue Flooding)

The retry logic is incorrect and can lead to queue starvation or performance issues.

**Problem:**
When a job fails and needs a retry, `processJob` does the following:

1.  It sleeps (`time.Sleep`).
2.  It sends the job back to the queue (`wp.jobQueue <- job`).
3.  It returns.

This approach is problematic because the worker thread performing the processing **is blocked** for the entire retry delay. If a worker fails a job and sleeps for 5 seconds, that worker thread is useless for 5 seconds, reducing the effective concurrency of the pool.

**Fix:**
The delay (backoff) should be handled **outside** the critical worker thread path. The worker should signal the job's failure, and a separate component (like a scheduler or timer goroutine) should be responsible for waiting and re-submitting the job.

**Minimal Fix (to keep structure):** If you must keep the sleep inside the worker, you must ensure the retry logic doesn't block the entire pool. However, the best design is to push the responsibility of delay to the caller/scheduler.

### 3. Missing Lock in Job State Updates (Design/Consistency Bug)

While the `Job` struct is passed by pointer, the updates to its fields (Status, Result, DoneAt, etc.) happen inside `processJob`. If multiple systems were trying to read/write status outside of `GetJob`, you would have a race.

**Problem:**
The workers are modifying the job state *without* using `wp.mu`. While this is technically safe if the job pointer is only held by one worker at a time, the **system should rely on the central synchronization primitive (`wp.mu`) for any critical state transition.**

**Fix:**
Although overkill for this specific setup where the worker is the sole modifier, for a truly robust system, job state transitions (e.g., moving from "running" to "failed") should ideally involve a lock or an atomic state machine check, especially if multiple endpoints could trigger state changes (e.g., an admin API to force retry).

## ⚙️ Design and Architecture Issues

### 4. Rate Limiter Design (Lack of Context Safety)

The `RateLimiter` implementation is functional but lacks integration safety into the worker pool lifecycle.

**Problem:**
The rate limiter is passed into the worker pool via a closure (`rateLimitMiddleware`). However, there is no mechanism to ensure that if the pool is stopped (`wp.Stop()`), any ongoing rate limiting checks stop cleanly, though this is minor since the rate limiter itself doesn't rely on the pool's context.

**Major Design Flaw:** The rate limiting middleware is applied globally. If the job processor encounters an error, it fails and exits the worker's loop (in case of middleware failure) but the rate limiter still decrements the token if `Allow()` is called successfully. If the job immediately fails and is not retried, this might be fine. But if the rate limit middleware is designed to enforce a limit *per job*, the token consumption should only happen if the job is actually processed successfully or enters the queue successfully.

**Fix:**
Ensure the rate limit check is tied to the successful transition into the active processing pipeline. If the job fails validation before running, the token should potentially be refunded, or the rate limiter should handle the failure gracefully. For simplicity, ensure the worker's state transitions correctly reflect rate-limiting failure.

### 5. Job Submission Contract (Mutability and Safety)

The `Submit` method mutates the input `*Job` struct directly.

**Problem:**
The caller is responsible for creating the job structure, but `Submit` then changes `Status`, sets `CreatedAt`, and potentially sets `MaxRetry`. This mutation is fine, but it means the caller's original job object is permanently modified and placed into the pool's persistent state (`results`). If the caller expected the submitted object to remain pristine, they will be surprised.

**Fix:**
The `Submit` function should clone the incoming job to ensure that the internal state of the `WorkerPool` maintains its own, immutable record of the job, while the pointer to the original caller's job might be used for input validation, but not stored.

### 6. Error Handling in Worker Loop (The `Select` Default Case)

**Problem:**
The `worker` function uses a simple `select` loop:

```go
select {
case <-wp.ctx.Done():
    return
case job := <-wp.jobQueue:
    wp.processJob(id, job)
}
```

If the worker is processing a job (`processJob`) and `wp.Stop()` is called (signaling `wp.ctx.Done()`), the worker will finish its current job and then immediately check the `select`. If the channel is empty, it will detect the context done signal and exit. This is standard and correct.

However, if `wp.Stop()` is called *while* `wp.jobQueue` is empty, the worker correctly exits. If the worker is currently busy running a job, it will complete that job, then check the context, and exit. This is generally acceptable graceful shutdown behavior for processing workers.
