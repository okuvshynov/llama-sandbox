## 🐞 Bugs and Concurrency Issues (Critical)

### 1. Race Condition in `get(K key)` (The "Thundering Herd" Problem)
**The Issue:** The `get()` method uses the cache to load values if they are missing or expired.
```java
// Inside get()
// ... (lock released)
metrics.recordMiss();
V value = loader.apply(key); // <-- Expensive blocking call happens here outside the lock
if (value != null) {
    put(key, value); // This acquires the lock again
}
return value;
```
If multiple threads simultaneously call `get(key)` and the key is absent or expired, they all proceed past the initial check, execute `loader.apply(key)`, and independently try to load the data, leading to redundant, expensive operations (the "Thundering Herd" problem).

**The Fix:** Use a synchronization mechanism (like a `ConcurrentHashMap` combined with `computeIfAbsent` logic, or a custom locking wrapper) to ensure only one thread computes the value for a given key while others wait for the result.

### 2. Inconsistent State Management Between `store` and `accessOrder`
**The Issue:** The `accessOrder` (a `LinkedList`) is intended to track LRU order, but its manipulation is fragile.
*   **`touchEntry(key)`:** This method manually removes and adds the key to the `LinkedList`. Since `accessOrder` is a standard `LinkedList`, `accessOrder.remove(key)` takes $O(N)$ time, where $N$ is the cache size. This is a significant performance bottleneck inside a frequently called method (`get()`).
*   **`evictExpired()`:** When an entry expires, the code calls `accessOrder.remove(entry.getKey())`. This again requires an $O(N)$ scan of the `LinkedList`.

**The Fix:** To achieve $O(1)$ lookup and $O(1)$ ordering updates (which is standard for an efficient LRU/LFU cache), the `LinkedList` must be replaced with a `LinkedHashMap` configured for access order, or a combination of `HashMap` and a custom Doubly Linked List. Since the cache requires TTL checks, using a custom structure might be necessary, but the $O(N)$ `LinkedList` removal must be eliminated.

### 3. `CacheEntry.getValue()` vs. `CacheEntry.isExpired()`
**The Issue:** The `CacheEntry` class attempts to manage both the expiration time (`ttlMs`) and the last access time (`lastAccessedAt`), but the logic only checks expiration based on `createdAt`.
```java
// Inside CacheEntry
boolean isExpired() {
    return System.currentTimeMillis() - createdAt > ttlMs; // Uses createdAt
}
V getValue() {
    lastAccessedAt = System.currentTimeMillis(); // Updates lastAccessedAt
    return value;
}
```
If the cache were designed to be a *sliding window* cache (where TTL resets on access), the check should be `System.currentTimeMillis() - lastAccessedAt > ttlMs`. Since the code only uses `createdAt`, the `lastAccessedAt` field is functionally useless for expiration purposes in the current implementation.

**The Fix:**
1. **If TTL is fixed (Time Since Creation):** Remove `lastAccessedAt` entirely.
2. **If TTL is sliding (Time Since Last Use):** Change `isExpired()` to use `lastAccessedAt`, and ensure that `getValue()` updates it correctly *before* the check (or during the check).

### 4. Memory Leak Risk in `warmUp`
**The Issue:** The `warmUp` method creates a new `ExecutorService` (`pool`) inside the method and submits tasks. However, it never calls `pool.shutdown()` or `pool.close()`. While the pool might eventually be garbage collected, relying on GC is dangerous and can lead to resource leaks (active threads/tasks lingering).

**The Fix:** Always explicitly manage the lifecycle of executors. Add `pool.shutdown()` after the loop finishes.

## 🐢 Performance Issues

### 5. Synchronization Bottleneck (The Global Lock)
**The Issue:** The entire cache is protected by a single `ReentrantLock`. Any operation (`get`, `put`, `remove`, `evictExpired`, `snapshot`) requires acquiring this lock. In a highly concurrent environment, this single global lock becomes a major contention point, serializing all cache operations and severely limiting throughput.

**The Fix:**
1. **Sharding/Striping:** Instead of one map, use an array of smaller, independent maps (shards), each protected by its own lock. This allows concurrent access to different parts of the cache.
2. **Optimistic Concurrency:** For read-heavy scenarios, consider using `ReadWriteLock`s, allowing multiple readers concurrently but ensuring exclusive writes.

### 6. Expensive `accessOrder` Manipulation (Redundant $O(N)$ Operations)
*(See Issue #2)* The reliance on `LinkedList.remove(key)` inside `touchEntry` and `evictExpired` means that fundamental cache operations are $O(N)$ instead of the required $O(1)$, making the cache perform poorly as capacity grows.

### 7. `snapshot()` Performance
The `snapshot()` method iterates over the entire underlying `HashMap` and performs a separate expiration check for every entry. If the cache is large, this is an $O(N)$ operation, which is acceptable for a snapshot, but the constant locking requirement makes it a resource-intensive blocking call.

## 📐 Design and Implementation Problems

### 8. `CacheEntry` Immutability and Mutability
**The Issue:** `CacheEntry` is a static nested class, which is good. However, the `value` field is mutable, and the `getValue()` method modifies the `lastAccessedAt` state. While necessary for tracking access, this mix of fields (some `final`, some mutable) and the reliance on internal state mutation makes reasoning about concurrency difficult, even with the external lock.

**The Improvement:** If possible, encapsulate the state update logic entirely within the `put` or `get` methods of the main cache class, minimizing state mutation within the entry object itself.

### 9. `CacheMetrics` Thread Safety and Overhead
**The Issue:** `CacheMetrics` uses `AtomicLong` for counters, which is good. However, it uses `ArrayList<Long> responseTimes`. Appending to an `ArrayList` is thread-safe *if* synchronization is applied externally, but because the metric record itself is called from various paths without coordination, the list access itself might be dangerous if not guaranteed to be protected (though typical Java `ArrayList` internal structure might handle simple append under CAS, it's not designed for high-throughput concurrent modifications). Furthermore, iterating over `responseTimes` in `getAverageResponseTime()` is an $O(N)$ operation that can be costly if many responses are recorded.

**The Improvement:**
1. Use a thread-safe queue or `LongAdder` if storing response times.
2. For average response time, consider maintaining a running sum and a count using `AtomicLong`s, making the calculation $O(1)$.

### 10. Lack of Graceful Shutdown Handling
**The Issue:** In `close()`, you set `closed = true` and call `cleanupExecutor.shutdown()`. However, a caller relying on the cache might try to submit new tasks or rely on the cache logic *after* `shutdown()` but *before* the threads fully terminate.

**The Improvement:** Implement a robust shutdown pattern, potentially waiting for the cleanup executor to terminate gracefully using `awaitTermination()` in the `close()` method to ensure all scheduled tasks have finished.
