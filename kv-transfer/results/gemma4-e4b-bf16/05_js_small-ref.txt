### 🐞 Bug 1: Rescheduling Breaks the Execution Order

The `reschedule` method updates the `time` property of an event but fails to re-sort the `this.events` array.

**Why it is a bug:**
The `schedule` method ensures that `this.events` is always sorted by time. When you change an event's time using `reschedule`, the array is immediately unsorted. When `run()` executes, it iterates over the array in its current, unsorted order, potentially running events out of time sequence.

**Example Scenario:**
1. Schedule Event A (Time 1000ms).
2. Schedule Event B (Time 2000ms). (Queue: [A, B])
3. Call `reschedule('A', 3000)`. (Queue: [A, B], but A is now 3000ms)
4. `run()` starts. It processes A (3000ms) first, even though B was originally scheduled earlier.

**How to fix it:**
The `reschedule` method must call the same sorting logic used in `schedule` after modifying the time.

```javascript
// Corrected reschedule method
reschedule(name, newTime) {
    for (const event of this.events) {
        if (event.name == name) {
            event.time = newTime;
        }
    }
    // !!! ADD THIS LINE !!!
    this.events.sort((a, b) => a.time - b.time); 
}
```

### 🐞 Bug 2: Flawed Time Calculation in `run()` (Accumulated Latency)

The timing logic inside the `run` method incorrectly calculates the wait time by comparing against a single starting timestamp (`startTime`). It fails to account for the actual time spent executing the previous events or waiting for the previous event.

**Why it is a bug:**
The scheduler is running tasks serially. If Task 1 is scheduled for 1000ms and takes 50ms to execute, the real time elapsed is 1050ms. If Task 2 is scheduled for 2000ms, the wait for Task 2 should only be $2000
