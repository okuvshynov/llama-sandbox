### 1. Critical Bugs & Logic Errors

#### A. Inventory Cleanup Logic (The "Wipeout" Bug)
In `inventory.py`, the `cleanup_expired_reservations` method is completely broken.
*   **Issue:** It iterates through all reservations and adds every single one to the `expired` list, regardless of the timestamp or the `reservation_timeout`.
*   **Impact:** Calling this method will delete all pending order reservations in the system, allowing the same stock to be sold multiple times (over-selling).
*   **Fix:** Store a timestamp when the reservation is created and compare it against `datetime.now() - self.reservation_timeout`.

#### B. Cohort Analysis Logic
In `analytics.py`, the `cohort_analysis` method has a hardcoded index.
*   **Issue:** `cohort_users[period_idx].add(uid)` uses `period_idx = 0` always.
*   **Impact:** Every user who ever joined is placed in "Cohort 0." You cannot actually analyze retention across different time cohorts (e.g., users who joined in Jan vs. Feb).
*   **Fix:** Calculate `period_idx` based on the `first_event_time` relative to a fixed start date or the start of a week/month.

#### C. Floating Point Precision for Money
Throughout `models.py` and `order_processor.py`, `float` is used for prices and totals.
*   **Issue:** Binary floating-point math (IEEE 754) leads to rounding errors (e.g., `0.1 + 0.2 != 0.3`).
*   **Impact:** In a financial system, this leads to "penny discrepancies" in totals and taxes, causing accounting failures.
*   **Fix:** Use the `decimal.Decimal` class for all currency-related fields.

### 2. Performance Issues

#### A. O(N) File I/O (The JSON Bottleneck)
In `inventory.py`, `_save_to_disk` rewrites the **entire** product database to a JSON file every time a single product is added or a reservation is committed.
*   **Issue:** As the product catalog grows, the time to save increases linearly. This will block the `threading.Lock` for longer periods, freezing the entire inventory system.
*   **Fix:** Use a proper database (SQLite, PostgreSQL) to perform atomic updates on specific rows.

#### B. SMTP Connection Overhead
In `order_processor.py`, the `EmailService.send_email` method opens a new SMTP connection, performs a TLS handshake, and logs in for **every single email**.
*   **Issue:** Network handshakes are expensive. Sending 100 confirmation emails requires 100 separate TCP/TLS connections.
*   **Fix:** Use a persistent connection pool or, better yet, an asynchronous task queue (like Celery) with a dedicated email provider API (SendGrid, AWS SES).

#### C. Inefficient Order Search
In `order_processor.py`, `cancel_order` performs a list comprehension search: `if order_id in [o.id for o in self.queue.queue]`.
*   **Issue:** This is an $O(N)$ operation. If the queue is large, canceling an order becomes slow.
*   **Fix:** Maintain a separate `set` of `queued_order_ids` for $O(1)$ lookups.

#### D. Analytics Complexity
In `analytics.py`, `analyze_funnel` sorts all events for every user within a loop.
*   **Issue:** If you have 10,000 users and 100,000 events, the computational cost of sorting and iterating becomes prohibitive.
*   **Fix:** Store events in a time-sorted structure or use a database
