### 1. Nil Map Panic
**The Bug:** In `NewCounter`, the `Counter` struct is returned, but the `counts` map is never initialized.
```go
func NewCounter() *Counter {
    return &Counter{} // counts map is nil
}
```
In Go, reading from a `nil` map is safe (it returns the zero value), but **writing to a `nil` map causes an immediate panic**. When `Increment` calls `c.counts[key]++`, the server will crash.

**The Fix:** Initialize the map using `make`.
```go
func NewCounter() *Counter {
    return &Counter{
        counts: make(map[string]int),
    }
}
```

### 2. Concurrent Map Read/Write Panic (Data Race)
**The Bug:** The `Get` method accesses the map without acquiring the mutex lock.
```go
func (c *Counter) Get(key string) int {
    return c.counts[key] // No lock!
}
```
Go maps are not thread-safe. If one goroutine is writing to the map (via `Increment`) while another is reading from it (via `Get`), the Go runtime will detect a concurrent map read/write and trigger a **fatal panic** that cannot be recovered.

**The Fix:** Use the mutex in `Get`.
```go
func (c *Counter) Get(key string) int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.counts[key]
}
```

### 3. Map Leakage (Race Condition in `GetAll`)
**The Bug:** The `GetAll` method returns the internal map reference while holding the lock, but the lock is released *before* the map is actually used.
```go
func (c *Counter) GetAll() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.counts // Returns a reference to the internal map
}

func handleGetAll(w http.ResponseWriter, r *http.Request) {
    data := counter.GetAll() 
    json.NewEncoder(w).Encode(data) // Reading map HERE without a lock
}
```
Because maps are reference types, `data` points to the same memory as `c.counts`. The mutex is unlocked as soon as `GetAll` returns. When `json.NewEncoder().Encode()` iterates over the map to turn it into JSON, it is reading the map without protection. If another request calls `Increment` at the same time, the server will **panic**.

**The Fix:** Create a copy of the map while holding the lock and return the copy.
```go
func (c *Counter) GetAll() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    snapshot := make(map[string]int, len(c.counts))
    for k, v := range c.counts {
        snapshot[k] = v
    }
    return snapshot
}
```

### 4. Missing Return after `http.Error`
**The Bug:** In `handleIncrement`, when a key is missing, `http.Error` is called, but the function does not stop executing.
```go
if key == "" {
    http.Error(w, "missing key", 400)
    // Missing 'return' here!
}
val := counter.Increment(key) // This still runs!
```
`http.Error` sends a response header and body, but it does not terminate the handler. The code will proceed to increment a counter with an empty string key and attempt to write to the `ResponseWriter` again. This results in a "superfluous response.WriteHeader" warning in the logs and sends an invalid response to the client.

**The Fix:** Add a `return` statement.
```go
if key == "" {
    http.Error(w, "missing key", http.StatusBadRequest)
    return 
}
```
