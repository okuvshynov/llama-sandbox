### 1. Memory Safety & Undefined Behavior (Critical)

#### A. The "Lock-Free" Deque is Unsafe
The `TaskDeque::steal` method contains severe Undefined Behavior.
- **The Issue:** It uses `std::ptr::read` on an `Option<Task>` while another thread (the owner) may be calling `pop` or `push` on that same memory location. Since `Option<Task>` is not an atomic type, this is a **data race**.
- **The Issue:** If the `compare_exchange` fails, you call `std::mem::forget(task)`. However, `ptr::read` has already duplicated the `Task` (and its `Arc` internals). This doesn't undo the data race and can lead to double-frees or corrupted memory.
- **Fix:** Use a proven lock-free deque implementation (like `crossbeam-deque`) or wrap the buffer elements in `AtomicPtr` or `Mutex`.

#### B. Memory Leaks via Circular References
In `Task::with_dependencies`, you create a strong reference cycle.
- **The Issue:** `TaskInner` has `dependencies: Vec<Arc<TaskInner>>` and `dependents: Mutex<Vec<Arc<TaskInner>>>`. When Task B depends on Task A, A points to B (dependent) and B points to A (dependency). Neither will ever be dropped.
- **Fix:** Change `dependents` to store `Weak<TaskInner>`.

#### C. Waker Implementation Risks
The `futures_waker` uses `Arc::into_raw` and `Arc::from_raw`.
- **The Issue:** The `wake` function consumes the `Arc` via `from_raw`, but the `Waker` might be cloned or used multiple times. If `wake` is called, the underlying `TaskInner` is dropped even if other wakers exist.
- **Fix:** Use `std::task::Waker`'s standard construction or a simpler `Arc`-based waker.

### 2. Correctness & Logic Bugs (High)

#### A. Async Tasks are Dropped on Yield
This is the most significant logic bug in the executor.
- **The Issue:** In `execute_task`, you call `pinned.poll(&mut cx)`. If the result is `Poll::Pending`, the function simply returns. Because `fut` is a local variable, **the future is dropped**. An async task that awaits I/O will be deleted from the system and never resume.
- **Fix:** If `Poll::Pending` is returned, the future must be put back into `task.inner.future` so it can be polled again when the waker is triggered.

#### B. Priority Inversion (Wrong Order)
- **The Issue:** `PriorityTaskQueue::pop_highest` iterates `for i in 0..4`. Since `Priority::Low` is 0 and `Critical` is 3, the scheduler processes **Low priority tasks first**.
- **Fix:** Change the loop to `for i in (0..4).rev()`.

#### C. Broken Result Channel
- **The Issue:** In `Scheduler::submit`, you create a `oneshot::channel()`. You give the `Receiver` to the `TaskHandle`, but the `Sender` (`result_tx`) is dropped immediately at the end of the function.
- **Fix:** Store the `oneshot::Sender` inside `TaskInner` so the worker can send the result when the task completes.

#### D. Blocking the Async Runtime
- **The Issue:** `worker_loop` calls `ctx.overflow.pop_blocking(...)`. This uses `std::sync::Condvar`, which blocks the entire OS thread. Since this is running inside `tokio::spawn`, you are blocking a Tokio worker thread, which can lead to runtime starvation or deadlocks.
- **Fix:** Use `tokio::sync::mpsc` or `tokio::sync::Notify` for the overflow queue to allow the thread to yield.

---

### 3. Performance & Design Issues (Medium)

#### A. Busy-Spinning on Dependencies
- **The Issue:** If a task is not ready (`!task.is_ready()`), the worker immediately pushes it back to the local queue. If many tasks are waiting on one slow task, the workers will spin at 100% CPU constantly re-queueing tasks.
- **Fix:** Implement a "waiting" state. Only move a task to the ready queue when its last dependency calls `wake()`.
