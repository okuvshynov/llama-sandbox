### 1. Security Vulnerabilities (Critical)

#### 1.1 SQL Injection
**Issue:** There are multiple instances where user input is concatenated directly into SQL strings instead of using parameterized queries.
*   **Locations:** `UserRepository.findByEmail`, `DocumentRepository.search`, `SearchService.getSuggestions`, `AuditLogRepository.findByUser`.
*   **Impact:** An attacker can bypass authentication, delete the entire database, or leak all user data.
*   **Fix:** Always use parameterized queries (`$1`, `$2`).
    *   *Wrong:* `"SELECT * FROM users WHERE email = '" + email + "'"`
    *   *Right:* `pool.query("SELECT * FROM users WHERE email = $1", [email])`

#### 1.2 Broken Authentication & Weak Hashing
**Issue A:** In `AuthService.login`, the code hashes the password using `sha256(password)`, but in `register`, it used `sha256(password + salt)`. The login will **always fail** because the salt is ignored.
**Issue B:** SHA-256 is a fast cryptographic hash, making it vulnerable to GPU-accelerated brute-force attacks.
*   **Impact:** Users cannot log in; if they could, passwords would be easily cracked.
*   **Fix:** Use `bcrypt` or `argon2`. These libraries handle salting automatically and are computationally expensive (slow), which thwarts brute-force attacks.

#### 1.3 Information Exposure (Stack Traces)
**Issue:** The `/api/auth/register` route returns `err.stack` to the client.
*   **Impact:** Reveals internal file structures, library versions, and logic flow to attackers.
*   **Fix:** Only return the stack trace if `process.env.NODE_ENV === 'development'`.

#### 1.4 Path Traversal / Unsafe File Access
**Issue:** `FileService.downloadFile` uses `attachment.storagePath` directly from the database to create a read stream.
*   **Impact:** If an attacker manages to modify a database record (via the SQL injection mentioned above), they can set the path to `/etc/passwd` and download sensitive system files.
*   **Fix:** Validate that the resolved path remains within the `config.uploadDir`.

### 2. Performance Issues

#### 2.1 The N+1 Query Problem
**Issue:** `DocumentRepository.getDocumentsWithAuthors` fetches a list of documents, then loops through them to perform a separate database query for every single author.
*   **Impact:** If a folder has 100 documents, the server makes 101 database calls. This will crash the DB under moderate load.
*   **Fix:** Use a `JOIN` in the SQL query.
    *   `SELECT d.*, u.username as author_name FROM documents d JOIN users u ON d.author_id = u.id WHERE d.folder_id = $1`

#### 2.2 Recursive Database Calls
**Issue:** `FolderRepository.getFolderTree` is a recursive function that calls the database at every level of the folder hierarchy.
*   **Impact:** Extremely slow for deep folder structures.
*   **Fix:** Use a **Recursive Common Table Expression (CTE)** in PostgreSQL to fetch the entire tree in one single query.

#### 2.3 Memory Leaks (SSE & Cache)
**Issue A (SSE):** `NotificationService` stores `Response` objects in a `Map`. It never removes them when a user disconnects.
**Issue B (Cache):** `SearchService.searchCache` is a simple `Map` that grows indefinitely.
*   **Impact:** The server will eventually run out of RAM (Out of Memory crash).
*   **Fix A:** Listen for the connection close event: `res.on('close', () => this.listeners.delete(userId))`.
*   **Fix B:** Use a cache with a Time-To-Live (TTL) like `node-cache` or an external store like Redis.

#### 2.4 Inefficient Bulk Updates
**Issue:** `DocumentService.bulkUpdateTags` loops through IDs and performs individual `findById` and `update` calls.
*   **Impact:** High latency and DB overhead.
*   **Fix:** Use a single SQL statement: `UPDATE documents SET tags = $1 WHERE id = ANY($2) AND author_id = $3`.
