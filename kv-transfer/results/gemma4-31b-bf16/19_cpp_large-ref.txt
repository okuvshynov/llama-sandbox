C++ codebase implementing a Lock-Free Hash Map, a Slab Allocator, a Hazard Pointer system, a Lock-Free Stack, and a Ring Buffer.

    *   *SlabAllocator:*
        *   Uses `std::aligned_alloc` (C11/C++17).
        *   Locking is via `std::mutex` (not actually lock-free, but it's a pool).

        *   `SlabBlock` size is `64` bytes (cache line aligned).
        *   Wait, `SlabBlock` has a `next` pointer and padding. If `object_size_` is small, the padding is there. But `object_size_` is used for the offset in `grow()`.
        *   *Potential issue:* If `object_size_` is not a multiple of `alignof(SlabBlock)`, `reinterpret_cast` might lead to misaligned access. `std::aligned_alloc` ensures the start is aligned, but subsequent offsets might not be.
        *   *Potential issue:* `SlabBlock` is meant to be the "free" state of the memory. When allocated, the user gets the pointer. The user might overwrite the `next` pointer.     *   *HazardPointerDomain:*
        *   `acquire_for_current_thread()`: Tries to reuse, then allocates `new HazardPointerRecord()`.
        *   The list of records is managed via a lock-free stack (`head_`).
        *   `retire()`: Uses `thread_local` vector.
        *   `scan()`: Collects *all* hazard pointers from *all* records into a vector, sorts it, and then checks retired items.
        *   *Major Issue (Race Condition):* In `scan()`, it loads the `head_` and iterates. Since `head_` is a lock-free list and records can be added, the list is technically stable for traversal, but there is no mechanism to remove records when threads exit other than `release_record`.

        *   `protect()`: Classic HP pattern. Load $\to$ Store $\to$ Load-and-compare.

        *   *Design:* Linear probing.

        *   *Issue (Concurrency):* `insert` does `table_[probe].key.compare_exchange_strong(expected, k, ...)`. If it succeeds, it then does `pool_.allocate()` and `value.store()`.

### 1. Critical Bugs (Correctness & Memory Safety)

#### A. The "Resize" Use-After-Free (UAF)
The `LockFreeHashMap::resize()` method is catastrophic. 
- **Issue:** It allocates a new table, moves elements, and then calls `delete[] table_`. However, other threads may currently be inside `find()`, `insert()`, or `erase()`, holding a pointer to the old `table_`.
- **Consequence:** Immediate segmentation fault or silent memory corruption when a thread accesses the deleted array.
- **Fix:** You cannot simply `delete[]` a table in a lock-free structure. You must use the **Hazard Pointer** system (which you already implemented!) to protect the `table_` pointer itself, or use a "Quiescent State" mechanism to ensure no threads are accessing the old table before freeing it.

#### B. Lock-Free Stack ABA Problem
The `LockFreeStack::pop()` implementation is a textbook example of the **ABA problem**.
