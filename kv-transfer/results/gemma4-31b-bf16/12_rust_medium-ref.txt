### 1. Soundness & Correctness Issues

#### **Panic on `NaN` values**
In the `percentile` method:
```rust
sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
```
`f64` does not implement `Ord` because `NaN` values cannot be compared. If a `NaN` value is ever recorded into a metric, `partial_cmp` returns `None`, and the `.unwrap()` will cause the entire thread (or the whole application) to **panic**.
*   **Fix:** Use `sorted.sort_by(|a, b| a.partial_cmp(b).unwrap_or(std::cmp::Ordering::Equal))` or filter out `NaN`s.

#### **Potential Out-of-Bounds/Panic on Empty Data**
In `percentile`, you check `if self.values.is_empty()`. However, if `p` is passed as a negative number or an extremely large number, the logic `idx.min(sorted.len() - 1)` might behave unexpectedly. While the `is_empty` check prevents the most obvious panic, the mathematical logic for the index calculation is fragile.

#### **Dead Code**
The `Histogram` and `HistogramBucket` structs are fully implemented but **never used** anywhere in the `MetricsRegistry` or the simulation. This increases the maintenance burden without providing value.

### 2. Performance Problems

#### **Algorithmic Complexity of Percentiles ($O(N \log N)$)**
The `percentile` method is called three times during every `snapshot()`. Each call:
1. Clones the entire `values` vector ($O(N)$).
2. Sorts the entire vector ($O(N \log N)$).
For a metric with 1 million samples, a single snapshot will perform three massive allocations and three expensive sorts.
*   **Fix:** Use a T-Digest or an HdrHistogram for constant-time/approximate percentile tracking.

#### **Unbounded Memory Growth (Memory Leak)**
The `Metric` struct stores every single value and timestamp in a `Vec`. There is a `trim_older_than` method, but **it is never called**. In a real system, this will consume all available RAM until the process is killed by the OOM (Out of Memory) killer.
*   **Fix:** Implement a sliding window using a `VecDeque` or call `trim_older_than` automatically during `record`.

#### **Inefficient "Trim" Operation ($O(N)$)**
The `trim_older_than` method uses `drain(..pos)`. In a `Vec`, removing elements from the front requires shifting all subsequent elements to the left.
*   **Fix:** Use a `std::collections::VecDeque`, which allows $O(1)$ removal from the front.

#### **Linear Scan for Rate Calculation ($O(N)$)**
The `rate` method iterates through the entire `timestamps` vector to count events within a window. As the history grows, calculating the rate becomes slower and slower.
*   **Fix:** Maintain a rolling counter or use a bucketed histogram.

### 3. API Design Concerns

#### **Leaking Synchronization Primitives**
`get_or_create` returns `Arc<Mutex<Metric>>`. 
By returning the `Mutex` directly to the caller, the registry loses control over its own concurrency. A user could call `get_or_create`, lock the mutex, and then perform a long-blocking operation. This would block the `snapshot()` function, potentially freezing the entire monitoring system.
*   **Fix:** Keep the `Mutex` private. Provide methods on `MetricsRegistry` (like `record`) that handle the locking internally and briefly.

#### **Double Locking / Lock Contention**
The `MetricsRegistry` uses a `RwLock<HashMap<..., Arc<Mutex<Metric>>>>`. 
To record a value, the code:
1. Takes a Read lock on the Map.
2. Takes a Mutex lock on the Metric.
In `snapshot()`, the code holds the Read lock on the Map for the **entire duration** of the snapshot while sequentially locking every single metric. If there are thousands of metrics, this blocks any new metrics from being created (which requires a Write lock) for a long time.
