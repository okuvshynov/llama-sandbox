### 1. MemoryArena (Memory Management)

#### Issues:
*   **Use-After-Free (UAF):** In `main()`, `arena.reset()` is called, which deletes the memory blocks. Immediately after, the code attempts to access `records[0]->tag`. This is a crash/undefined behavior.
*   **Resource Leaks:** `MemoryArena::create<T>` uses placement new, but `reset()` and the destructor only delete the raw memory blocks. If `T` is a non-POD type (e.g., `std::string`), its destructor is never called, leading to leaks of internal resources.
*   **Redundant Alignment Logic:** The `allocate` method repeats the `std::align` and offset calculation logic twice (once before `grow()` and once after).

#### Suggested Fixes:
*   **Lifecycle Management:** The user must ensure no pointers to arena memory exist after `reset()`.
*   **Destructor Tracking:** If the arena must support non-POD types, it needs to store a list of "cleanup" functions (function pointers to destructors) that are called during `reset()` or destruction.
*   **Refactor `allocate`:** Move the alignment logic into a helper function to avoid duplication.
