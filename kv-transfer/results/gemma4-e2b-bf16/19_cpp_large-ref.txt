## 1. Critical Issues (Memory Safety & Concurrency Bugs)

### Issue 1.1: Hazard Pointer Allocation Strategy (Memory Safety)
**Location:** `HazardPointerDomain::acquire_for_current_thread()`
```cpp
// Allocate new record
auto* rec = new HazardPointerRecord(); // <-- DANGER
```
**Problem:** The Hazard Pointer Domain allocates `HazardPointerRecord` objects using raw `new`. Since `HazardPointerDomain` is a singleton accessed concurrently by multiple threads, this `new` operation is performed in a highly contested, lock-free path (though the list insertion is lock-free). If the system relies on specialized memory pools (like the `SlabAllocator`) for general allocation, mixing raw `new` here can lead to fragmented memory, potential heap corruption, or failure if the general heap allocator is under stress. Furthermore, if the domain destructor runs while threads are active, it might try to delete objects that are still referenced by a guard.

**Fix:**
1. **Use a dedicated pool:** The `SlabAllocator` should be used to allocate `HazardPointerRecord` structures instead of `new`.
2. **Review Lifetime Management:** If raw pointers are used, ensure the management (list head manipulation in the destructor) is strictly correct. Since the destructor iterates through `head_.load()`, it correctly traverses the list, but this requires the list itself to be stable.

### Issue 1.2: Inefficient Hazard Pointer Scanning (Performance/Correctness)
**Location:** `HazardPointerDomain::scan()`
```cpp
// Collect all hazard pointers
// ... (collection logic) ...
std::sort(hazard_ptrs.begin(), hazard_ptrs.end()); // <-- MAJOR BOTTLENECK
// Partition retired list
auto it = retired.begin();
while (it != retired.end()) {
    if (!std::binary_search(hazard_ptrs.begin(), hazard_ptrs.end(), it->first)) { // <-- Inefficient search
        // ... removal logic
    } else {
        ++it;
    }
}
```
**Problem:** The `scan` function performs a full sort ($O(N \log N)$) of all currently active hazard pointers in every scan cycle, and then uses binary search inside the removal loop. This is extremely inefficient, especially if `SCAN_THRESHOLD` is large and many pointers are active.

**Fix:**
Replace the sorting and binary search with a linear partition approach (similar to quickselect, or a two-pointer approach if the input list is sorted). Since the goal is to check if `it->first` exists in `hazard_ptrs`, and `hazard_ptrs` is sorted, you can partition `retired` into two lists in $O(N)$ time, based on the sorted list.
