## 1. Performance Issues (Critical)

### Issue 1.1: Inefficient LRU Implementation
The `LRUEvictionPolicy` uses a standard Python list (`self.access_order`) to track access.

*   **Problem:** Operations like `record_access` (which involves searching and removing an element: `if key in self.access_order: self.access_order.remove(key)`) and retrieval of the candidate (`self.access_order[0]`) are all **O(N)**, where N is the number of cache entries. In a high-throughput cache, this linear time complexity for core operations is unacceptable.
*   **Fix:** Replace the list with a structure that provides O(1) access, insertion, and deletion, such as a **Doubly Linked List (DLL)** combined with a dictionary mapping keys to list nodes. This allows O(1) updates when an item is accessed or moved to the front/back.
    *   *Alternatively, in Python, using `collections.OrderedDict` or `functools.lru_cache` (if applicable) can simplify this, but a custom DLL implementation provides the necessary control for eviction policies.*

### Issue 1.2: Excessive Locking and Global Contention
The `CacheStore` uses a single `threading.Lock()` to guard *all* access (reads, writes, size checks, eviction, and database logging).

*   **Problem:** This design serializes all cache operations. If one thread is performing a long operation (e.g., a large `put` involving many evictions or logging), all other threads are blocked. This severely limits the scalability and concurrency of the cache, defeating the purpose of using `threading`.
*   **Fix:** Implement finer-grained locking.
    *   Use separate locks for the main entry dictionary (`self.entries`) manipulation and statistics updates.
    *   For instance, `get` operations could potentially read data without a lock if the data structure is designed to be thread-safe (e.g., using concurrent dictionaries or immutable structures for values), but since the LRU/size tracking is complex, a read/write lock pattern might be necessary, or simply using `threading.RLock` if nested locks were required, though typically separate locks are better for performance.

### Issue 1.3: Synchronous Database Commit
The method `_log_operation` commits the SQLite transaction inside the main cache lock.

*   **Problem:** Database I/O operations (especially commits) are slow. Holding the cache lock while waiting for disk I/O will cause severe blocking and reduce overall throughput drastically.
*   **Fix:** **Decouple logging from cache operations.**
    1.  Change `_log_operation` to simply queue the log entry (e.g., append to an in-memory queue or a dedicated logging buffer).
    2.  Introduce a separate, dedicated worker thread or process responsible for periodically flushing this log queue to the SQLite database in bulk. This ensures that the main cache thread is never blocked by database latency.

## 2. Design and Maintainability Issues

### Issue 2.1: Tight Coupling of Eviction Policies
The `CacheStore` tightly couples its eviction logic by directly managing instances of `LRUEvictionPolicy` and `TTLEvictionPolicy`.

*   **Problem:** This makes it difficult to swap out policies (e.g., using LFU or FIFO) without modifying the core `CacheStore` logic.
*   **Fix:** Abstract the eviction and expiration logic.
    1.  Define an `EvictionPolicy` interface (or ABC) with methods like `record_access`, `get_candidate`, and `remove`.
    2.  Define a `TimeBasedPolicy` interface with methods like `record_access` and `get_expired_keys`.
    3.  The `CacheStore` should accept these policy objects in its constructor (Dependency Injection), making it highly flexible.
