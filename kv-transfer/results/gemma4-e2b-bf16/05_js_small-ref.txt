## 🐞 Bug 1: Synchronous Blocking within the Event Loop (`run` method)

**The Bug:** The `run()` method uses `await new Promise(resolve => setTimeout(resolve, wait));` inside a `for...of` loop. This makes the entire scheduler **synchronously blocking**. The scheduler will pause execution at every scheduled time, waiting for the required delay before proceeding to the next event.

**Why it's a problem:**
1. **Single-Threaded Bottleneck:** Since JavaScript is single-threaded, if one event takes a long time (or if the calculated `wait` time is large), the entire scheduler freezes. It cannot handle other tasks or concurrent operations.
2. **Inefficient Scheduling:** True non-blocking schedulers typically use `setTimeout` recursively or set up a single event loop mechanism (like a priority queue) to manage time, rather than pausing the main execution thread for every single wait period.

**Suggested Fix:**
For a more robust, non-blocking scheduler, you should refactor `run()` to use `setTimeout` to schedule the *next* event, rather than using `await` to pause the current execution thread.

## 🐞 Bug 2: Race Condition / Non-Atomic Listener Execution

**The Bug:** Inside the `run()` method, the event's callback runs first, and *then* all associated listeners are executed:

```javascript
event.callback();

const handlers = this.listeners[event.name];
if (handlers) {
  for (const handler of handlers) {
    handler(event); // Listener execution happens here
  }
}
```

**Why it's a problem:**
If any of the registered handlers (listeners) are asynchronous (e.g., they perform network requests or long computations and return a Promise), they will execute *while* the scheduler is attempting to move on to the next event. This breaks the strict sequential flow of the scheduler, potentially causing the order of operations to be violated or leading to unexpected state changes if listeners rely on the scheduler being fully idle.

**Suggested Fix:**
If you need the scheduler to wait for the consequences of the event, the `event.callback()` should ideally return a Promise. The `run` loop should then `await` this callback, ensuring that the event (and all its listeners) are fully resolved before moving to the next event.
