## 1. Critical Bugs and Safety Issues

### Issue 1.1: Task Memory Leak (Circular `Arc` References)
**Location:** `task.rs`, `Task::with_dependencies`

When creating dependent tasks, you register the dependency as a dependent of the current task, and the current task as a dependency of the dependent.

```rust
// task.rs
// Register as dependent on each dependency — creates circular Arc
// references: dep.dependents -> inner, inner.dependencies -> dep.
// Neither side uses Weak, so these will never be freed.
for dep in &dep_inners {
    dep.dependents.lock().unwrap().push(inner.clone());
}
```

**Problem:** Because `TaskInner` uses `Arc` for dependencies and dependents, these cycles prevent the tasks (and their internal state) from ever being dropped, leading to a memory leak.

**Fix:** Use `std::sync::Weak` pointers for back-references (dependents). This breaks the strong reference cycle, allowing tasks to be safely deallocated when no strong references remain.

### Issue 1.2: Data Race and UB in Lock-Free Queue (`TaskDeque::steal`)
**Location:** `queue.rs`, `TaskDeque::steal`

The attempt at a lock-free steal operation is fundamentally broken due to a lack of atomicity protection over the underlying buffer data.

```rust
// queue.rs, TaskDeque::steal
// Data race: buffer is not atomic, owner can modify buffer[idx]
// while we read it. This needs unsafe + atomic reads or a
// different design.
let task_ptr = &self.buffer[idx] as *const Option<Task>;
let task = unsafe { std::ptr::read(task_ptr) };

// ... CAS logic relies on duplicated ownership
if self.head.compare_exchange(...) {
    task
} else {
    // Another stealer won — we already did ptr::read, so we've
    // duplicated ownership. This is UB.
    std::mem::forget(task);
    None
}
```

**Problem:** The `Task` stored in the `Vec<Option<Task>>` is accessed via raw pointers inside a critical section, without ensuring that the memory location hasn't been mutated by a concurrent `push` or `pop` operation (even if that operation is internally guarded by a mutex, the state change is not visible across the raw pointer read). Furthermore, relying on `std::ptr::read` and then managing ownership via CAS is extremely dangerous and leads to Undefined Behavior (UB).

**Fix:** If you want a lock-free queue, use structures designed for this (like `crossbeam::queue` or a carefully implemented hazard pointer scheme). If you stick with `Mutex<VecDeque>`, simply rely entirely on the `Mutex` lock for safety, accepting the performance hit.
