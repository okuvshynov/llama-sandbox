## 1. Concurrency and Thread Safety Issues (Critical)

The primary concern is the inconsistent handling of shared data structures, particularly when combined with the single global lock.

### Issue 1.1: Non-Thread-Safe Metrics Accumulation

The `CacheMetrics.responseTimes` list is an `ArrayList` and is modified by multiple threads (via `get()`) without any synchronization.

**Explanation:** If multiple threads call `get()` concurrently, they might attempt to add response times to `responseTimes` simultaneously, leading to a `ConcurrentModificationException` or data corruption in the underlying `ArrayList`.

**Fix:** Synchronize access to the `responseTimes` list, or use a concurrent collection if performance is paramount.

### Issue 1.2: ConcurrentModificationException in `snapshot()` (Critical)

The `snapshot()` method iterates over the `store` map without acquiring the global lock.

**Explanation:** The `store` map is actively modified by several methods (`put`, `remove`, and the scheduled `evictExpired`) which all acquire the lock. If `snapshot()` runs concurrently with a modification, the iterator will fail, throwing a `ConcurrentModificationException`.

**Fix:** The snapshot operation must acquire the lock to ensure a consistent view of the cache state.

```java
public Map<K, V> snapshot() {
    lock.lock(); // Acquire lock
    try {
        Map<K, V> result = new HashMap<>();
        for (Map.Entry<K, CacheEntry<K, V>> e : store.entrySet()) {
            if (!e.getValue().isExpired()) {
                result.put(e.getKey(), e.getValue().getValue());
            }
        }
        return result;
    } finally {
        lock.unlock(); // Release lock
    }
}
```

## 2. Performance and Efficiency Issues

The cache implementation uses global locking and linear data structures, which leads to poor performance under high concurrency or large capacity.

### Issue 2.1: Coarse-Grained Locking (Bottleneck)

The entire cache operation (read, write, eviction, cleanup) is protected by a single `ReentrantLock`.

**Explanation:** This design severely limits concurrency. If one thread is performing a long operation (like a `put` involving eviction or an `evictExpired` sweep), all other threads, even simple `get` operations, must wait.

**Fix:** Implement finer-grained locking or use more specialized concurrent data structures.
1. **Read/Write Lock:** Replace `ReentrantLock` with a `ReentrantReadWriteLock`. Reads (`get`, `containsKey`) could proceed concurrently, only blocking when a write operation (`put`, `remove`, `evictExpired`) is taking place.
2. **Concurrent Map:** Use `ConcurrentHashMap` for the `store` if possible, reducing the need for the global lock for simple lookups and insertions. However, this still complicates the maintenance of the `accessOrder` linked list.

### Issue 2.2: Inefficient Access Order Management (O(N) Operations)

The cache uses a `LinkedList<K>` (`accessOrder`) to maintain the access order (acting like an LRU queue). Operations on this list are inefficient.

**Explanation:**
1. **`touchEntry(K key)`:** This method calls `accessOrder.remove(key)` followed by `accessOrder.addFirst(key)`. Removing an arbitrary element from a `LinkedList` requires traversing the list, making this operation $O(N)$ where $N$ is the capacity.
2. **`evictExpired()`:** When an entry expires, it calls `accessOrder.remove(entry.getKey())`. This is also $O(N)$.

**Fix:** To achieve $O(1)$ time complexity for updating access order (necessary for an efficient LRU cache), the `accessOrder` structure must be a **Doubly Linked List** (like `java.util.LinkedHashMap`'s internal structure) where we store a direct reference to the node in the list within the `CacheEntry` itself.

If sticking to `LinkedList`, you would need a separate `Map<K, LinkedList.Node<K>>` to allow $O(1)$ node removal, significantly complicating the implementation.
