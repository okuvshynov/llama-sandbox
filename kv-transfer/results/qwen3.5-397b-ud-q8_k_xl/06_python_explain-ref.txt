This Python program solves the **Topological Sorting** problem for a directed graph. Specifically, it models a set of tasks with dependencies (like a build pipeline) and determines valid orders in which these tasks can be executed.

Here is a breakdown of what the program does and the algorithms it employs.

### 1. Program Functionality
The program simulates a software build process where certain steps must happen before others.
- **Input**: A list of edges representing dependencies. For example, `("compile", "link")` means "compile" must happen before "link".
- **Output**:
  1. **One valid order**: It prints a single valid sequence of tasks using the `solve` function.
  2. **All valid orders**: It calculates and prints every possible valid sequence of tasks using the `find_all` function.

If the dependencies contain a cycle (e.g., A depends on B, and B depends on A), `solve` will return `None`, indicating no valid order exists.

### 2. Algorithm Description

The code implements two variations of **Kahn's Algorithm** for topological sorting.

#### Part A: Graph Construction (`build_graph`)
Before sorting, the program constructs an adjacency list representation of the graph and calculates the **in-degree** for every node.
- **Adjacency List (`graph`)**: Maps a node to a list of nodes that depend on it (outgoing edges).
- **In-Degree (`in_degree`)**: Counts how many prerequisites each node has.
- **Logic**: It iterates through the edges, populating the graph and incrementing the in-degree count for the destination node. It ensures every node exists in the `in_degree` map, initializing those with no incoming edges to 0.

#### Part B: Finding One Valid Order (`solve`)
This function uses the standard **Breadth-First Search (BFS)** approach of Kahn's Algorithm.

1.  **Initialization**: Identify all nodes with an `in_degree` of 0 (no dependencies) and add them to a queue.
2.  **Processing**:
    -   Dequeue a node and add it to the `result` list.
    -   For each neighbor (dependent task) of this node:
        -   Decrement the neighbor's `in_degree` by 1 (simulating that one prerequisite is now complete).
        -   If the neighbor's `in_degree` becomes 0, add it to the queue.
3.  **Cycle Detection**: After the queue is empty, if the length of `result` is less than the total number of nodes, a cycle exists in the graph, and the function returns `None`. Otherwise, it returns the sorted list.

**Time Complexity**: $O(V + E)$, where $V$ is the number of vertices (tasks) and $E$ is the number of edges (dependencies).

#### Part C: Finding All Valid Orders (`find_all`)
Finding *all* topological sorts is more complex because at any step, there might be multiple nodes with an `in_degree` of 0. Choosing different nodes leads to different valid paths. This function uses **Backtracking (Depth-First Search)**.

1.  **State**: It maintains the current `path` and a snapshot of the current `remaining_in_degree`.
2.  **Recursive Step (`backtrack`)**:
    -   **Base Case**: If the `path` length equals the total number of nodes, a complete valid order is found; save it.
    -   **Candidate Selection**: Iterate through all nodes. A node is a candidate if:
        1.  It is not already in the current `path`.
        2.  Its current `in_degree` is 0.
        3.  *(Note: The code sorts nodes before iterating to ensure the output lists are generated in lexicographical order).*
    -   **Explore**:
        -   Add the candidate to the `path`.
        -   Create a copy of the in-degree map and decrement the counts for the candidate's neighbors.
        -   Recursively call `backtrack`.
    -   **Backtrack**: Remove the candidate from the `path` (pop) to try the next candidate.

**Time Complexity**: In the worst case, the number of topological sorts can be factorial $O(V!)$, making this approach exponential. It is suitable only for small graphs.
