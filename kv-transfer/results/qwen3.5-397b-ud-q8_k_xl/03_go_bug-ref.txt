# Bugs in the Go HTTP Server

This Go HTTP server contains several critical bugs that can lead to runtime panics, data races, and unsafe behavior. Below is a detailed explanation of each issue and how to fix it.

---

## 1. ❌ Uninitialized Map (Will Panic)

**Location:** `NewCounter()`

```go
func NewCounter() *Counter {
	return &Counter{}  // counts is nil!
}
```

**Problem:** The `counts` map is never initialized. Any attempt to write to it (e.g., in `Increment`) will cause a runtime panic: `assignment to entry in nil map`.

**Fix:**

```go
func NewCounter() *Counter {
	return &Counter{
		counts: make(map[string]int),
	}
}
```

---

## 2. ❌ Race Condition in `Get` Method

**Location:** `Get` method

```go
func (c *Counter) Get(key string) int {
	return c.counts[key]  // No lock!
}
```

**Problem:** This method reads from the shared `counts` map without acquiring the mutex, while other methods like `Increment` write to it with locking. This introduces a data race.

**Fix:**

```go
func (c *Counter) Get(key string) int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts[key]
}
```

## 3. ❌ Missing `return` After Error Handling

```go
if key == "" {
```

**Fix:**

## 4. ❌ Returning Internal Map Reference (Unsafe)

**Location:** `GetAll` method

```go

```

**Problem:** This returns a reference to the internal map. After the lock is released, callers can read or modify the map without synchronization, breaking encapsulation and risking data races.

**Fix:**

```go

func NewCounter() *Counter {

	c.mu.Lock()
	defer c.mu.Unlock()
	c.counts[key]++
	return c.counts[key]
}

func (c *Counter) Get(key string) int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts[key]

func (c *Counter) GetAll() map[string]int {

	defer c.mu.Unlock()
```

## 🔍 Bonus Recommendations

- **Use `http.StatusBadRequest`** instead of raw `400` for clarity.
- **Set `Content-Type` header** when returning JSON:
  ```go
  w.Header().Set("Content-Type", "application/json")
  ```
- **Use `sync.RWMutex`** if read operations greatly outnumber writes, to improve concurrency.
- **Validate HTTP methods** (e.g., allow only POST for `/inc`, GET for `/all`).
- **Add graceful shutdown** and request timeout handling for production robustness.
