1.  **Understand the Goal:** The user wants a code review of a Go distributed task scheduler implementation. I need to identify bugs, race conditions, performance issues, and design problems across four files (`scheduler.go`, `ratelimiter.go`, `circuitbreaker.go`, `workerpool.go`). For each issue, I need to explain it and suggest a fix.

2.  **Initial Scan & Architecture Overview:**
    *   **`scheduler.go`:** Core logic. Manages tasks, a priority queue (heap), workers, dependencies, and state. Uses a global mutex (`sync.RWMutex`) for almost everything.
    *   **`ratelimiter.go`:** Token bucket and sliding window implementations. Used for limiting task execution rates (though not explicitly integrated into the scheduler logic in the provided snippets, it's part of the package).
    *   **`circuitbreaker.go`:** Standard circuit breaker pattern. Registry for managing multiple breakers.
    *   **`workerpool.go`:** Manages a pool of worker goroutines that pull tasks from the scheduler. Includes auto-scaling logic.

    *   **Overall Architecture:** It looks like a *single-process* scheduler with a worker pool, despite the name "distributed task scheduler". There's no RPC, no shared state store (Redis/DB), no leader election. It's essentially an in-memory job queue with worker threads. This is a significant design mismatch if the intent was truly "distributed." I need to point this out.

3.  **Detailed Analysis - File by File:**

    *   **`scheduler.go`:**
        *   *Concurrency:* Heavy reliance on `s.mu`. This is a global lock bottleneck. `AssignTask`, `CompleteTask`, `SubmitTask`, `Heartbeat` all lock the whole scheduler.
        *   *Race Conditions:*
            *   `CompleteTask`: Modifies `worker.Load` inside the lock, but `WorkerStatus` fields like `TasksCompleted` are updated atomically *outside* the lock logic sometimes? Wait, `CompleteTask` holds `s.mu`. Inside, it accesses `s.workers[task.WorkerID]`. Then it does `atomic.AddInt64`. This is safe but mixing mutex and atomics on the same struct fields is confusing.
            *   `checkWorkerHealth`: Requeues tasks. It pushes to the heap. `AssignTask` pops from the heap. Both hold `s.mu`. This is safe *within* the process, but...
            *   `SubmitTask` dependency logic: It checks `s.completed`. If a dependency completes *while* `SubmitTask` is running (before it adds to `tasks` map), there might be a race, but `s.mu` covers it.
            *   **Critical Race:** In `CompleteTask`, there's a `go func()` for retry logic. Inside that goroutine, it locks `s.mu`. However, the task state is modified *before* the goroutine starts (`task.State = TaskRetrying`). If `AssignTask` runs between the state change and the goroutine sleeping, it might pick up a task marked `Retrying` but not yet `Pending` in the queue? Actually, `AssignTask` checks `task.State == TaskCancelled`. It doesn't check `TaskRetrying`. If a task is `TaskRetrying`, it's *not* in the heap yet (it's sleeping). But `tasks` map has it. If `AssignTask` iterates the heap, it won't see it. But wait, `CompleteTask` removes it from the worker.
            *   **Deadlock Risk:** `CompleteTask` calls `heap.Push`. `AssignTask` calls `heap.Pop`. Both hold `s.mu`. This is fine.
            *   **Goroutine Leak:** In `CompleteTask`, the retry goroutine captures `task` and `s`. If `s.Stop()` is called, `s.ctx` is cancelled. The retry goroutine does `time.Sleep`. It doesn't check `s.ctx`. It will wake up, try to lock `s.mu`, and push. If the scheduler is stopped, this is bad.
            *   **Worker Load:** `worker.Load` is modified inside `s.mu` in `AssignTask` and `CompleteTask`. But `WorkerStatus` is a struct pointer. `checkWorkerHealth` also modifies `worker.IsAlive` inside `s.mu`. This seems consistent, but `Stats()` iterates workers.
            *   **Dependency Graph:** `depGraph` maps DependencyID -> []DependentIDs. When a task is cleaned up (`cleanupCompletedTasks`), it deletes from `s.tasks` and `s.completed`. It does *not* clean up `depGraph`. Memory leak.
            *   **Task State Consistency:** `Task` struct is passed around by pointer. Multiple goroutines (workers) hold pointers to tasks. `s.mu` protects the *scheduler's* view, but if a worker holds a `*Task` and reads fields while the scheduler modifies them (e.g., during retry logic or cancellation), that's a race. `AssignTask` returns `*Task`. The worker processes it. Meanwhile, `CancelTask` might modify that same `*Task` struct. `CancelTask` locks `s.mu`. The worker *does not* lock `s.mu` while processing `task.Payload` etc. This is a data race on the `Task` struct fields.
        *   *Design:*
            *   "Distributed" is a misnomer. It's local.
            *   `rand` usage: `rand.Int63n` is not safe for concurrent use without locking or a local source. `math/rand` global functions are protected by a mutex, but it's a bottleneck. Better to use `rand.New(rand.NewSource(...))` per goroutine or `crypto/rand`.
            *   `cleanupCompletedTasks`: Deletes from `s.tasks`. What if a worker is still processing a task that was just "completed" and moved to `s.completed`? `CompleteTask` moves it. `AssignTask` only pulls from `queue`. So once completed, workers shouldn't touch it. But `CancelTask` looks in `s.tasks`. If `cleanup` deletes from `s.tasks`, `CancelTask` might fail to find a recently completed task if called quickly? No, `CancelTask` checks terminal states.
            *   `AssignTask` recursion: `if task.State == TaskCancelled { return s.AssignTask(workerID) }`. This is recursive while holding the lock. If many cancelled tasks are at the top of the heap, this could cause stack overflow or hold the lock too long.
            *   `WorkerStatus` in `Scheduler`: The scheduler tracks workers. But `WorkerPool` also tracks workers. There's duplication of state.

    *   **`ratelimiter.go`:**
        *   *Token Bucket:* `cleanupLoop` runs forever. No way to stop it (no context). Goroutine leak on process exit.
        *   *Sliding Window:* `AllowN` creates a new slice `valid` every time. High allocation pressure under load.
        *   *Sliding Window:* `cleanupLoop` also has no context/cancel. Goroutine leak.
        *   *Concurrency:* `sync.Mutex` protects the whole map. High contention if many keys.
        *   *Logic:* `cleanup` in SlidingWindow iterates all keys. If map is huge, this pauses all `Allow` calls (since it takes the lock).

    *   **`circuitbreaker.go`:**
        *   *Half-Open Concurrency:* `halfOpenCurrent` is modified inside `canExecute` (locked) and `recordResult` (locked). This looks okay, but `Execute` calls `canExecute` then `fn()` then `recordResult`. If `fn()` takes a long time, the `halfOpenCurrent` count stays high, blocking other half-open requests. This is intended behavior (concurrency limit in half-open), but `halfOpenCurrent` is never decremented if `fn()` panics.
        *   *Panic Safety:* `Execute` doesn't recover from panics in `fn()`. If `fn()` panics, `recordResult` is never called. `halfOpenCurrent` stays incremented forever. Circuit breaker gets stuck.
        *   *Registry:* `Get` uses Double-Checked Locking. In Go, this is usually okay with `sync.Mutex` but `sync.RWMutex` requires care. The pattern used (`RLock`, check, `Unlock`, `Lock`, check again`) is generally safe for initialization, but `breakers` map write needs the write lock. The implementation looks mostly correct for lazy init, but `sync.Map` might be cleaner.
        *   *Stats:* `lastFailure` is `time.Time`. Copying `time.Time` is fine.

    *   **`workerpool.go`:**
        *   *Scheduler Coupling:* `Worker` calls `w.pool.scheduler.handlers`. `handlers` is a map inside `Scheduler`. `Scheduler` protects `handlers` with `s.mu`. `Worker.Run` accesses `handlers` *without* locking `s.mu`. **Data Race.**
        *   *Scheduler Coupling:* `Worker` calls `w.pool.scheduler.AssignTask`. This locks `s.mu`.
        *   *Scheduler Coupling:* `Worker` calls `w.pool.scheduler.CompleteTask`. This locks `s.mu`.
        *   *Worker State:* `w.running` (atomic)
