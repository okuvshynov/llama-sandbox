### 1. Critical Security Vulnerabilities

#### A. SQL Injection
**Issue:** Multiple repositories construct SQL queries using string interpolation instead of parameterized queries.
*   **Location:** `UserRepository.findByEmail`, `AuditLogRepository.findByUser`, `SearchService.getSuggestions`.
*   **Risk:** Attackers can bypass authentication, modify data, or delete the database.
*   **Fix:** Use parameterized queries (`$1`, `$2`) for all dynamic inputs.

```typescript
// ❌ Bad
const result = await pool.query(
  "SELECT * FROM users WHERE email = '" + email + "'"
);

// ✅ Good
const result = await pool.query(
  "SELECT * FROM users WHERE email = $1",
  [email]
);
```

#### B. Hardcoded Secrets & Weak Defaults
**Issue:** The `config` object provides default values for sensitive credentials (JWT Secret, DB Password).
*   **Location:** `config` object definition.
*   **Risk:** If these defaults are used in production, credentials are static and easily guessable.
*   **Fix:** Enforce environment variables. Exit the process if secrets are missing.

```typescript
// ❌ Bad
jwtSecret: process.env.JWT_SECRET || "default-secret-change-me",
dbPassword: process.env.DB_PASSWORD || "admin123",

// ✅ Good
if (!process.env.JWT_SECRET) throw new Error("JWT_SECRET is required");
if (!process.env.DB_PASSWORD) throw new Error("DB_PASSWORD is required");
```
