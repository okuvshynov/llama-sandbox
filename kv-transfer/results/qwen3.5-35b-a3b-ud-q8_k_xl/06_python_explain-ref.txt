This Python program solves the **Topological Sorting** problem for a Directed Acyclic Graph (DAG). It is commonly used to determine valid execution orderings for tasks with dependencies (e.g., build systems, course prerequisites).

The program provides two distinct functions:
1.  **`solve`**: Finds **one** valid topological ordering.
2.  **`find_all`**: Finds **all** possible valid topological orderings.

---

### 1. High-Level Logic

The input is a list of edges representing dependencies. For example, `("lint", "compile")` means "lint" must happen before "compile".
The program constructs a graph where nodes are tasks and directed edges represent "must precede" relationships. It then calculates the **in-degree** (number of incoming edges) for each node.

-   **Topological Sort Condition**: A node can only be processed (added to the result) if its in-degree is 0 (all its dependencies are satisfied).
-   **Cycle Detection**: If the graph contains a cycle, a topological sort is impossible. The `solve` function detects this by checking if the final result list contains all nodes.

---

### 2. Step-by-Step Algorithm Breakdown

#### Part A: Graph Construction (`build_graph`)
This helper function converts the raw list of edges into a structured format.
1.  **Adjacency List**: Creates a `defaultdict(list)` where keys are source nodes and values are lists of destination nodes (neighbors).
2.  **In-Degree Calculation**: Tracks how many dependencies each node has.
3.  **Node Registration**: Ensures isolated nodes (nodes with no edges) are included in the `nodes` set.
4.  **Initialization**: Iterates through all nodes to ensure every node has an entry in `in_degree` (defaulting to 0 if no incoming edges exist).

#### Part B: Finding One Valid Order (`solve`)
This implements **Kahn's Algorithm** for topological sorting.

1.  **Initialization**:
    -   Create a `deque` (queue) containing all nodes with an initial in-degree of 0. These are the starting tasks.
    -   Initialize an empty `result` list.
2.  **Processing Loop**:
    -   While the queue is not empty:
        -   Dequeue a `node`.
        -   Append it to `result`.
        -   For each `neighbor` of this node:
            -   Decrement the neighbor's in-degree (simulating the completion of the dependency).
            -   If the neighbor's in-degree drops to 0, it is now ready; add it to the queue.
3.  **Cycle Check**:
    -   After the loop, check if `len(result) == len(nodes)`.
    -   If they are equal, return `result` (a valid order).
    -   If not, a cycle exists (some nodes were never added to the queue); return `None`.

#### Part C: Finding All Valid Orders (`find_all`)
This function uses **Backtracking** (Depth-First Search) to explore every possible valid path.

1.  **State Management**:
    -   It maintains a `path` (current sequence of tasks) and a copy of the `remaining_in_degree` dictionary.
2.  **Recursive Backtracking (`backtrack`)**:
    -   **Base Case**: If `len(path)` equals the total number of nodes, a complete valid order is found. Add a copy of `path` to `results`.
    -   **Recursive Step**:
        -   Iterate through all `nodes` (sorted for consistent output).
        -   Check if the node is **not** already in the current `path` AND its `remaining_in_degree` is 0.
        -   If valid:
            1.  Add node to `path`.
            2.  Create a deep copy of `remaining_in_degree`.
            3.  Decrement the in-degree of all neighbors of the current node in this copy.
            4.  Recursively call `backtrack`.
            5.  **Backtrack**: Remove the node from `path` to explore other possibilities.

### 3. Analysis of the Provided Example

**Input Edges:**
-   `deps` $\to$ `compile`, `deps` $\to$ `lint`
-   `lint` $\to$ `compile`
-   `compile` $\to$ `link`, `compile` $\to$ `test`
-   `link` $\to$ `package`
-   `test` $\to$ `package`

**Graph Structure:**
-   **Start Nodes**: `deps` (in-degree 0).
-   **Flow**:
    1.  `deps` must run first.
    2.  After `deps`, `lint` becomes available (in-degree 0).
    3.  After `lint`, `compile` becomes available.
    4.  After `compile`, both `link` and `test` become available simultaneously.
    5.  `package` can only run after both `link` and `test` are done.

**Expected Output:**
-   **`solve`**: Will return one valid sequence, likely `['deps', 'lint', 'compile', 'link', 'test', 'package']` (or `link` and `test` swapped).
-   **`find_all`**: Will return all permutations where `link` and `test` can be swapped after `compile`, but `package` remains last.
    -   Order 1: `deps` $\to$ `lint` $\to$ `compile` $\to$ `link` $\to$ `test` $\to$ `package`
    -   Order 2: `deps` $\to$ `lint` $\to$ `compile` $\to$ `test` $\to$ `link` $\to$ `package`

### 4. Complexity Analysis

| Function | Time Complexity | Space Complexity | Notes |
| :--- | :--- | :--- | :--- |
| **`build_graph`** | $O(V + E)$ | $O(V + E)$ | $V$ = nodes, $E$ = edges |
| **`solve`** | $O(V + E)$ | $O(V)$ | Standard Kahn's algorithm |
| **`find_all`** | $O(V! \cdot (V+E))$ | $O(V)$ | In the worst case (no dependencies), there are $V!$ permutations. The complexity is exponential. |

### Conclusion
This program is a robust utility for dependency resolution.
-   Use **`solve`** if you only need *any* valid schedule (e.g., for a build system).
-   Use **`find_all`** if you need to analyze the full search space of valid schedules, though be aware it will be slow for large graphs with many nodes due to the exponential nature of generating all permutations.
