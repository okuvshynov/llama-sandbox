This codebase demonstrates a solid attempt at building a typed, event-driven task pipeline. However, there are several **critical architectural flaws**, **type safety gaps**, and **runtime bugs** that need addressing before this can be considered production-ready.

Here is a detailed review categorized by issue type.

---

### 1. Critical Type Safety & Generics Issues

#### A. The `any` Leak in `EventMap` and `onceListeners`
**Issue:** The `EventMap` interface uses `any[]` for arguments, and `Set<Function>` is used for listeners. This completely defeats the purpose of the `TypedEventEmitter`.
*   **Current:** `Set<Function>` allows any function to be added.
*   **Current:** `T[K]` in `on` is typed, but `onceListeners` stores `Set<Function>`, losing the specific signature of the listener.
*   **Consequence:** If you pass a listener with the wrong argument count to `once`, TypeScript won't catch it at the call site, and runtime errors will occur during `emit`.

**Fix:**
1.  Change `Set<Function>` to `Set<T[K]>`.
2.  Ensure `EventMap` uses specific tuple types for args, not `any[]`.

#### B. Generic Type Inference in `addTask`
**Issue:** The `addTask` method is generic (`<T>`) but the return type is `this`. While this allows chaining, the internal `Task<T>` type is created with `any` if `T` is not explicitly inferred, or the method forces `T` to be `any` if not passed.
*   **Specifically:** `const task: Task<T> = ...`. If the user doesn't explicitly pass `<string>` to `addTask`, `T` defaults to `any` in the context of the class properties unless inferred from the `execute` function.
*   **Risk:** The `execute` function might return a specific type, but the `Task` interface treats it as `any`, losing type safety for the `result` property.

**Fix:**
Use a helper type to infer `T` from the `execute` function signature automatically, or ensure the generic is strictly enforced.
