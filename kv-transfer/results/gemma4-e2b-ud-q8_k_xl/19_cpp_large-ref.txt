## 1. Memory Safety and Correctness Bugs

### A. Lock-Free Hash Map: The ABA Problem (Critical)

**Issue:** The `LockFreeHashMap` uses a standard CAS loop based only on the `uint64_t` key (`table_[probe].key`). This structure is susceptible to the **ABA problem**. If a key changes from $A \rightarrow B \rightarrow A$, a thread performing a CAS operation might mistakenly succeed because the key value is the same, even though the underlying state of the slot has changed (e.g., the value pointer or occupation status has been altered).

**Impact:** Incorrect state transitions, leading to data corruption or lost entries.

**Fix:** Implement a method to mitigate the ABA problem, such as using a **tagged pointer** (if you were using pointers) or a **counter/version number** alongside the key in the atomic field.

1.  Change the `Entry::key` to include a version counter:

    ```cpp
    struct Entry {
        std::atomic<uint64_t> key;
        std::atomic<uint64_t> version; // New version counter
        std::atomic<V*> value;
        // ...
    };
    ```
2.  Modify the CAS logic to operate on both key and version simultaneously (if possible, using a structure or a more complex CAS sequence). If using a separate counter, the CAS would look for both the key and the expected version.
