## 1. Type Safety and TypeScript Issues

### Issue 1.1: Over-reliance on `any` in Event Handlers

The `EventMap` interface uses `(...args: any[]) => void`. While this allows maximum flexibility, it bypasses TypeScript's benefits entirely for event payload handling.

**Why it's a problem:** If a developer emits an event with a specific payload (e.g., `task:complete` should contain a `Task` object and a `Result`), TypeScript cannot enforce this structure, leading to runtime errors.

**Suggested Improvement:**
Define specific interfaces for the payload of each event type, or use generics if the emitter class is generalized.

```typescript
// Instead of EventMap
interface TaskCompleteEvent {
  task: Task;
  result: any;
}

interface PipelineEvents extends EventMap {
  'task:complete': TaskCompleteEvent; // Use specific types
  // ... other events
}
```

### Issue 1.2: Weak Typing in Task Results and Dependencies

The `Task<T = any>` interface is too generic. The `T` parameter allows the task result type to be anything, which weakens the type safety of the entire pipeline data flow.

**Why it's a problem:** When retrieving results (e.g., `pipeline['results'].get('fetch-users')`), the type is inferred as `any`, forcing developers to perform manual casting or runtime checks.

**Suggested Improvement:**
Use generics more strictly. If possible, make the `Pipeline` itself generic, or ensure that the task type is strongly defined when adding tasks.

### Issue 1.3: Implicit Type Handling in `executeTask`

In `executeTask`, error handling uses a weak guard:

```typescript
lastError = err instanceof Error ? err : new Error(String(err));
```

This is often necessary in JS environments, but in a strongly typed system, we should strive to maintain `Error` objects throughout.

**Suggested Improvement:**
Ensure that the `execute` function signatures are strictly typed to return `Promise<T>` (or `Promise<any>`), which helps the `catch` block handle standard `Error` objects automatically.

## 2. Architectural and Encapsulation Problems

### Issue 2.1: Violation of Encapsulation in Task Execution (Critical)

The most significant architectural flaw lies in how tasks access shared pipeline state within their `execute` functions.

**Example of the flaw:**

```typescript
// Inside addTask setup:
.addTask({
  // ...
  execute: async () => {
    const users = pipeline['results'].get('fetch-users'); // Accessing external state
    // ...
  },
})
```

**Why it's a problem:**
1. **Tight Coupling:** Tasks are tightly coupled to the concrete `Pipeline` instance. They cannot be easily tested or reused outside the pipeline context.
2. **Lack of Data Flow Control:** The pipeline orchestrator controls the flow of data (`results`), but the task itself needs access to that data. This mixing of responsibilities makes the system fragile.
3. **Type Safety Loss:** As noted above, accessing `pipeline['results']` bypasses the type system entirely.

**Suggested Improvement: Dependency Injection (The "Pipeline Context")**

Tasks should not be standalone functions. They should be objects or classes that receive the necessary context (e.g., the `results` map, or a method to retrieve results) upon execution.
