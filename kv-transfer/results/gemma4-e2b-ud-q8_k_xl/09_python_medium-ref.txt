## 1. Performance Issues

### Issue 1: Inefficient LRU Implementation ($O(N)$ Time Complexity)

The `LRUEvictionPolicy` uses a standard Python list (`self.access_order`) to track access.

*   **Problem:** The `record_access` method checks if the key exists and, if so, uses `self.access_order.remove(key)`. List removal requires searching for the item ($O(N)$) and shifting subsequent elements ($O(N)$). In a high-traffic cache, this becomes a major performance bottleneck, defeating the purpose of an efficient LRU policy.

*   **Fix:** Use an optimized data structure for LRU, such as a **doubly linked list combined with a hash map (dictionary)**. This allows for $O(1)$ access, insertion, and deletion of nodes.

    *   **Implementation Suggestion:** Use `collections.OrderedDict` (if Python version allows) or implement a custom dictionary structure where the key maps to the node in a doubly linked list.

### Issue 2: Inefficient TTL Expiration Check ($O(N)$ Time Complexity)

The `TTLEvictionPolicy.get_expired_keys` method iterates through every single entry in the cache dictionary.

*   **Problem:** If the cache grows large, checking for expired keys requires $O(N)$ time, and this operation is performed periodically within the cache's main lock (`cleanup_expired`).

*   **Fix:** Instead of iterating over the entire map, structure the TTL policy to only check entries when they are *about* to expire or use a structure that allows quick access to the expiration time (e.g., a min-heap prioritized by `expires_at` time, although this complicates removal).

    *   **Simpler Fix:** During a standard `get` or `put` operation, check the expiration time for that specific key immediately. If the key is found to be expired, remove it right then and there. This turns expiration checking into an $O(1)$ operation on demand rather than an $O(N)$ sweep.

### Issue 3: Excessive Database I/O (Logging Bottleneck)

The `_log_operation` method commits a transaction to the SQLite database on *every single cache operation* (`get`, `put`, `delete`).

*   **Problem:** Database I/O is orders of magnitude slower than memory operations. Committing frequently will serialize all cache access operations, turning a high-performance cache into a slow, disk-bound system.

*   **Fix:** **Batch the logging operations.** Instead of committing immediately, buffer the log entries in memory and write them to the database in batches (e.g., every 100 operations, or every few seconds). The database commit should only happen at the end of the batch or when explicitly flushed.

### Issue 4: Key Hashing Performance

The `_hash_key` method uses `hashlib.md5`. While secure, MD5 hashing is generally slower than non-cryptographic hashing functions when used solely for internal key lookups.

*   **Problem:** MD5 is a cryptographic hash, introducing unnecessary computational overhead for simple cache key mapping.

*   **Fix:** Use a faster, non-cryptographic hash function like **FNV-1a** or Python's built-in `hash()` function (if sufficient collision resistance for the application context is acceptable, though caution is needed with `hash()` across processes/sessions).

## 2. Design and Concurrency Issues

### Issue 5: Coarse-Grained Locking (Lock Contention)

The `CacheStore` uses a single `threading.Lock()` to protect *all* internal state (`entries`, `current_size`, `lru`, `ttl`, and database operations).

*   **Problem:** This lock means that if one thread is performing a long operation (e.g., a large `put` that triggers multiple evictions, or a database log write), all other threads attempting any operation (`get`, `put`, `delete`) are blocked. This severely limits concurrency and throughput.

*   **Fix:** **Refine the locking strategy.**
    1.  **Read/Write Lock (RWLock):** Use a mechanism that allows multiple readers simultaneously but locks out writers. Since `get` operations are typically much more frequent than `put` or `delete`, this could significantly improve read throughput. (Note: Python standard library doesn't have a native RWLock, requiring a custom implementation or external library.)
    2.  **Fine-Grained Locking:** Separate locks for independent components. For instance, use one lock for the main `entries` dictionary and separate locks for the eviction policies if they were independently managed.
