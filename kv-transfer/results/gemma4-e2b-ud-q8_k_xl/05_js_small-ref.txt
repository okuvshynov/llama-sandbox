## 🐞 Critical Bugs (Functional Errors)

### 1. Bug: `reschedule` does not maintain the sorted order of events.

**Location:** `reschedule(name, newTime)`

**Explanation:**
The `schedule` method correctly sorts the `this.events` array by time (`this.events.sort((a, b) => a.time - b.time)`). However, the `reschedule` method simply finds the event by name and updates its `time` property.

It fails to re-sort the array. If an event's time is moved earlier or later, the array is no longer chronologically ordered. When `run()` executes, it relies entirely on this sorted order to correctly calculate the required waiting time (`wait = event.time - elapsed`). If the array is unsorted, the scheduler will execute events in the wrong sequence, leading to incorrect timing and behavior.

**Fix:**
After updating the event time, the scheduler must re-sort the list:

```javascript
  reschedule(name, newTime) {
    for (const event of this.events) {
      if (event.name === name) { // Use === for strict comparison
        event.time = newTime;
        break; // Optimization: found it, stop searching
      }
    }
    // CRITICAL FIX: Re-sort the events after modifying the time
    this.events.sort((a, b) => a.time - b.time); 
  }
```

### 2. Bug: Inefficient search in `reschedule`.

**Location:** `reschedule(name, newTime)`

**Explanation:**
The current implementation of `reschedule` uses a linear loop (`for (const event of this.events)`) to find the event by name. If the number of scheduled events ($N$) is large, this operation is $O(N)$.

**Fix (Optimization):**
If performance is critical, the `this.events` structure should be changed from a simple array to a Map or a structure that allows $O(1)$ lookup by name, perhaps storing event objects in a separate map keyed by the event name.

## ⚠️ Minor Bugs & Design Flaws (Style & Robustness)

### 3. Flaw: Inconsistent comparison in `reschedule`.

**Location:** `reschedule(name, newTime)`

**Explanation:**
The code uses loose equality (`==`) when comparing the event name: `if (event.name == name)`. While this might work in simple cases, it is best practice in JavaScript to use strict equality (`===`) to avoid unexpected type coercion issues.

**Fix:**
Change `event.name == name` to `event.name === name`.

### 4. Flaw: Lack of event cleanup in `cancel`.

**Location:** `cancel(name)`

**Explanation:**
The `cancel` method removes the event from the `this.events` list, but it does not remove the corresponding handlers from the `this.listeners` map. If a user cancels an event, the associated handler still exists and will fire when the scheduler runs, potentially causing errors or unnecessary execution.

**Fix:**
To fully clean up, you would need a way to map event names to the handlers registered for them, and then iterate through those handlers to remove the specific `handler` provided to `off` (which currently doesn't exist, `off` only removes by handler reference). A simpler approach for this structure might be to provide a method to clear all listeners for a specific event name.

### 5. Flaw: Potential for Blocking the Event Loop (Design Constraint).

**Location:** `run()`

**Explanation:**
The scheduler uses `await new Promise(resolve => setTimeout(resolve, wait));` to pause execution between events. This is the correct way to handle asynchronous waiting in JavaScript. However, if any of the `event.callback()` functions execute a very long-running synchronous task (e.g., heavy computation, synchronous I/O), they will block the entire Node.js/Browser event loop, causing all other scheduled events and I/O operations to stall.

**Mitigation (Design Recommendation):**
For a production-grade scheduler, you should consider running the event callbacks inside separate worker threads (if in Node.js) or ensuring that all callbacks are non-blocking (asynchronous) to maintain the responsiveness of the environment.
