## 🐞 Major Bug: Missing Lock in `Get` Method

The most critical flaw is that the `Get` method does not acquire the mutex before reading the shared `c.counts` map. This creates a **data race** when used concurrently with the `Increment` method.

### Code Snippet (The Bug)

```go
func (c *Counter) Get(key string) int {
	return c.counts[key] // <-- BUG: Reading a map without a lock
}
```

### Explanation

1.  **Race Condition:** The `Increment` method modifies the map (`c.counts`). If a goroutine calls `Increment` while another goroutine calls `Get`, both are accessing or modifying the underlying map structure simultaneously.
2.  **Unsafe Read:** While reading from a Go map is sometimes safe in isolation, if another goroutine is simultaneously resizing the map (which happens during an insertion/increment), the reading goroutine can encounter undefined behavior, leading to crashes, corrupted data, or inconsistent results.
3.  **Requirement:** Any function that reads or writes to a shared data structure (like a map) that is also being written to by other goroutines **must** be protected by the same mutex used for writing.

### Solution

The `Get` method must acquire the lock, read the value, and release the lock.

```go
func (c *Counter) Get(key string) int {
	c.mu.Lock() // Acquire the lock
	defer c.mu.Unlock() // Ensure the lock is released when the function exits
	return c.counts[key]
}
```

## 💡 Minor Issue 1: Inefficient `Increment` Logic (Minor Performance/Clarity)

While the current implementation of `Increment` is technically thread-safe because it holds the lock for the entire operation, it can be slightly simplified and made clearer.

### Code Snippet (The Code)

```go
func (c *Counter) Increment(key string) int {
	c.mu.Lock()
	c.counts[key]++
	val := c.counts[key]
	c.mu.Unlock()
	return val
}
```

### Explanation

The current code locks, increments, reads the value again, and unlocks. While correct, retrieving the value inside the lock is necessary for thread safety.

A slight optimization is to retrieve the value *before* the lock if you only needed to return the current value, but since you need to return the *new* incremented value, holding the lock for the full read/write cycle is the correct approach. The logic is fine, but it highlights the necessity of holding the lock for the entire transaction.

**No change is strictly required here, but it's worth noting that the lock is correctly held throughout the operation.**
