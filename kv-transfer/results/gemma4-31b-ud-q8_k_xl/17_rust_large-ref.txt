### 1. Critical Bugs & Soundness (Undefined Behavior)

#### A. Data Race in `TaskDeque::steal`
**Issue:** The `steal` method uses `std::ptr::read` to move a `Task` out of a `Vec<Option<Task>>` while the owner thread might be concurrently modifying that same index in `push` or `pop`.
**Explanation:** `Vec` and `Option` are not atomic. Reading a value while another thread writes to it is a data race, which is **Undefined Behavior (UB)** in Rust. Furthermore, `ptr::read` creates a duplicate of the `Task` (and its internal `Arc`). If the CAS fails, you call `mem::forget`, but if it succeeds, you have effectively "stolen" the value without the owner thread knowing the slot is now logically empty, leading to potential double-frees.
**Fix:** Use a proper lock-free deque implementation (like `crossbeam-deque`) or protect the buffer with a `Mutex`/`RwLock`. If building from scratch, use `AtomicPtr` or `UnsafeCell` with strict memory barriers.

#### B. Future Dropping on `Poll::Pending`
**Issue:** In `WorkerContext::execute_task`, the future is polled exactly once. If it returns `Poll::Pending`, the future is dropped.
**Explanation:** This is the most significant logic error. In async Rust, `Pending` means "I'm not done, wake me up later." By dropping the future, the task is deleted from existence. It will never complete.
**Fix:** If `poll` returns `Pending`, the future must be put back into `task.inner.future` and the task must be re-enqueued in the scheduler to be polled again when its waker is triggered.

#### C. Circular References (Memory Leak)
**Issue:** `Task::with_dependencies` creates a strong `Arc` cycle.
**Explanation:** `inner.dependencies` holds `Arc<TaskInner>` of the parents, and `dep.dependents` holds `Arc<TaskInner>` of the child. Neither side uses `Weak`. These tasks will never be dropped, even after completion.
**Fix:** Change `dependents` to `Mutex<Vec<Weak<TaskInner>>>`.

#### D. Broken Result Channel
**Issue:** In `Scheduler::submit`, the `oneshot::Sender` (`result_tx`) is created and immediately dropped.
**Explanation:** Because the sender is not stored anywhere, the `result_rx` in the `TaskHandle` will immediately receive a `RecvError` (disconnected) when `wait_result` is called.
**Fix:** Store the `oneshot::Sender<TaskResult>` inside the `TaskInner` struct. The worker thread should send the result through this channel upon completion.

### 2. Concurrency & Performance Issues

#### A. Blocking the Tokio Runtime
**Issue:** `worker_loop` calls `ctx.overflow.pop_blocking`, which uses `std::sync::Condvar`.
**Explanation:** The workers are spawned via `tokio::spawn`. Tokio uses a cooperative multi-threading model. Calling a blocking `std::sync` primitive blocks the entire OS thread. If all workers block on the `Condvar`, the entire Tokio runtime freezes, and no other async tasks (including the `dispatch_loop`) can run.
**Fix:** Replace `OverflowQueue` with a `tokio::sync::mpsc` channel or a `tokio::sync::Semaphore` to allow workers to yield the thread while waiting for tasks.
