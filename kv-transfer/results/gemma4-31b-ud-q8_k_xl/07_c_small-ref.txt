Here is the detailed breakdown of the bugs:

### 1. Memory Leak on Value Update (`table_put`)
In `table_put`, when a key is found that already exists in the table, the program updates the value using `strdup`.
```c
if (strcmp(t->entries[i].key, key) == 0) {
    t->entries[i].value = strdup(value); // BUG: Memory Leak
    return;
}
```
**The Issue:** `t->entries[i].value` already points to a string allocated by a previous `strdup`. By assigning a new pointer to it without calling `free()` on the old pointer first, the previous string becomes unreachable in memory.

**The Fix:**
```c
if (strcmp(t->entries[i].key, key) == 0) {
    free(t->entries[i].value); // Free the old value first
    t->entries[i].value = strdup(value);
    return;
}
```

### 2. Memory Leak on Entry Deletion (`table_delete`)
The `table_delete` function uses a "swap-with-last" strategy to remove an element in $O(1)$ time.
```c
if (strcmp(t->entries[i].key, key) == 0) {
    t->entries[i] = t->entries[t->size - 1]; // BUG: Memory Leak
    t->size--;
    return;
}
```
**The Issue:** When you overwrite `t->entries[i]` with the last entry in the array, the pointers to the `key` and `value` of the entry being deleted are overwritten. Since those strings were allocated via `strdup`, they are now leaked.

**The Fix:**
```c
if (strcmp(t->entries[i].key, key) == 0) {
    free(t->entries[i].key);   // Free deleted key
    free(t->entries[i].value); // Free deleted value
    t->entries[i] = t->entries[t->size - 1];
    t->size--;
    return;
}
```

### 3. Unsafe `realloc` Usage (`table_put`)
When the table reaches capacity, it expands using `realloc`.
```c
t->entries = realloc(t->entries, sizeof(Entry) * t->capacity);
```
**The Issue:** This is a classic C pitfall. If `realloc` fails, it returns `NULL`. By assigning the result directly back to `t->entries`, you overwrite the only pointer you had to the existing memory block. This results in:
1. A memory leak of the original `entries` array.
2. A crash (segmentation fault) when the program subsequently tries to access `t->entries`.

**The Fix:** Use a temporary pointer to hold the result of `realloc`.
```c
int new_capacity = t->capacity * 2;
Entry* temp = realloc(t->entries, sizeof(Entry) * new_capacity);
if (temp == NULL) {
    // Handle allocation failure (e.g., return error or exit)
}
t->entries = temp;
t->capacity = new_capacity;
```

### 4. Missing NULL Checks (General)
The program calls `malloc`, `realloc`, and `strdup` throughout, but never checks if they return `NULL`.
**The Issue:** On systems with limited memory, any of these calls could fail. Attempting to dereference a `NULL` pointer (e.g., `t->entries[t->size].key = strdup(key)`) will cause the program to crash immediately.
