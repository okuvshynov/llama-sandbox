### 1. Critical Bugs & Memory Leaks

#### A. The "Zombie Timer" Leak
In `executeTask`, you use `Promise.race` to implement a timeout. 
```typescript
const result = await Promise.race([
  task.execute(),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Task ${task.id} timed out`)), task.timeout)
  ),
]);
```
**Issue:** The `setTimeout` is never cleared. If `task.execute()` finishes in 10ms but the timeout is 30s, the timer remains in the Node.js/Browser event loop for the full 30s. In a high-throughput pipeline, this will lead to massive memory consumption and delayed process exit.
**Fix:** Use a variable to hold the timer ID and call `clearTimeout()` immediately after the race resolves.

#### B. "Wave" Concurrency vs. True Concurrency
The `run()` method processes tasks in batches:
```typescript
const batch = ready.slice(0, this.concurrency);
await Promise.all(batch.map(task => this.executeTask(task)));
```
**Issue:** This is not true concurrency; it is "wave" execution. If your concurrency is 4, and 3 tasks finish in 1 second while 1 task takes 10 seconds, the pipeline will sit idle for 9 seconds before starting the next batch.
**Fix:** Use a worker-pool pattern where a new task is started immediately whenever a slot becomes available.

#### C. The Polling Loop
The `while(true)` loop with `await new Promise(r => setTimeout(r, 100))` is an anti-pattern.
**Issue:** It introduces unnecessary latency (up to 100ms per cycle) and wastes CPU cycles polling for state changes.
**Fix:** Transition to an event-driven architecture. When a task completes, it should trigger a check for newly "ready" tasks.

#### D. Ineffective Cancellation
The `cancel()` method simply changes the status to `'cancelled'`.
**Issue:** Since JavaScript Promises are not natively cancellable, the `execute()` function continues to run in the background until it completes or times out, potentially causing side effects or consuming resources.
**Fix:** Pass an `AbortSignal` to the `execute` function.

### 2. Type Safety Issues

#### A. Use of `Function` and `any`
In `TypedEventEmitter`, you use `Set<Function>`.
**Issue:** `Function` is a "top type" that provides no type safety for arguments. While the `emit` method uses `Parameters<T[K]>` to ensure the *caller* provides the right types, the internal storage loses all safety.
**Fix:** Use a generic internal type or a type assertion to `(...args: any[]) => void`.

#### B. Result Type Erasure
The `Task` interface allows a generic `T`, but the `Pipeline` class stores tasks in a `Map<string, Task>`.
**Issue:** The specific return type of each task is lost. In the usage example, the user has to use `pipeline['results'].get(...)` and cast it to `any`.
**Fix:** While mapping task IDs to types is complex in TypeScript, you can improve the `results` map by providing a typed `getResult<T>(id: string): T` method.

#### C. Private Member Access
In the usage example: `pipeline['results'].get('fetch-users')`.
**Issue:** Using string indexing to access `private` members bypasses the compiler's access modifiers. This indicates a missing public API.
