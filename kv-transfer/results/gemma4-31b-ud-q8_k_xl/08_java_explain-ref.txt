This Java program implements an **LRU (Least Recently Used) Cache**. 

An LRU Cache is a specialized cache that stores a limited number of items. When the cache reaches its maximum capacity and a new item needs to be added, it automatically removes (**evicts**) the item that has not been accessed for the longest period of time.

---

### How it Works (The Logic)

The program manages data based on "recency" of use:
1.  **`get(K key)`**: When you retrieve a value, that item becomes the "most recently used." The program moves it to the front of the queue.
2.  **`put(K key, V value)`**: 
    *   If the key already exists, its value is updated, and it is moved to the front.
    *   If the key is new and the cache is full, the program identifies the "least recently used" item (the one at the very back of the queue), deletes it from both the map and the list, and then inserts the new item at the front.

---

### The Data Structure
The program uses a **hybrid data structure** combining a **HashMap** and a **Doubly Linked List**. This combination is used to achieve $O(1)$ (constant time) performance for both adding and retrieving elements.

#### 1. The HashMap (`Map<K, Node<K, V>>`)
*   **Purpose**: Fast lookup.
*   **Role**: It stores the key and a reference (a pointer) to the corresponding node in the linked list. Without the map, finding an item in the list would take $O(n)$ time; with the map, it takes $O(1)$.

#### 2. The Doubly Linked List (`Node` class)
*   **Purpose**: Maintaining chronological order.
*   **Role**: 
    *   **The Head**: Represents the **Most Recently Used (MRU)** end.
    *   **The Tail**: Represents the **Least Recently Used (LRU)** end.
    *   **Nodes**: Each node contains the key, the value, and pointers to both the `next` and `prev` nodes. This allows the program to remove a node from the middle of the list and move it to the front in $O(1)$ time.
*   **Sentinels**: The `head` and `tail` nodes are "dummy" nodes. They don't hold actual data but act as boundaries to simplify the logic (preventing `null` pointer checks when adding or removing nodes).

### Complexity Analysis

| Operation | Time Complexity | Space Complexity | Reason |
| :--- | :--- | :--- | :--- |
| **`get()`** | $O(1)$ | $O(1)$ | HashMap provides instant access; moving a node in a linked list is a constant-time pointer update. |
| **`put()`** | $O(1)$ | $O(1)$ | Inserting into a HashMap and adding to the head of a linked list are both constant-time operations. |
| **Overall** | | $O(\text{capacity})$ | The space used grows linearly with the defined capacity of the cache. |

### Summary of the Example in `main`
1.  **Put a, b, c**: Cache is `[c, b, a]`.
2.  **Get a**: `a` is accessed, so it jumps to the front. Cache: `[a, c, b]`.
3.  **Put d**: Cache is full (capacity 3). The last item (`b`) is evicted. `d` is added to the front. Cache: `[d, a, c]`.
4.  **Put b**: Cache is full. The last item (`c`) is evicted. `b` is added to the front. Cache: `[b, d, a]`.
