While the logic is generally clear, there are several critical **soundness** issues (potential panics), **performance** bottlenecks (algorithmic complexity), and **API design** flaws (memory leaks and dead code).

### 1. Soundness & Safety Issues

#### Floating Point Panics (`NaN` handling)
In `Metric::percentile`, you use:
```rust
sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
```
**Issue:** `f64` does not implement `Ord` because `NaN` values cannot be compared. If a `NaN` ever enters your metrics (e.g., from a division by zero elsewhere in the app), `partial_cmp` returns `None`, and `.unwrap()` will cause the entire thread to **panic**.
**Fix:** Use `a.total_cmp(b)` (available since Rust 1.62), which provides a total ordering for floats, including `NaN`.

#### Integer Underflow
In `Metric::percentile`:
```rust
sorted[idx.min(sorted.len() - 1)]
```
**Issue:** Although you have an `is_empty` check at the top, this pattern is dangerous. If the `is_empty` check were removed or modified, `sorted.len() - 1` would underflow on an empty vector, causing a panic.
**Fix:** Use `sorted.get(idx.min(sorted.len().saturating_sub(1)))`.

### 2. Performance Problems

#### Computational Complexity of Percentiles
The `percentile` method is called three times per metric during every `snapshot()`:
```rust
let mut sorted = self.values.clone();
sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
```
**Issue:** This is an $O(N \log N)$ operation that clones the entire dataset. If a metric has 1 million samples, you are cloning and sorting 3 million elements every time you take a snapshot. This will cause massive latency spikes in the system being monitored.
**Fix:** Use a **Histogram** (which you've implemented but aren't using!) or a data structure like `T-Digest` or `HdrHistogram` to calculate approximate quantiles in constant time.

#### Unbounded Memory Growth (Memory Leak)
The `Metric` struct stores every single value and timestamp in a `Vec`.
**Issue:** While you implemented `trim_older_than`, **it is never called**. The `values` and `timestamps` vectors will grow indefinitely until the application runs out of memory (OOM).
**Fix:** Implement a background cleanup thread or call `trim_older_than` inside the `record` method periodically.

#### Linear Scan for Rates
The `rate` method iterates through the entire history:
```rust
self.timestamps.iter().filter(|t| now.duration_since(**t) < window).count();
```
**Issue:** This is $O(N)$. As the application runs longer, calculating the rate becomes slower and slower.
**Fix:** Since timestamps are inserted monotonically, use **binary search** (`slice::binary_search_by`) to find the index of the first timestamp within the window, then subtract that index from the total length. This reduces complexity to $O(\log N)$.

#### Cache Locality
You store data in two parallel vectors: `values: Vec<f64>` and `timestamps: Vec<Instant>`.
**Issue:** This is cache-unfriendly. When calculating rates or trimming, the CPU has to jump between two different memory regions.
**Fix:** Store them as a single vector of tuples: `Vec<(Instant, f64)>`.
