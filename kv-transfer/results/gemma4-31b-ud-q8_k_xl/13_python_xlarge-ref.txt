### 1. Critical Bugs & Logic Errors

#### A. The "Wipe-All" Reservation Cleanup
**Issue:** In `InventoryManager.cleanup_expired_reservations`, the code iterates through all reservations and adds every single one to the `expired` list, regardless of the actual timestamp.
**Explanation:** 
```python
for order_id in self.reservations:
    expired.append(order_id) # This collects EVERY reservation
```
This means calling this method deletes all pending reservations, potentially causing orders to fail at the payment stage.
**Fix:** Store a timestamp with the reservation and compare it against `datetime.now() - self.reservation_timeout`.

#### B. Broken Cohort Analysis
**Issue:** In `AnalyticsEngine.cohort_analysis`, every user is assigned to `period_idx = 0`.
**Explanation:** 
```python
period_idx = 0
cohort_users[period_idx].add(uid)
```
The logic fails to calculate which period (day/week) the user actually joined. As a result, the cohort analysis is just a general retention rate for the entire history of the app.
**Fix:** Calculate `period_idx` by finding the difference between the event timestamp and a fixed reference date (e.g., the start of the month).

#### C. Floating Point Currency
**Issue:** Use of `float` for prices, taxes, and totals across all files.
**Explanation:** Floating-point math (`0.1 + 0.2 != 0.3`) leads to rounding errors in financial transactions.
**Fix:** Use the `decimal.Decimal` class for all monetary values.

#### D. Inventory Leak on Cancellation
**Issue:** In `OrderProcessor.cancel_order`, if an order has already been processed (committed), the payment is refunded, but the stock is never returned to the inventory.
**Explanation:** `inventory.release_reservation` only works for pending orders. Once `commit_reservation` is called, the stock is deducted from `product.stock_count`.
**Fix:** Call `inventory.restock()` for each item in the order when cancelling a `CONFIRMED` or `SHIPPED` order.
