### 1. Nil Map Panic (Critical)
In `NewCounter()`, the `Counter` struct is initialized, but the `counts` map is not.
```go
func NewCounter() *Counter {
    return &Counter{} // counts is nil here
}
```
In Go, reading from a `nil` map is safe (it returns the zero value), but **writing to a `nil` map causes an immediate runtime panic**. As soon as someone hits the `/inc` endpoint, the server will crash.

**Fix:** Initialize the map using `make`.
```go
func NewCounter() *Counter {
    return &Counter{
        counts: make(map[string]int),
    }
}
```

---

### 2. Concurrent Map Read/Write Panic (Critical)
The `Get` method accesses the map without acquiring the mutex.
```go
func (c *Counter) Get(key string) int {
    return c.counts[key] // No lock!
}
```
Go maps are not thread-safe. If one goroutine is writing to the map (via `Increment`) while another is reading from it (via `Get`), the Go runtime will trigger a non-recoverable panic: `fatal error: concurrent map read and map write`.

**Fix:** Add `c.mu.Lock()` and `c.mu.Unlock()` (or use a `sync.RWMutex` for better performance on reads).

---

### 3. Map Reference Leak / Race Condition (Critical)
The `GetAll` method returns the internal map directly.
```go
func (c *Counter) GetAll() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.counts // Returns the reference to the map
}
```
Because maps are reference types in Go, the caller (`handleGetAll`) receives a pointer to the internal map. The mutex is unlocked the moment `GetAll` returns. Then, `json.NewEncoder(w).Encode(data)` starts iterating over that map to turn it into JSON. 

If a request to `/inc` arrives while the JSON encoder is still reading the map, the server will panic due to a **concurrent map read and map write**.

**Fix:** Create a copy of the map while holding the lock and return the copy.
```go
func (c *Counter) GetAll() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    copy := make(map[string]int)
    for k, v := range c.counts {
        copy[k] = v
    }
    return copy
}
```

### 4. Missing Return after `http.Error` (Logic Bug)
In `handleIncrement`, if the `key` is missing, the server sends an error response but continues executing the function.
```go
if key == "" {
    http.Error(w, "missing key", 400)
    // Missing 'return' here!
}
val := counter.Increment(key) // This still runs with key == ""
```
The user will receive a 400 Bad Request response, but the server will still increment a counter for the empty string `""` and then attempt to write the result to the response body (which will result in a "superfluous response.WriteHeader" warning in the logs).

**Fix:** Add a `return` statement after `http.Error`.
