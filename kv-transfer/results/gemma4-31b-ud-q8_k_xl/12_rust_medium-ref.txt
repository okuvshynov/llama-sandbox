### 1. Soundness & Correctness Issues

#### A. Panic on `NaN` Values
In `Metric::percentile`, the code uses:
```rust
sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
```
In Rust, `f64` does not implement `Ord` because `NaN` (Not-a-Number) cannot be compared. If a `NaN` value ever enters the `values` vector, `partial_cmp` returns `None`, and `.unwrap()` will cause the entire thread to **panic**.
*   **Fix:** Use `sort_by(|a, b| a.partial_cmp(b).unwrap_or(std::cmp::Ordering::Equal))` or filter out `NaN`s during recording.

#### B. Unbounded Memory Growth (Memory Leak)
The `Metric` struct stores every single recorded value and timestamp in two `Vec`s. While a `trim_older_than` method exists, **it is never called** anywhere in the registry or the simulation. 
*   In a production system, this will lead to an `OutOfMemory` (OOM) crash as the vectors grow indefinitely.
*   **Fix:** Implement a background cleanup task or call `trim_older_than` inside `record` or `snapshot`.

#### C. Potential Division by Zero
In `Metric::rate`, the code divides by `window.as_secs_f64()`. If a user passes `Duration::ZERO`, the result is `Inf` or `NaN`. While not a crash, it can corrupt downstream data aggregation.

---

### 2. Performance Problems

#### A. Massive Allocation/Sorting in `percentile`
The `percentile` method is the most expensive part of the code:
```rust
let mut sorted = self.values.clone(); // O(N) allocation
sorted.sort_by(...);                // O(N log N) CPU
```
In `snapshot()`, this is called **three times** (p50, p95, p99). If a metric has 1 million samples, `snapshot()` will allocate and sort 3 million floats every time it is called.
*   **Fix:** Sort the vector **once** inside `snapshot()` and pass the sorted slice to a helper function that calculates percentiles. Better yet, use a **Histogram** (which you implemented but didn't use) for constant-time/space approximations.

#### B. Linear Scans in `rate()`
The `rate` method performs a linear scan (`.filter().count()`) over the entire history of timestamps. As the number of samples grows, the cost of taking a snapshot increases linearly, making the monitoring system the primary bottleneck of the application.
*   **Fix:** Use a sliding window counter or a ring buffer.

#### C. Lock Contention (RwLock + Mutex)
The `MetricsRegistry` uses a `RwLock<HashMap<... Arc<Mutex<Metric>>>>`. 
1.  To record a value, the code acquires a **read lock** on the map.
2.  Then it acquires a **mutex lock** on the specific metric.
3.  If the metric doesn't exist, it drops the read lock and acquires a **write lock** on the map.

While this "double-checked locking" is okay, the `snapshot()` method holds the `RwLock` read lock for the entire duration of the snapshot, while sequentially locking every single `Mutex` in the map. This can block any new metrics from being created for a significant amount of time.
*   **Fix:** Use a concurrent map like `DashMap` to reduce lock contention.

---

### 3. API Design Concerns

#### A. Unused Implementation (`Histogram`)
You implemented a sophisticated `Histogram` struct with linear interpolation for quantiles, but the `MetricsRegistry` doesn't use it. It uses the naive `Metric` (which stores all raw samples) instead.
