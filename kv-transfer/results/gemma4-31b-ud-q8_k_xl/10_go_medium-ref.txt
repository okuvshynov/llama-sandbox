        *   *Issue:* This is a blocking send. If the queue is full, the worker is stuck. Worse, it might create a deadlock if all workers are stuck trying to requeue failing jobs while the system is full.

### 1. Critical Bugs & Race Conditions

#### A. Data Race on `Job` Object
**Problem:** The `Job` struct is shared between worker goroutines and HTTP handler goroutines. 
- The worker modifies `job.Status`, `job.Result`, and `job.Retries`.
- The `/status` and `/stats` handlers read these same fields.
- **Why it's a bug:** In Go, concurrent read/write access to the same memory location is a data race, which leads to undefined behavior or crashes. The `WorkerPool.mu` only protects the **map**, not the **individual Job objects** inside the map.

**Fix:** Add a `sync.Mutex` to the `Job` struct and lock it whenever any field is read or written.

#### B. Panic on Shutdown (Nil Pointer Dereference)
**Problem:** In the `worker` method:
```go
case job := <-wp.jobQueue:
    wp.processJob(id, job)
```
**Why it's a bug:** When `Stop()` is called, it closes `wp.jobQueue`. When a channel is closed, any subsequent receives return the zero value immediately. For a channel of pointers (`*Job`), the zero value is `nil`. The worker will receive `nil`, pass it to `processJob`, and the program will panic the moment it hits `job.Status = "failed"`.

**Fix:** Use the "comma-ok" idiom to check if the channel was closed:
```go
case job, ok := <-wp.jobQueue:
    if !ok { return }
    wp.processJob(id, job)
```

#### C. Blocking `Submit` and Deadlock Risk in Retries
**Problem:** `wp.jobQueue <- job` is a blocking operation.
- **Submit:** If the `queueSize` is reached, the HTTP request will hang indefinitely until a worker frees a slot.
- **Retries:** If all workers are processing jobs that fail and try to requeue themselves while the queue is full, the system can deadlock or experience severe performance degradation.

**Fix:** Use a `select` block with a `default` case in `Submit` to return a `429 Too Many Requests` or `503 Service Unavailable` error if the queue is full.

### 2. Design Issues

#### A. Context Neglect
**Problem:** The `WorkerPool` has a context, but it is only used to stop the workers. It is **not** passed to the `handlers`.
- **Why it's an issue:** If a handler is performing a long-running task (e.g., a database query or API call), it will continue running even after `Stop()` is called, because the handler has no way of knowing the system is shutting down.

**Fix:** Update the handler signature to `func(context.Context, string) (string, error)`.

#### B. Lack of Thread-Safety for Configuration
**Problem:** `RegisterHandler` and `Use` modify maps and slices. 
- **Why it's an issue:** If you ever decided to add handlers or middleware dynamically while the server is running, the program would crash due to concurrent map writes.

**Fix:** Protect `handlers` and `middleware` with the `WorkerPool` mutex.
