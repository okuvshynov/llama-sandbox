### 1. Nil Map Panic (Initialization Bug)
In Go, a map that is declared but not initialized is `nil`. While you can read from a nil map, **writing to a nil map causes a runtime panic.**

*   **The Bug:** `NewCounter` returns `&Counter{}`. This leaves the `counts` map as `nil`.
*   **The Result:** The first time anyone calls `/inc`, the line `c.counts[key]++` will cause the entire server to crash with a panic: `panic: assignment to entry in nil map`.
*   **The Fix:** Initialize the map using `make` in the constructor:
    ```go
    func NewCounter() *Counter {
        return &Counter{
            counts: make(map[string]int),
        }
    }
    ```

### 2. Concurrent Map Access (Data Race)
The `Get` method accesses the map without acquiring the mutex.

*   **The Bug:** `func (c *Counter) Get(key string) int { return c.counts[key] }` does not use `c.mu.Lock()`.
*   **The Result:** Even though `Get` isn't used in the current handlers, if it were, this would create a data race. If one goroutine calls `Increment` (write) while another calls `Get` (read), Go will trigger a fatal error: `fatal error: concurrent map read and map write`.
*   **The Fix:** Wrap the return in a lock/unlock block.

### 3. Map Reference Leak (Severe Data Race)
This is the most subtle and dangerous bug. In Go, maps are reference types.

*   **The Bug:** `GetAll()` locks the mutex, but it returns the **actual reference** to the internal map: `return c.counts`.
*   **The Result:** The mutex only protects the act of *returning* the map reference. Once `handleGetAll` receives the map, it passes it to `json.NewEncoder(w).Encode(data)`. The JSON encoder iterates over the map to serialize it. If `handleIncrement` modifies the map while the encoder is reading it, the server will crash with a fatal concurrent map access error.
*   **The Fix:** Create a copy of the map inside the lock and return the copy:
    ```go
    func (c *Counter) GetAll() map[string]int {
        c.mu.Lock()
        defer c.mu.Unlock()
        cp := make(map[string]int, len(c.counts))
        for k, v := range c.counts {
            cp[k] = v
        }
        return cp
    }
    ```

### 4. Missing Return after `http.Error` (Logic Bug)
In Go HTTP handlers, calling `http.Error` sends a response to the client, but it **does not** stop the execution of the function.

*   **The Bug:** In `handleIncrement`, when `key == ""`, `http.Error` is called, but there is no `return` statement following it.
*   **The Result:** The code will send a `400 Bad Request` response, but then it will immediately proceed to call `counter.Increment("")` and try to write the resulting number to the response body via `fmt.Fprintf`. This results in "superfluous response.WriteHeader" warnings in the logs and sends malformed responses to the client.
*   **The Fix:** Add a `return` statement:
    ```go
    if key == "" {
        http.Error(w, "missing key", 400)
        return // Stop execution here
    }
    ```
