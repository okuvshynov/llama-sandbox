### 1. Critical Security Vulnerabilities

#### A. SQL Injection (High Severity)
There are multiple locations where user input is concatenated directly into SQL strings instead of using parameterized queries. An attacker could drop tables, steal the entire database, or bypass authentication.
*   **Locations:** 
    *   `UserRepository.findByEmail`: `WHERE email = '" + email + "'"`
    *   `DocumentRepository.search`: `ILIKE '%${query}%'`
    *   `SearchService.getSuggestions`: `ILIKE '${prefix}%'`
    *   `AuditLogRepository.findByUser`: `BETWEEN '${startDate}' AND '${endDate}'`
*   **Fix:** Always use parameterized queries (e.g., `WHERE email = $1`).

#### B. Broken Password Hashing & Authentication (High Severity)
The authentication logic is fundamentally broken and insecure.
1.  **Algorithm:** It uses `sha256`, which is too fast and vulnerable to GPU-accelerated brute-force attacks.
2.  **Logic Bug:** In `register`, you create a salt and store a salted hash. However, in `login`, you hash the password **without the salt**. The login will always fail because the hashes will never match.
3.  **Fix:** Use `bcrypt` or `argon2`. These libraries handle salting automatically and are computationally expensive to prevent brute-forcing.

#### C. Privilege Escalation (Medium Severity)
In the `/api/auth/register` route, the `role` is taken directly from `req.body`. Any user can send `{ "role": "admin" }` during registration to gain full administrative access.
*   **Fix:** Hardcode the registration role to `"viewer"` and require an existing admin to promote users via a protected endpoint.

#### D. Sensitive Information Disclosure (Low Severity)
The `/api/health` endpoint returns the `dbHost` and `dbPort` when the database is down. This provides attackers with internal network topology.
*   **Fix:** Return a generic "Database unavailable" message.

### 2. Bugs & Reliability Issues

#### A. Unawaited Promises (Race Conditions)
Several database calls are made without `await`. This means the API response may be sent before the data is saved, or errors will become "unhandled promise rejections," potentially crashing the process.
*   **Locations:** `UserRepository.updateLastLogin` and `AuditLogRepository.create`.
*   **Fix:** Add `await` before `pool.query`.

#### B. Document Update Crash
In `DocumentRepository.update`, you iterate over `Object.entries(updates)`. If a client sends an extra field in the JSON body that doesn't exist as a column in the DB (e.g., `{ "foo": "bar" }`), the generated SQL will be invalid and the request will crash.
*   **Fix:** Use a whitelist of allowed fields to update.

#### C. Incorrect Password Verification Logic
As mentioned in security, the `login` method ignores the salt stored in the database.
*   **Fix:** Use `bcrypt.compare(password, passwordHash)`.

---

### 3. Performance Issues

#### A. The N+1 Query Problem (High Impact)
In `DocumentRepository.getDocumentsWithAuthors`, you fetch a list of documents and then perform a separate DB query for every single document to find its author. If a folder has 100 documents, you make 101 queries.
*   **Fix:** Use a SQL `JOIN`: 
    `SELECT d.*, u.username FROM documents d JOIN users u ON d.author_id = u.id WHERE d.folder_id = $1`.
