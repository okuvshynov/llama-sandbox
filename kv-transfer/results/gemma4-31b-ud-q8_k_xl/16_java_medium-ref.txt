### 1. Critical Bugs & Thread Safety

#### A. Concurrent Modification Exceptions
The methods `snapshot()` and `containsKey()` access the `store` (a `HashMap`) without acquiring the `lock`. 
- **Issue:** `HashMap` is not thread-safe. If one thread calls `snapshot()` while another thread is performing a `put()` or `evictExpired()`, the iterator will throw a `ConcurrentModificationException`. In some JVM versions, concurrent reads/writes to a `HashMap` can even lead to infinite loops during resizing.
- **Fix:** Wrap the logic in `snapshot()`, `containsKey()`, and `size()` within the `lock.lock()` block.

#### B. The "Thundering Herd" Problem (Cache Stampede)
In the `get(K key)` method, the lock is released **before** the `loader.apply(key)` is called.
- **Issue:** If 100 threads request the same missing key simultaneously, all 100 will bypass the cache and call the `loader` (which is often a slow DB call) at the same time. Then, all 100 will attempt to `put()` the result back into the cache.
- **Fix:** Use `ConcurrentHashMap.computeIfAbsent()` or a map of `CompletableFuture` to ensure the loader is only executed once per key.

#### C. Memory Leak in Metrics
The `CacheMetrics` class contains a `List<Long> responseTimes = new ArrayList<>()`.
- **Issue:** Every single request that records a response time adds an element to this list. This list grows indefinitely until the application runs out of memory (`OutOfMemoryError`).
- **Fix:** Use a rolling average (e.g., `average = average + (newValue - average) / count`) or a fixed-size window/histogram.

#### D. Thread Pool Leak in `warmUp`
The `warmUp` method creates a new `ExecutorService` every time it is called: `ExecutorService pool = Executors.newFixedThreadPool(...)`.
- **Issue:** This pool is never shut down. Each call to `warmUp` leaks several threads and memory.
- **Fix:** Use a `try-finally` block to call `pool.shutdown()` after the futures are processed.
