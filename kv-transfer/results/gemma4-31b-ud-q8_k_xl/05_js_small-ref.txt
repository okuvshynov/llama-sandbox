### 1. The `reschedule` Sorting Bug
**The Bug:** The `reschedule` method updates the `time` property of an event, but it does not re-sort the `this.events` array.

**Why it's a problem:** The `run()` method relies on the `this.events` array being sorted chronologically to calculate the `wait` time correctly. If you reschedule an event that was supposed to happen at 5000ms to happen at 100ms, it will still sit at the end of the array. The scheduler will process all other events first, and then execute the rescheduled event much later than intended.

**The Fix:** Call `this.events.sort((a, b) => a.time - b.time);` at the end of the `reschedule` method.

---

### 2. The "Stale Array" Bug in `run()`
**The Bug:** The `run()` method uses a `for...of` loop over `this.events`. In JavaScript, when you reassign an array (which happens in the `cancel` method via `.filter()`), the loop continues to iterate over the **original** array reference that existed when the loop started.

**Why it's a problem:** If you call `scheduler.cancel("backup")` while `scheduler.run()` is currently awaiting a timeout, the event will **still execute**. The loop is iterating over a snapshot of the array from the moment `run()` was called, ignoring any deletions made via `cancel`.

**The Fix:** Instead of a `for...of` loop, use a `while` loop that checks the length of the array and shifts the first element out, or check if the event still exists in the master list before executing it.

---

### 3. Unawaited Callbacks (Async Bug)
**The Bug:** The `run()` method calls `event.callback()` and then immediately moves to the listeners and the next event. It does not `await` the callback.

**Why it's a problem:** If a callback is an `async` function (which is common for "backups" or "reports"), the scheduler will trigger the "completed" listeners and move to the next event **before the callback has actually finished**. This defeats the purpose of a sequential scheduler.

**The Fix:** Change the call to `await event.callback();`.

---

### 4. Non-Unique Event Names
**The Bug:** The `cancel` and `reschedule` methods target events by `name`. 

**Why it's a problem:** The `schedule` method allows multiple events to have the same name. If you schedule three different "backup" events at different times, calling `cancel("backup")` will delete **all** of them, and `reschedule("backup", 100)` will move **all** of them to the same timestamp.

**The Fix:** Either prevent duplicate names in `schedule()` or, ideally, return a unique `eventId` from `schedule()` and use that ID for canceling and rescheduling.
