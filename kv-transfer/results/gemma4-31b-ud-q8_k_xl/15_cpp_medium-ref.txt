### 1. MemoryArena
**Issues:**
*   **Use-After-Free (Critical):** In `main()`, `arena.reset()` is called, which deletes the memory blocks. Immediately after, the code attempts to access `records[0]->tag`. This is undefined behavior and will likely crash.
*   **Memory Leak / Destructor Neglect (Critical):** `MemoryArena` provides a `create<T>` method that uses placement `new`. However, `reset()` and the destructor `~MemoryArena()` only delete the raw memory blocks. They **never call the destructors** of the objects created. If `T` is `std::string` or `std::vector`, the internal memory of those objects is leaked.
*   **Logic Error in `reset()`:** `blocks_ = { blocks_.back() };` keeps the *most recently allocated* block and deletes the others. Usually, an arena should keep the *first* block to maximize reuse of the initial allocation.

**Suggested Fixes:**
*   **Destructor Tracking:** To support non-trivial types, the arena needs to store a list of cleanup functions (e.g., `std::vector<std::function<void()>>`) that call the destructors of created objects in reverse order.
*   **Lifetime Management:** Ensure the user knows that `reset()` invalidates all pointers.

### 2. ThreadPool & TaskQueue
**Issues:**
*   **Potential Deadlock (Critical):** In `parallel_sort_impl`, the code calls `f1.get()` and `f2.get()`. This blocks the worker thread. If the pool is small and the recursion depth is high, all worker threads can end up blocking on futures that are still sitting in the queue, waiting for a free thread. This is a classic **ThreadPool Deadlock**.
*   **Wait-All Inefficiency (Performance):** `wait_all()` uses a "busy-wait" loop with `std::this_thread::yield()`. This consumes 100% CPU on one core while waiting for tasks to complete.
*   **Race Condition in `pending_tasks_` (Bug):** `pending_tasks_` is incremented *inside* the worker loop *after* the task is popped. If `submit` is called and `wait_all` is called immediately, `wait_all` might see `pending_tasks_ == 0` and `queues` as empty before the worker has had a chance to increment the counter.
