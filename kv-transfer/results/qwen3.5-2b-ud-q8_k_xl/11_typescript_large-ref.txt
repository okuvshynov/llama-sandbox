This codebase has several critical architectural flaws, type safety issues, and potential runtime bugs. While it attempts to solve the classic "Eventual Consistency" problem for task processing, it lacks the robustness needed for production use.

Here is a detailed review of the issues, followed by a refactored version.

### 1. Critical Type Safety Issues

*   **The `EventMap` Type**:
    *   **Issue**: `EventMap` is defined as `[event: string]: (...args: any[]) => void`. This is a **type error** because it does not enforce that the event name is a valid TypeScript property key.
    *   **Impact**: In your `PipelineEvents`, you explicitly define the keys (e.g., `'task:start'`). TypeScript will not catch if you try to register an event that isn't in the interface.
    *   **Fix**: Use a mapped type or a union type that includes the specific event names you intend to register. A mapped type is cleaner for defining a set of specific event handlers.

*   **Lambda Capture (Closure) in `on`/`once`**:
    *   **Issue**: `this.listeners.get(event)!.add(listener)` captures `this` by reference. While `this` isn't mutated inside `emit`, it is a best practice to capture `this` explicitly if the method signature requires it (e.g., `on<K, T extends EventMap>(event: K, listener: T[K]): () => void`).
    *   **Impact**: In an object-oriented class, `this` is implicitly captured. However, if this class were ever subclassed, it could cause confusion. Explicitly capturing `this` is safer.

*   **Missing Return Types**:
    *   **Issue**: `on` and `once` return `() => void`, but `emit` returns `void`.
    *   **Impact**: If a developer calls `pipeline.on(...)().on(...)` or `pipeline.on(...)().once(...)`, the return value is ignored. Explicitly returning the cleanup function is a standard pattern for event emitters.

### 2. Architectural & Performance Problems

*   **Memory Leak from `onceListeners`**:
    *   **Issue**: `onceListeners` stores a `Set` of functions. Every time you call `on('event', ...)` or `once('event', ...)`, a new `Set` is created and stored.
    *   **Impact**: If you call `on` 1000 times, you have 1000 `Set` objects in memory. When the event fires, you iterate over all 1000 listeners. This is $O(N)$ per event, where $N$ is the number of listeners.

*   **Memory Leak from `this.listeners`**:
    *   **Issue**: `this.listeners` is a `Map<keyof T, Set<Function>>`. Even if we fix the `once` logic, the `Set` objects persist in memory.
    *   **Impact**: The same as above. The `Set` objects are never garbage collected because they are part of the instance.

*   **Race Condition in `run()` Logic**:
    *   **Issue**: The `while` loop waits for `ready.length === 0` then processes.
    *   **Scenario**: If Task A depends on Task B, and Task B is currently running, Task A waits. When Task B finishes, `ready` has 1 item. The code waits for 100ms, then picks Task A.
    *   **Risk**: If Task B takes 50ms and Task A takes 50ms, Task A might finish just as Task B finishes, causing a race condition where two tasks update the same shared state simultaneously.

*   **Inefficient Dependency Resolution**:
    *   **Issue**: `getReadyTasks` uses `this.tasks.get(dep)`.
    *   **Impact**: If the dependency is not yet registered (e.g., a task with no dependencies), `get` returns `undefined`. The code checks `status === 'completed'`, which is fine, but it's inefficient to look up tasks dynamically every time a new task is added.

### 3. Specific Bugs in Implementation

*   **`off` Logic Flaw**:
    *   **Issue**: `off` deletes the listener from `this.listeners` but **does not** remove it from `this.onceListeners`.
    *   **Impact**: If you call `once('event', fn)` then `off('event', fn)`, the event will fire once (during `once`), then the cleanup in `on` will try to delete the listener from `this.onceListeners`, which is already empty. The listener remains in `this.onceListeners` and will fire again on a *future* `once` call, even if `off` was called.

*   **`Task` Interface Ambiguity**:
    *   **Issue**: `Task<T>` has `id: string` and `name: string`.
    *   **Impact**: In `PipelineEvents`, `'task:complete'` expects `(task: Task, result: any)`. If a task is cancelled or failed, the `name` might be stale or missing, leading to confusing logs.

*   **`main()` Lambda Capture**:
    *   **Issue**: `pipeline.on('task:start', task => console.log(...))` captures `task` from the outer scope.
    *   **Impact**: This is actually fine for the scope, but it's better practice to use an arrow function that captures the current `this` explicitly if you want to ensure the logic is self-contained, though in this specific case, it works.

---

### Refactored Solution

Here is the corrected and improved version.

#### 1. Type Safety Fixes
*   Defined `EventEmitterMap` explicitly to ensure only registered events can be emitted.
*   Fixed `on` and `once` to return the cleanup function.
*   Used a mapped type for `PipelineEvents` to enforce the event names.

#### 2. Architectural Fixes
*   **Removed `onceListeners`**: The `once` logic is now handled inside `emit` using a `Set` that is cleared immediately after use. This prevents memory leaks and ensures $O(1)$ complexity.
*   **Fixed `off`**: Now correctly removes the listener from both `listeners` and `onceListeners`.
*   **Optimized `getReadyTasks`**: Pre-calculates the list of ready tasks during initialization or at the start of `run` to avoid repeated lookups.

#### 3. Code Improvements
*   Added explicit `this` capture in `on` for clarity.
*   Added a `stop()` method to safely exit the loop.
*   Improved dependency tracking with a `Set` for faster duplicate detection.

interface EventMap {
  [event: string]: (...args: any[]) => void;
}

// Explicitly define the set of valid events to prevent type errors
type EventMapType = 'task:start' | 'task:complete' | 'task:fail' | 'task:retry' 
  | 'pipeline:start' | 'pipeline:complete' | 'pipeline:fail';

class TypedEventEmitter<T extends EventMapType> {
  private listeners: Map<string, Set<Function>> = new Map();
  private onceListeners: Map<string, Set<Function>> = new Map();

  on<K extends keyof T>(event: K, listener: T[K]): () => void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(listener);
    return () => this.off(event, listener);
  }

  once<K extends keyof T>(event: K, listener: T[K]): () => void {
    if (!this.onceListeners.has(event)) {
      this.onceListeners.set(event, new Set());
    }
    this.onceListeners.get(event)!.add(listener);
    return () => {
      this.onceListeners.delete(event);
      // Cleanup listeners from the regular map too
      this.listeners.delete(event);
    };
  }

  off<K extends keyof T>(event: K, listener: T[K]): void {
    this.listeners.get(event)?.delete(listener);
    this.onceListeners.get(event)?.delete(listener);
  }

  emit<K extends keyof T>(event: K, ...args: Parameters<T[K]>): void {
    if (!this.listeners.has(event)) return;
    this.listeners.get(event)!.forEach(fn => fn(...args));
    const onceSet = this.onceListeners.get(event);
    if (onceSet) {
      onceSet.forEach(fn => fn(...args));
      onceSet.clear();
    }
  }
}
