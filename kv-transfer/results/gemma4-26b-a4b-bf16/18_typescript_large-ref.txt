### 1. Critical Security Vulnerabilities

#### A. SQL Injection
**Problem:** Multiple methods use string concatenation or template literals to build SQL queries instead of parameterized queries.
*   `UserRepository.findByEmail`: `... WHERE email = '" + email + "'"`
*   `DocumentRepository.search`: `... ILIKE '%${query}%'`
*   `AuditLogRepository.findByUser`: `... BETWEEN '${startDate}' AND '${endDate}'`
*   `SearchService.getSuggestions`: `... ILIKE '${prefix}%'`

**Impact:** An attacker can bypass authentication, extract the entire database, or even gain shell access (depending on DB permissions).

**Fix:** Always use parameterized queries provided by `pg`.
```typescript
// Correct way
const result = await pool.query("SELECT * FROM users WHERE email = $1", [email]);
```

#### B. Broken Authentication & Weak Hashing
**Problem 1:** The `AuthService.register` method generates a `salt` but **does not save it** to the database.
**Problem 2:** The `AuthService.login` method attempts to verify the password by hashing it without the salt, meaning it will never match the stored hash.
**Problem 3:** `sha256` is a fast cryptographic hash, making it vulnerable to brute-force attacks.

**Impact:** Users cannot log in, and even if they could, the password storage is insecure.

**Fix:** Use `bcrypt` or `argon2`. These libraries handle salt generation and storage automatically within the hash string.
```typescript
import bcrypt from "bcrypt";

// In register
const passwordHash = await bcrypt.hash(password, 12);

// In login
const isMatch = await bcrypt.compare(password, user.passwordHash);
```

#### C. Information Leakage
**Problem:** 
1. `auth/register` returns `err.stack` in the response.
2. `health` check returns `dbHost` and `dbPort` in the error response.

**Impact:** Attackers can learn about your file structure, library versions, and internal network topology to craft specific exploits.

**Fix:** Never return stack traces or internal configuration to the client. Log them internally and return a generic error message.

#### D. Broken Access Control (IDOR)
**Problem:** The `downloadFile` and `deleteFile` methods in `FileService` do not verify if the `userId` has permission to access the specific `documentId` associated with the attachment.

**Impact:** Any logged-in user can download any file in the system simply by guessing the `attachmentId`.

**Fix:** In the service layer, check if the user has `read` access to the parent document before serving the file.

### 2. Performance Issues

#### A. The $N+1$ Query Problem
**Problem:** 
1. `DocumentRepository.getDocumentsWithAuthors`: It fetches $N$ documents, then loops through them to run a separate SQL query for every single author.
2. `FolderRepository.getFolderTree`: This is a recursive function that performs a database query for every single child node in the tree.

**Impact:** As the number of documents or folders grows, the API response time will increase exponentially, eventually crashing the database.

**Fix:** 
1. Use a `JOIN` in SQL to fetch documents and authors in a single query.
2. For folders, use a Common Table Expression (CTE) to fetch the entire tree in one query.

#### B. In-Memory Scaling Issues (Rate Limiter & Search Cache)
**Problem:** `rateLimiter` and `SearchService.searchCache` use local `Map` objects.

**Impact:** If you deploy this app in a containerized environment (like Kubernetes) with multiple instances, the rate limit and cache will not be shared. A user could hit 100 requests per instance, effectively bypassing your limit.

**Fix:** Use an external store like **Redis** for distributed caching and rate limiting.
