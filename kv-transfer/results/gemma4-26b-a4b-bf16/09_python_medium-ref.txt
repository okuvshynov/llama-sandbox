### 1. Critical Bugs & Thread Safety

#### A. Potential Deadlocks (Manual Lock Management)
**Problem:** The code uses `self.lock.acquire()` and `self.lock.release()` manually. If an exception occurs between these two calls (e.g., `json.dumps` fails because an object isn't serializable, or a database error occurs), `release()` is never called.
**Impact:** The entire application will hang indefinitely on the next attempt to access the cache.
**Fix:** Always use the lock as a context manager: `with self.lock:`.

#### B. SQLite Threading Violation
**Problem:** `sqlite3.connect()` creates a connection that, by default, cannot be shared across threads. The `PeriodicCleaner` runs in a separate thread and calls `cleanup_expired`, which calls `_evict`, which calls `_log_operation`, which uses `self.db`.
**Impact:** This will raise `sqlite3.ProgrammingError: SQLite objects created in a thread can only be used in that same thread.`
**Fix:** Either create a new connection per thread, use a thread-safe queue to send logs to a dedicated logging thread, or initialize the connection with `check_same_thread=False` (though the latter requires external synchronization).

---

### 2. Performance Issues (Complexity & Latency)

#### A. $O(N)$ LRU Implementation
**Problem:** `LRUEvictionPolicy` uses a standard Python `list` for `access_order`. The `record_access` and `remove` methods call `self.access_order.remove(key)`.
**Impact:** `list.remove()` is an $O(N)$ operation. As the cache grows to thousands of items, every `get` and `put` becomes increasingly slow.
**Fix:** Use `collections.OrderedDict`. It is designed specifically for this; moving an item to the end (marking it as most recent) and deleting items is $O(1)$.

#### B. $O(N)$ TTL Expiry Scanning
**Problem:** `TTLEvictionPolicy.get_expired_keys` iterates through every single entry in the dictionary to check if it has expired.
**Impact:** `cleanup_expired` becomes a "stop-the-world" event that consumes massive CPU as the cache size increases.
**Fix:** Use a **Min-Heap** (`heapq` module) to store `(expiry_timestamp, key)` tuples. This allows you to peek at the earliest expiration in $O(1)$ and only process the keys that have actually expired.

#### C. The "I/O inside Lock" Bottleneck
**Problem:** `_log_operation` performs a synchronous disk write (`self.db.commit()`) while holding `self.lock`.
**Impact:** This is the single biggest performance killer. The cache speed is no longer limited by RAM, but by the latency of your hard drive. All other threads will block waiting for the disk I/O to complete.
**Fix:** Use an asynchronous logging pattern. Push log events to a `queue.Queue` and have a background thread write them to SQLite in batches.

#### D. Inefficient Size Computation
**Problem:** `_compute_size` uses `len(json.dumps(value))`.
**Impact:** This is extremely slow for large or complex objects because it must traverse and serialize the entire object just to estimate its size. It also crashes if the object is not JSON-serializable.
**Fix:** Use `sys.getsizeof(value)` for a rough estimate, or better
