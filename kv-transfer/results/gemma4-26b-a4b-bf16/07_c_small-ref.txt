### 1. Memory Leak in `table_put` (Update Case)
When updating an existing key, the program overwrites the existing `value` pointer without freeing the memory it previously pointed to.

```c
if (strcmp(t->entries[i].key, key) == 0) {
    t->entries[i].value = strdup(value); // BUG: Old value is leaked!
    return;
}
```
**Fix:** You must `free(t->entries[i].value)` before assigning the new `strdup` result.

### 2. Memory Leak in `table_delete` (The "Swap" Bug)
The program uses the efficient "swap with last element" method to delete an entry. However, it copies the pointers from the last element into the current slot but never frees the memory associated with the entry being deleted.

```c
if (strcmp(t->entries[i].key, key) == 0) {
    t->entries[i] = t->entries[t->size - 1]; // BUG: The strings at entries[i] are leaked!
    t->size--;
    return;
}
```
**Fix:** Before performing the swap, you must free the `key` and `value` of the element at `t->entries[i]`.

### 3. Unsafe `realloc` Usage in `table_put`
The code assigns the result of `realloc` directly back to the original pointer. If `realloc` fails, it returns `NULL`. By assigning `NULL` to `t->entries`, you lose the address of the original memory block, making it impossible to free, resulting in a memory leak and a subsequent crash.

```c

```
**Fix:** Always use a temporary pointer:
