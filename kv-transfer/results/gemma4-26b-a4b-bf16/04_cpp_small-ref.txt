This program implements **Huffman Coding**, a famous lossless data compression algorithm. The goal of Huffman coding is to assign variable-length binary codes to characters based on their frequency: characters that appear more often are assigned shorter codes, while characters that appear less often are assigned longer codes. This reduces the overall number of bits needed to represent the text.

#### Step-by-Step Logic:

1.  **Frequency Counting (`buildTree` part 1):**
    The program first iterates through the input string and uses an `std::unordered_map` to count how many times each character appears.

2.  **Building the Huffman Tree (`buildTree` part 2):**
    *   It creates a "leaf node" for each unique character and pushes them into a **min-priority queue** (a min-heap). The priority is determined by the character's frequency.
    *   It repeatedly extracts the two nodes with the lowest frequencies from the queue.
    *   It creates a new internal node with a frequency equal to the sum of the two extracted nodes. This new node becomes the parent of the two.
    *   This process continues until only one node remains in the queue: the **Root** of the Huffman Tree.

3.  **Generating Binary Codes (`buildCodes`):**
    The program performs a depth-first traversal (DFS) of the tree.
    *   Every time it moves **left**, it appends a `'0'` to the current path.
    *   Every time it moves **right**, it appends a `'1'` to the current path.
    *   When it reaches a leaf node (a node with no children), the accumulated string of 0s and 1s is stored as the unique code for that character.

4.  **Encoding (`encode`):**
    The program iterates through the original text character by character, looks up the generated binary code in the map, and concatenates them into one long string of bits.

5.  **Output and Comparison:**
    Finally, it prints the codebook (sorted by code length for readability), calculates the size of the original text (assuming standard 8-bit ASCII), calculates the size of the encoded version, and displays the compression ratio.

### Complexity Analysis

Let **$N$** be the total number of characters in the input string and **$K$** be the number of unique characters in the input string.

#### 1. Time Complexity

| Phase | Operation | Complexity | Explanation |
| :--- | :--- | :--- | :--- |
| **Frequency Count** | Iterating through `text` | $O(N)$ | We visit every character in the input string once. |
| **Heap Initialization** | Pushing $K$ nodes into PQ | $O(K \log K)$ | Each insertion into a priority queue takes $O(\log K)$. |
| **Tree Construction** | While loop (PQ operations) | $O(K \log K)$ | There are $K-1$ iterations. Each iteration involves two `pop` and one `push` operation, each taking $O(\log K)$. |
| **Code Generation** | DFS Traversal | $O(K)$ | A Huffman tree with $K$ leaves has exactly $2K-1$ nodes. We visit each node once. |
| **Encoding** | Mapping `text` to bits | $O(N)$ | We iterate through the original text and perform an $O(1)$ average-case lookup in the hash map. |
| **Sorting/Printing** | Sorting codebook | $O(K \log K)$ | Sorting the unique character codes for display. |

**Total Time Complexity:** $O(N + K \log K)$

*Note: In most practical scenarios, $K$ is limited by the character set (e.g., $K \le 256$ for extended ASCII). If $K$ is considered a constant, the complexity is effectively $O(N)$.*

#### 2. Space Complexity

| Component | Space Complexity | Explanation |
| :--- | :--- | :--- |
| **Frequency Map** | $O(K)$ | Stores one entry per unique character. |
| **Huffman Tree** | $O(K)$ | The tree contains $2K-1$ nodes. |
| **Code Map** | $O(K)$ | Stores the binary string for each unique character. |
| **Encoded String** | $O(N)$ | In the worst case (though highly unlikely with Huffman), the bitstring could be proportional to $N$. |

**Total Space Complexity:** $O(N + K)$
