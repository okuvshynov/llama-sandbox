### 1. Critical Bugs (Correctness & Data Integrity)

#### A. Floating-Point Errors in Financial Calculations
*   **File:** `models.py`
*   **Issue:** The code uses `float` for `price`, `subtotal`, `tax`, and `total`. Floating-point arithmetic (IEEE 754) cannot precisely represent many decimal fractions (e.g., `0.1 + 0.2 != 0.3`). In a microservice handling money, this will lead to rounding errors that accumulate over time, causing balance mismatches.
*   **Fix:** Use the `decimal.Decimal` class for all currency-related fields and calculations.

#### B. Broken Reservation Cleanup Logic
*   **File:** `inventory.py` (`cleanup_expired_reservations`)
*   **Issue:** The method currently iterates through all reservations and deletes **every single one** without checking if they are actually expired.
    ```python
    for order_id in self.reservations:
        expired.append(order_id) # This adds EVERYTHING
    ```
*   **Fix:** Check the timestamp of the reservation against `now - self.reservation_timeout` before adding it to the `expired` list.

#### C. Data Loss in Persistence
*   **File:** `inventory.py` (`_save_to_disk`)
*   **Issue:** The `_save_to_disk` method manually constructs a dictionary but only saves a subset of the `Product` attributes (`id`, `name`, `price`, `stock_count`, `category`). It **ignores** `description`, `weight_kg`, `tags`, `discount_percent`, `is_active`, etc. Every time the service restarts, all product metadata is wiped.
*   **Fix:** Use a proper serialization method (like `asdict` from `dataclasses`) or, preferably, use a database.

#### D. Lack of Atomicity (The "Distributed Transaction" Problem)
*   **File:** `order_processor.py` (`process_next`)
*   **Issue:** The sequence is: 1. Charge Payment $\rightarrow$ 2. Commit Inventory $\rightarrow$ 3. Update Status. If the service crashes or the database fails *after* step 1 but *before* step 2, the customer has been charged, but the inventory is never decremented and the order is lost.
*   **Fix:** Implement the **Saga Pattern**. If the inventory commit fails, you must issue a compensating transaction (an automatic refund) via the `payment.refund` method.

### 2. Security Vulnerabilities

#### A. Predictable Tracking Numbers
*   **File:** `order_processor.py` (`ship_order`)
*   **Issue:** The tracking number is generated using `hashlib.md5(f"{order_id}{time.time()}".encode())`. MD5 is cryptographically broken and highly susceptible to collision attacks. Furthermore, because `order_id` and `time.time()` are somewhat predictable, an attacker could potentially guess tracking numbers to scrape customer shipping info.
*   **Fix:** Use `secrets.token_hex(6)` or a UUID for non-predictable identifiers.

#### B. In-Memory Sensitive Data
*   **File:** `order_processor.py` (`PaymentGateway`)
*   **Issue:** Transactions are stored in a dictionary `self.transactions` in memory. This is a massive security and reliability risk. If the process restarts, all record of who was charged disappears, making audits and refunds impossible.
*   **Fix:** Store transaction logs in a persistent, encrypted database.
