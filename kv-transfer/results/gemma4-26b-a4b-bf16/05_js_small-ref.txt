### 1. The `reschedule` Sorting Bug
**The Problem:** The `schedule` method keeps the `events` array sorted by time. However, the `reschedule` method modifies an event's `time` without re-sorting the array.

**The Consequence:** Because the `run()` method iterates through the array linearly using a `for...of` loop, the order of execution depends entirely on the array index. If you reschedule an event from 5000ms to 100ms, it will still be located at the end of the array. The scheduler will wait for all other events to finish before it even reaches the rescheduled event, making the "new time" incorrect.

**The Fix:** Call `this.events.sort((a, b) => a.time - b.time);` at the end of the `reschedule` method.

---

### 2. The "Fire and Forget" Async Bug
**The Problem:** In the `run()` method, the scheduler `await`s the `setTimeout` (the wait period), but it **does not** `await` the `event.callback()` or the `handler(event)`.

**The Consequence:** If a callback is an `async` function or performs a long-running task, the scheduler will immediately move to the next event in the loop without waiting for the previous one to finish. This can lead to race conditions where multiple events are running simultaneously when the user might have expected them to run sequentially.

**The Fix:** Change the calls to `await event.callback()` and `await handler(event)` (and ensure the `run` loop treats them as such).

---
