### 1. Critical Bugs & Logic Errors

#### A. The `once()` Listener Wipeout (Logical Bug)
In `TypedEventEmitter`, the `once` implementation is broken. 
```typescript
// Current implementation
onceSet.forEach(fn => fn(...args));
onceSet.clear();
```
**The Issue:** If you register three different `once` listeners for the same event, calling `emit` once will execute all of them and then `clear()` the entire set. This means if `once` is used for different logic paths, one trigger will accidentally satisfy (and delete) all other pending "once" listeners.
**Improvement:** The standard pattern is to wrap the listener in a function that calls `off()` immediately before executing the original function.

#### B. Batch-based vs. Sliding-window Concurrency (Performance Bug)
In `Pipeline.run()`, you use `await Promise.all(batch.map(...))`.
**The Issue:** This is not true concurrency control; it is **batching**. If your concurrency is 4, and you start 4 tasks, and 3 finish in 1 second but 1 takes 60 seconds, the entire pipeline stalls for 59 seconds. The other 3 "slots" in your concurrency limit sit idle.
**Improvement:** Use a worker pool pattern or a "sliding window" where as soon as *any* task finishes, the next ready task is pulled from the queue.

#### C. "Soft" Cancellation (Functional Bug)
The `cancel()` method only changes a string property: `task.status = 'cancelled'`.
**The Issue:** This does **not** stop the actual `task.execute()` promise. If a task is performing a heavy network request or a long loop, it will continue to consume CPU/Memory/Network in the background despite being "cancelled."
**Improvement:** Pass an `AbortSignal` (from `AbortController`) into the `execute` function so the task can internally stop its work.
