Identify bugs, race conditions, and design issues. Explain each and provide fixes.

    *   *Structures:* `Job`, `WorkerPool`, `RateLimiter`.

    *   **Race Condition #1: Mutating `Job` fields.**
        *   In `Submit`, `job.Status` and `job.CreatedAt` are set.
        *   In `processJob`, `job.Status`, `job.StartedAt`, `job.Result`, `job.Retries`, etc., are set.
        *   In `GetJob`, the pointer to the `job` is returned.
        *   In `main` (`/status` handler), the pointer is encoded to JSON.
        *   *Problem:* The worker goroutine updates the `job` fields while the HTTP handler goroutine reads them. There is no mutex guarding the individual `Job` struct fields. This is a classic data race.

    *   **Race Condition #2: Map access in `processJob`?**
        *   Wait, `processJob` doesn't touch `wp.results`. It only modifies the `job` object itself.
        *   Wait, look at `Submit`: `wp.results[job.ID] = job`.
        *   Look at `stats` handler: Iterates over `pool.results`.
        *   Is `wp.results` protected? Yes, `wp.mu` is used in `Submit`, `GetJob`, and `/stats`.
        *   *But*, even if the map is protected, the *values* inside the map (the `Job` structs) are being mutated by workers without synchronization.

    *   **Race Condition #3: `Stop` vs `Submit`.**
        *   `Stop` closes `jobQueue`.
        *   If `Submit` is called after `Stop`, sending to a closed channel causes a panic.

    *   **Bug #1: Retry Logic & Channel Blocking.**
        *   In `processJob`, when a retry happens: `wp.jobQueue <- job`.
        *   This happens inside the worker loop. If the `jobQueue` is full, the worker will block. If all workers block on retrying, the whole system deadlocks.
        *   Also, `time.Sleep` inside the worker blocks that worker from doing anything else. While acceptable for a simple demo, in a production system, you'd want a non-blocking delay (e.g., a timer or a separate delayed queue).

    *   **Bug #2: The `Stop` method pattern.**
        *   `wp.cancel()` stops the workers.
        *   `close(wp.jobQueue)` is called.
        *   `wp.wg.Wait()` is called.

        *   *Actually*, in Go, reading from a closed channel returns the zero value. `job` will be `nil`. `wp.processJob(id, nil)` will panic.

    *   **Design Issue #1: Lack of Thread-Safety in `Job`.**
        *   The `Job` struct needs its own mutex if it's being read by HTTP handlers while being written by workers.

### 1. Critical Bugs & Race Conditions

#### A. Data Race on `Job` Fields (High Severity)
**The Problem:** The `Job` struct is shared between the worker goroutines (which write to `Status`, `Result`, etc.) and the HTTP handler goroutines (which read from the job via `GetJob` or `/stats`). There is no synchronization (mutex) inside the `Job` struct itself.
**The Consequence:** Running this with `go run -race` will trigger warnings. In production, this leads to undefined behavior, memory corruption, or crashes when the JSON encoder reads a string while a worker is modifying it.
**The Fix:** Add a `sync.RWMutex` to the `Job` struct and use it whenever reading or writing any field in `Job`.

#### B. Panic on Channel Closure (High Severity)
**The Problem:** In the `worker` function, the loop reads from `wp.jobQueue`. When `Stop()` is called, `close(wp.jobQueue)` is executed. In Go, reading from a closed channel returns the zero value of the channel's type. Since the channel holds `*Job`, it returns `nil`.
**The Consequence:** The worker will receive `nil`, pass it to `processJob(id, nil)`, and the line `log.Printf(..., job.ID)` will cause a **nil pointer dereference panic**, crashing the entire application.
**The Fix:** Check if the received job is `nil` inside the worker loop.
