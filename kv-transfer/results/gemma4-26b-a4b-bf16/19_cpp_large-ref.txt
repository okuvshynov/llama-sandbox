C++ codebase featuring a `SlabAllocator`, `HazardPointerDomain`, `LockFreeHashMap`, `LockFreeStack`, `RingBuffer`, and a benchmarking suite.

    *   **Module 1: `SlabAllocator`**
        *   *Design:* Uses a mutex for `allocate` and `deallocate`. This makes it a "synchronized allocator," not a lock-free one, though it manages chunks.
        *   *Bug/Issue (Alignment):* `std::aligned_alloc` takes `alignment` and `size`. The code uses `CACHE_LINE_SIZE` and `alloc_size`. However, `alloc_size` might not be a multiple of `CACHE_LINE_SIZE`, which is often required by `aligned_alloc` on some platforms (like POSIX/C11).
        *   *Bug (Memory Safety):* `grow` uses `std::aligned_alloc`. `~SlabAllocator` uses `std::free`. This is correct for `aligned_alloc` in C11/C++17.
        *   *Performance:* The mutex makes it a bottleneck in multi-threaded scenarios.
        *   *Bug (Alignment/Sizing):* `object_size_` is calculated as `max(object_size, sizeof(SlabBlock))`. But it doesn't ensure `object_size_` is a multiple of the alignment required by the items being stored. If the user wants to store 8-byte objects but the allocator doesn't align to 8, it might crash.
        *   *Memory Safety:* `grow` might throw `std::bad_alloc`. The destructor handles chunks. This looks okay.

    *   **Module 2: `HazardPointerDomain`**
        *   *Design:* Classical Hazard Pointer (HP) pattern.
        *   *Bug (Memory Safety - ABA/Race):* In `acquire_for_current_thread`, the thread searches for an empty `owner`. It uses `compare_exchange_strong` on the owner. This is okay.
        *   *Bug (Memory Safety - `scan`):* The `scan` function sorts `hazard_ptrs`. This is good for `binary_search`.
        *   *Bug (Memory Safety - `scan` algorithm):* The `while (it != retired.end())` loop. It uses `std::move(retired.back())` and `pop_back()`. This is a standard pattern for $O(1)$ removal from a vector, but there's a catch: if you swap the current element with the back and then `pop_back`, you *must not* increment the iterator `it` if you just moved a new (unprocessed) element into the current position. The code *does* correctly handle this by only incrementing `it` in the `else` branch.
        *   *Concurrency/Memory Order:* `head_.load(std::memory_order_relaxed)` in `scan`. This is dangerous. `scan` needs to see the most recent hazard pointers to avoid a Use-After-Free (UAF). It should at least be `acquire`.
        *   *Bug (Memory Safety - Leak):* `acquire_for_current_thread` uses `new HazardPointerRecord()`. There is no mechanism to free these records if a thread exits (the `owner` becomes an old thread ID). This is a permanent leak for long-running systems.
        *   *Bug (Memory Safety - `retire`):* `retired` is `thread_local`. This is fine.

    *   **Module 3: `LockFreeHashMap`**
        *   *Serious Bug (Concurrency/Race):* `resize()` is catastrophic. It's a single-threaded resize called inside `insert`. If multiple threads call `insert` and trigger `resize`, they all try to `new` a new table and `delete[] table_`. This is a massive race condition. Even with one thread resizing, while `delete[] table_` happens, other threads in `find` or `insert` are accessing the old `table_`. This is a **Use-After-Free** waiting to happen.
        *   *Design:* Linear Probing.
        *   *Bug (Memory Safety):* `insert` uses `pool_.allocate()`. If `insert` updates a value, it calls `retire`. The deleter calls `pool_.deallocate(p)`. This is the intended use of HP.
        *   *Bug (Concurrency/Logic):* `find` uses `HazardPointerGuard guard(hp_rec, 0);`. It only protects `index 0`. If the thread needs to protect multiple pointers (though here it only needs one), it's limited.
        *   *Bug (Concurrency/Logic):* The `insert` function:
            1. `table_[probe].key.compare_exchange_strong(...)`
            2. `pool_.allocate()`
            3. `table_[probe].value.store(...)`
            There is a window between step 1 and 3 where `key` is set but `value` is `nullptr`. A `find` call seeing the `key` will see a `nullptr` value and return `std::nullopt`, even though the key exists.
        *   *Bug (Concurrency/Logic):* `erase` uses `compare_exchange_strong` on the key. Then it sets `value` to `nullptr`. This is generally okay, but if a `find` was mid-execution, it might have already grabbed the `value` pointer before it was cleared.
        *   *Bug (Concurrency/Logic):* `resize` doesn't use Hazard Pointers to protect the `table_` pointer itself.
        *   *Design:* The `EMPTY_KEY` and `TOMBSTONE_KEY` are hardcoded. If the user provides these keys, the map breaks.

    *   **Module 4: `LockFreeStack`**
        *   *Serious Bug (Memory Safety):* The classic ABA problem. `pop()` uses `compare_exchange_weak(old_head, old_head->next, ...)`. If `old_head` is popped, deleted, and a new node is allocated at the same address, `old_head->next` might point to something completely different. This is a classic UAF/corruption bug. It needs Hazard Pointers or a similar mechanism.

    *   **Module 5: `RingBuffer`**
        *   *Bug (Concurrency):* This is a Single-Producer Single-Consumer (SPSC) design, but it's used in a Multi-Producer context (in `BenchmarkWorker`). In a Multi-Producer context, `head_.load` and `head_.store` will race. `next_head` could be calculated by two threads, both see the same `head`, and both write to the same `buffer_[head]`. It is **not** thread-safe for multiple producers.

    *   **Module 6: `Benchmark` logic**
        *   `BenchmarkWorker` uses `log_.push(...)`. Since `RingBuffer` is not thread-safe for multiple producers, the logging will corrupt the buffer or lose data.
