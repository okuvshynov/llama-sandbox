### 1. Nil Map Panic (Initialization Bug)
**The Bug:** In `NewCounter`, the `counts` map is never initialized.
```go
func NewCounter() *Counter {
	return &Counter{} // counts is nil here
}
```
**Explanation:** In Go, a map declared as part of a struct defaults to `nil`. While you can read from a nil map, **writing to a nil map causes an immediate runtime panic.** As soon as the first request calls `Increment`, the server will crash.

**The Fix:** Initialize the map using `make` in the constructor.
```go
func NewCounter() *Counter {
	return &Counter{
		counts: make(map[string]int),
	}
}
```

### 2. Data Race in `Get` (Concurrency Bug)
**The Bug:** The `Get` method accesses the map without acquiring the mutex.
```go
func (c *Counter) Get(key string) int {
	return c.counts[key] // Reading without a lock
}
```
**Explanation:** Go maps are not thread-safe. If one goroutine is writing to the map (via `Increment`) while another goroutine is reading from it (via `Get`), the program will crash with a `fatal error: concurrent map read and map write`. Even if it doesn't crash immediately, you may read stale or corrupted data.

**The Fix:** Acquire the lock before reading.
```go
func (c *Counter) Get(key string) int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts[key]
}
```

### 3. Map Reference Leak & Race in `GetAll` (Concurrency Bug)
**The Bug:** `GetAll` returns the actual map reference instead of a copy, and the lock is released before the data is used.
```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts // Returns the pointer to the internal map
}
```
**Explanation:** This is a two-part bug. 
1. **Race Condition:** The `GetAll` function releases the lock as soon as it returns the map. However, the `handleGetAll` function then passes that map to `json.NewEncoder(w).Encode(data)`. The JSON encoder iterates over the map *after* the lock is gone. If an `Increment` request happens while the JSON encoder is iterating, the server will panic.
2. **Encapsulation Violation:** Because you returned the actual map, any other part of the code could modify the internal state of the `Counter` without using the mutex.

**The Fix:** Create a copy of the map while holding the lock.
```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
	
	copy := make(map[string]int)
	for k, v := range c.counts {
		copy[k] = v
	}
	return copy
}
```
