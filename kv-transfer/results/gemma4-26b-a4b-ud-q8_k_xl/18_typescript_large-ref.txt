### 1. Security Vulnerabilities (Critical)

#### A. SQL Injection
There are multiple instances where user input is concatenated directly into SQL strings instead of using parameterized queries.
*   **`UserRepository.findByEmail`**: `WHERE email = '" + email + "'"`
*   **`DocumentRepository.search`**: `ILIKE '%${query}%'`
*   **`AuditLogRepository.findByUser`**: `BETWEEN '${startDate}' AND '${endDate}'`
*   **`SearchService.getSuggestions`**: `ILIKE '${prefix}%'`
*   **`DocumentRepository.update`**: While it uses parameters for values, the keys are taken directly from `Object.entries(updates)`. If a user sends a request with a key like `id = 1 --`, they can manipulate the query.

**Fix:** Always use parameterized queries (e.g., `WHERE email = $1`) provided by the `pg` library for both values and identifiers (though identifiers should be white-listed).

#### B. Broken Access Control (IDOR)
Many routes fetch resources based solely on an ID provided in the URL without checking if the authenticated user has permission to view that specific resource.
*   **`GET /api/documents/:id`**: Any logged-in user can view any document by guessing the ID.
*   **`GET /api/attachments/:id/download`**: Any logged-in user can download any file.
*   **`GET /api/folders/:id/documents`**: No check if the user belongs to that folder.

**Fix:** In the Service layer, before returning a resource, verify that `user.id` has `read` access to the parent folder or is the `authorId`.

#### C. Weak Password Hashing
The `AuthService` uses `sha256` with a manual salt. `sha256` is a fast cryptographic hash, making it vulnerable to high-speed brute-force attacks using GPUs.
**Fix:** Use a dedicated, slow password-hashing algorithm like **Argon2** or **bcrypt**.

#### D. Information Disclosure
*   **`GET /api/health`**: On failure, it returns `dbHost` and `dbPort`. This gives attackers architectural details.
*   **`POST /api/auth/register`**: The error handler explicitly returns `stack: (err as Error).stack` in the catch block. This can leak sensitive file paths and logic.

**Fix:** Never return stack traces or infrastructure details in API responses. Use a generic "Internal Server Error" and log the details internally.

### 2. Performance Issues

#### A. The N+1 Query Problem
*   **`DocumentRepository.getDocumentsWithAuthors`**: It fetches a list of documents, then loops through them and executes a separate database query for every single author. If there are 100 documents, it performs 101 queries.
*   **`DocumentService.bulkUpdateTags`**: It loops through IDs and performs a query for every single ID.
*   **`POST /api/documents/export`**: It loops through `documentIds` and queries the DB one by one.

**Fix:** Use SQL `JOIN`s to fetch authors in a single query, or use the `WHERE id IN (...)` syntax to fetch multiple records at once.

#### B. Recursive Database Queries
*   **`FolderRepository.getFolderTree`**: This uses recursion to build a tree. For a deep folder structure, this will trigger an explosion of database queries.

**Fix:** Use a **Common Table Expression (CTE)** with `WITH RECURSIVE` in PostgreSQL to fetch the entire tree in a single query.

#### C. Memory Leaks
*   **`NotificationService`**: The `listeners` Map stores `Response` objects. When a client disconnects, the entry is never removed from the Map. Over time, the server will run out of memory.
*   **`rateLimiter`**: The `requests` Map stores IP addresses and timestamps indefinitely. It never purges old IPs, leading to a memory leak.

**Fix:** 
*   **SSE**: Use `req.on('close', ...)` to remove the listener from the Map.
*   **Rate Limiter**: Use a sliding window with a TTL or an external store like **Redis**.
