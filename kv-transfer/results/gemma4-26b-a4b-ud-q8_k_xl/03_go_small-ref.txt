### 1. Nil Map Panic (Initialization Bug)
**The Bug:** In `NewCounter()`, you initialize the `Counter` struct, but you never initialize the `counts` map. In Go, the zero-value of a map is `nil`. While you can read from a nil map, **writing to a nil map causes an immediate runtime panic.**

**The Location:**
```go
func NewCounter() *Counter {
	return &Counter{} // counts is nil here
}

func (c *Counter) Increment(key string) int {
	c.mu.Lock()
	c.counts[key]++ // PANIC: assignment to entry in nil map
	// ...
}
```

**The Fix:** Initialize the map using `make`.
```go
func NewCounter() *Counter {
	return &Counter{
		counts: make(map[string]int),
	}
}
```

---

### 2. Missing `return` after Error (Logic Bug)
**The Bug:** In `handleIncrement`, when the `key` is missing, you call `http.Error`. However, `http.Error` does not stop the execution of the function; it only writes the error to the response body. The code will continue to execute and attempt to call `counter.Increment("")`.

**The Location:**
```go
if key == "" {
    http.Error(w, "missing key", 400)
    // Execution continues!
}
val := counter.Increment(key) // Will increment the empty string key
```

**The Fix:** Add a `return` statement after the error call.
```go
if key == "" {
    http.Error(w, "missing key", 400)
    return // Stop execution here
}
```

### 3. Unprotected Read (Race Condition)
**The Bug:** The `Get` method accesses the `counts` map without acquiring the mutex. In Go, maps are not thread-safe. If one goroutine is writing to the map (via `Increment`) while another goroutine is reading from it (via `Get`), the program will crash with a `fatal error: concurrent map read and map write`.

**The Location:**
```go
func (c *Counter) Get(key string) int {
	return c.counts[key] // No mutex lock!
}
```

**The Fix:** Wrap the access in a mutex lock.
```go
func (c *Counter) Get(key string) int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts[key]
}
```

---

### 4. Leaking Internal State (Concurrency/Data Race Bug)
**The Bug:** The `GetAll` method locks the mutex, but it returns the **actual reference** to the internal `counts` map. Because maps in Go are reference types, the caller (`handleGetAll`) now holds a pointer to the same map used inside the `Counter` struct. 

When `json.NewEncoder(w).Encode(data)` runs, it iterates over the map. If a user calls `/inc` at the exact same time, the `Increment` method will attempt to write to the map while the JSON encoder is reading it, causing a fatal crash.

**The Location:**
```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.counts // Returns the reference to the live map
}
```

**The Fix:** Create a copy of the map while holding the lock, then return the copy. This ensures the internal map is never accessed outside the protection of the mutex.
```go
func (c *Counter) GetAll() map[string]int {
	c.mu.Lock()
	defer c.mu.Unlock()
    
	// Create a snapshot copy
	copy := make(map[string]int)
	for k, v := range c.counts {
		copy[k] = v
	}
	return copy
}
```
