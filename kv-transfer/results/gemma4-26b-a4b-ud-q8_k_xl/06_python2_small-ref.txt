    *   *Topological Sort:* The process of ordering nodes in a directed graph such that for every directed edge $u \to v$, $u$ comes before $v$.
    *   *Kahn's Algorithm (for `solve`):*
        1.  Calculate in-degrees.
        2.  Put all nodes with in-degree 0 in a queue.
        3.  While queue is not empty:
            a. Pop $u$.
            b. Add to result.
            c. For each neighbor $v$, decrement in-degree.
            d. If in-degree of $v$ is 0, add to queue.
        4.  If result length matches node count, return result; else, there is a cycle.

        2.  Try each available node one by one.
        3.  For each choice, update the state (subtract in-degrees) and move to the next level.
        4.  Undo the choice (backtrack) to try the next available node at that same level.

This Python program is a **Dependency Resolver**. It takes a set of tasks and their dependencies (represented as directed edges) and determines the order in which those tasks must be performed so that no task is started before its prerequisites are met.

In computer science terms, this is a **Topological Sort** of a Directed Acyclic Graph (DAG).

---

### Function Breakdown

#### 1. `build_graph(edges)`
This is a preprocessing helper function. It converts the list of edge tuples into three data structures:
*   **`graph`**: An adjacency list (e.g., `{'compile': ['link', 'test']}`) used to quickly find which tasks depend on a completed task.
*   **`in_degree`**: A dictionary counting how many incoming edges each node has. An `in_degree` of `0` means the task has no prerequisites.
*   **`nodes`**: A set containing every unique task mentioned.

#### 2. `solve(edges)` — **Kahn's Algorithm**
This function finds **one single valid order** of execution. It uses **Kahn's Algorithm**, which is a Breadth-First Search (BFS) approach:
1.  **Initialize**: Find all nodes with an `in_degree` of `0` (tasks with no prerequisites) and put them in a `queue`.
2.  **Process**: While the queue isn't empty:
    *   Remove a task from the queue and add it to the `result` list.
    *   For every "neighbor" (tasks that depend on the current task), decrement their `in_degree` by 1.
    *   If a neighbor's `in_degree` hits `0`, it means all its prerequisites are finished; add it to the `queue`.
3.  **Cycle Detection**: If the number of tasks in the `result` is less than the total number of nodes, it means there is a **circular dependency** (e.g., A depends on B, and B depends on A), making a valid order impossible. It returns `None`.

#### 3. `find_all(edges)` — **Backtracking**
While `solve` finds one path, this function finds **every possible valid permutation** of tasks. It uses a **Backtracking (Depth-First Search)** algorithm:
1.  It looks for all tasks currently available (those with an `in_degree` of `0` that haven't been used yet).
2.  It picks one available task, "executes" it by temporarily reducing the in-degrees of its neighbors, and then calls itself recursively to pick the next task.
3.  **Backtrack**: Once a full path is found or a dead end is reached, it "undoes" the last step (restores the in-degrees and removes the task from the path) to try a different available task at that same step.

### The Algorithm Summary

| Feature | `solve()` | `find_all()` |
| :--- | :--- | :--- |
| **Algorithm** | **Kahn's Algorithm** (BFS-based) | **Backtracking** (DFS-based) |
| **Goal** | Find **one** valid topological sort. | Find **all** valid topological sorts. |
| **Time Complexity** | $O(V + E)$ (Efficient) | $O(V!)$ (Exponential/Brute Force) |
| **Use Case** | Real-world task scheduling (e.g., build systems). | Exhaustive analysis of all possible sequences. |

*(Where $V$ is the number of vertices/nodes and $E$ is the number of edges/dependencies.)*

### Example Execution Trace
Given the input:
`("lint", "compile"), ("deps", "compile"), ("compile", "link")...`

1.  **Initial state**: `lint` and `deps` have 0 prerequisites.
2.  **One valid order (`solve`)**: Might pick `deps` $\to$ `lint` $\to$ `compile` $\to$ `test` $\to$ `link` $\to$ `package`.
3.  **All valid orders (`find_all`)**: It will explore every combination, such as:
    *   `deps` $\to$ `lint` $\to$ `compile` $\to$ `test` $\to$ `link` $\to$ `package`
    *   `lint` $\to$ `deps` $\to$ `compile` $\to$ `test` $\to$ `link` $\to$ `package`
    *   ...and so on, accounting for the fact that `test` and `link` can swap places because they don't depend on each other.
