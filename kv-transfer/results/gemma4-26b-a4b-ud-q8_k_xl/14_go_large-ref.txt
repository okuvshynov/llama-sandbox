Go distributed task scheduler (though the implementation provided is actually a single-process, multi-goroutine scheduler—the "distributed" part is implied by the architecture like heartbeats and worker IDs).
Identify bugs, race conditions, performance issues, and design problems.
Explain issues and suggest fixes.

    *   `scheduler.go`: Core logic (priority queue, task management, worker registration, dependency management).
    *   `ratelimiter.go`: Token bucket and sliding window implementations.
    *   `circuitbreaker.go`: Circuit breaker implementation.
    *   `workerpool.go`: Worker management, scaling, and the actual execution loop.

    **File 1: `scheduler.go`**
    *   *Priority Queue (`TaskQueue`):* Uses `container/heap`. `Less` uses `CreatedAt.Before` for stability, which is good. `Pop` sets `index = -1`, good.
    *   *`SubmitTask`:*
        *   If task has dependencies, it adds to `depGraph` and checks `completed`.
        *   *Issue:* If a task is submitted with dependencies that aren't completed, it's added to `s.tasks` but *not* to `s.queue`. This is correct.
        *   *Issue:* `s.depGraph[depID] = append(s.depGraph[depID], task.ID)`. This builds a reverse mapping (dependency -> dependents).
        *   *Potential Issue:* `s.tasks[task.ID] = task` happens *before* the dependency check returns. If it returns early, the task is in `tasks` but not `queue`. That's fine.
        *   *Race/Logic:* If `task.ID` is empty, it generates one. Good.
    *   *`AssignTask`:*
        *   Uses a recursive call: `return s.AssignTask(workerID)` if task is cancelled.
        *   *Bug/Risk:* If there are many cancelled tasks in a row at the top of the heap, this could cause a stack overflow (unlikely but possible) or, more likely, it holds the lock `s.mu.Lock()` for a long time during recursion. *Correction:* It's a recursive call while holding the lock. This is actually fine for a few items, but if the queue is 100% cancelled tasks, it's a loop. It's better to use a loop.
    *   *`CompleteTask`:*
        *   *Crucial Bug:* `go func() { ... s.mu.Lock() ... }()` for retries. This goroutine accesses `task` without ensuring the task object isn't being modified elsewhere, though since it's a pointer and the scheduler owns it, it's mostly okay.
        *   *Race Condition:* `task.State = TaskRetrying` is set inside the lock, but then the goroutine sleeps. When it wakes up, it locks and pushes to the heap.
        *   *Bug:* `atomic.AddInt64(&worker.TasksFailed, 1)` is called *outside* the lock if `worker != nil`. While `TasksFailed` is an `int64` and uses `atomic`, the `worker` object itself is part of the `s.workers` map. Accessing `worker` needs to be safe.
        *   *Bug:* `if dependents, ok := s.depGraph[taskID]; ok && task.State == TaskCompleted { ... }`. This logic to unblock dependents is correct in theory.
    *   *`checkWorkerHealth`:*
        *   *Bug:* It pushes the task back to the queue: `heap.Push(&s.queue, task)`. But it doesn't check if the task is already being retried or is in a terminal state.
        *   *Bug:* If a worker is marked dead, and the task is re-queued, the `worker.Load` is never decremented. The dead worker's load remains high (though it's dead, so it doesn't matter much), but the *next* worker taking the task will increment its load. This is fine, but the "currentTask" logic might be messy.
    *   *`Stats`:* `stats["active_workers"] = stats["active_workers"].(int) + 1`. This is a type assertion on an interface that was just initialized as an `int`. Safe, but slightly clunky.

    **File 2: `ratelimiter.go`**
    *   *`TokenBucketLimiter`:*
        *   *Race Condition:* `getBucket` is called inside `AllowN` which holds `tbl.mu`. However, `getBucket` *writes* to the map. This is protected by the lock.
        *   *Performance:* Using a single mutex for *all* keys in a rate limiter is a massive bottleneck in a high-throughput distributed system.
        *   *Cleanup:* `cleanupLoop` runs every 10 mins.
    *   *`SlidingWindowLimiter`:*
        *   *Performance:* `valid := make([]time.Time, 0, len(entry.timestamps))` inside `AllowN`. This creates a new slice and copies elements every single request. This is $O(N)$ per request where $N$ is the number of requests in the window. This will kill performance under high load.
        *   *Memory Leak:* If many keys are used once and never again, they only get cleaned up by the `cleanupLoop`.

    **File 3: `circuitbreaker.go`**
    *   *`CircuitBreaker`:*
        *   *Logic Error:* `canExecute` calls `setState(CircuitHalfOpen)`. `setState` resets `failureCount` and `successCount`.
        *   *Logic Error:* In `HalfOpen` state, it allows `halfOpenMaxConcurrent` (set to 1).
        *   *Issue:* `Execute` calls `canExecute` (locks/unlocks), then `fn()` (no lock), then `recordResult` (locks/unlocks). This is actually a correct pattern to avoid holding the lock during the long-running `fn()`.
    *   *`CircuitBreakerRegistry`:*
        *   *Pattern:* Standard registry pattern. Looks okay.

    **File 4: `workerpool.go`**
    *   *`WorkerPool`:*
        *   *Race Condition:* `p.addWorker()` calls `p.scheduler.RegisterWorker`. `RegisterWorker` takes a lock. `addWorker` also takes a lock.
        *   *Bug:* `removeWorker` iterates through workers and calls `idlest.Stop()`. `idlest.Stop()` calls `w.cancel()`.
        *   *Bug:* `autoScale` uses `p.scheduler.Stats()`. `Stats()` is quite expensive (iterates all workers).
        *   *Race Condition:* `p.workers` is a slice. `addWorker` appends to it. `removeWorker` modifies it via slicing. `autoScale` and `logMetrics` iterate over it. While `p.mu` is used, `removeWorker` is called inside `autoScale` which holds `p.mu` (via `p.mu.Lock()` in `removeWorker`). Wait, `autoScale` calls `p.mu.RLock()`. Then `removeWorker` tries to call `p.mu.Lock()`. **Deadlock!**
    *   *`Worker.Run`:*
        *   *Infinite Loop:* If `handler` fails or is missing, it calls `CompleteTask`.
        *   *Bug/Race:* `w.pool.scheduler.handlers[task.Name]` is accessed without a lock. `handlers` is a map in `Scheduler`. Maps are not thread-safe for concurrent read/write. While `RegisterHandler` uses a lock, `Run` does not.
        *   *Performance/Busy Wait:* If `AssignTask` returns nil (no tasks), it sleeps 100ms. This is "okay" but not efficient. A condition variable or channel would be better.
