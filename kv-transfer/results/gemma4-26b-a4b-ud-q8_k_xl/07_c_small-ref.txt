### 1. Memory Leaks

**A. In `table_put` (Updating an existing key)**
When the function finds an existing key, it updates the value using `strdup`:
```c
if (strcmp(t->entries[i].key, key) == 0) {
    t->entries[i].value = strdup(value); // BUG: Old value is lost
    return;
}
```
The pointer to the original string stored in `t->entries[i].value` is overwritten by the new pointer returned by `strdup`. Since `free()` was never called on the old pointer, that memory becomes unreachable, causing a leak every time a value is updated.

**B. In `table_put` (Reallocating memory)**
The code uses `realloc` incorrectly:
```c
t->entries = realloc(t->entries, sizeof(Entry) * t->capacity);
```
If `realloc` fails, it returns `NULL`. By assigning the result directly back to `t->entries`, you overwrite the only pointer you had to the existing memory block. This makes the original memory impossible to free, resulting in a leak.

**C. In `table_delete` (The swap-and-pop method)**
The deletion logic uses a common optimization to avoid shifting elements:
```c
t->entries[i] = t->entries[t->size - 1];
t->size--;
```
While this is efficient for the array, the `Entry` being "removed" (the one at index `i`) is overwritten by the last element. Because the `key` and `value` of the removed entry were allocated via `strdup`, they are never `free()`'d. The memory they pointed to is leaked.
