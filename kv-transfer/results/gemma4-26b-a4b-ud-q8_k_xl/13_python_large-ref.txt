### 1. Critical: Data Integrity & Financial Accuracy

#### **A. Floating Point Math for Currency**
*   **Issue:** Throughout `models.py` and `order_processor.py`, you use `float` for prices, taxes, and totals. Floating-point math (IEEE 754) leads to rounding errors (e.g., `0.1 + 0.2 != 0.3`). In a financial system, these tiny discrepancies accumulate and cause audit failures.
*   **Fix:** Use the `decimal.Decimal` class for all monetary calculations.

#### **B. Non-Atomic Transactions (The "Dual-Write" Problem)**
*   **Issue:** In `OrderProcessor.process_next`, the code performs three distinct side effects: `payment.charge()`, `inventory.commit_reservation()`, and `email.send_order_confirmation()`.
    *   If the service crashes *after* the payment is charged but *before* the inventory is committed, the customer is charged, but the order is never fulfilled, and stock is never deducted.
*   **Fix:** Implement the **Saga Pattern** or a **Transactional Outbox**. Use a database that supports ACID transactions to record the intent to charge/commit, and use a background worker to ensure eventual consistency.

#### **C. Incorrect Tax Logic**
*   **Issue:** In `Order.tax()`, the tax is calculated on `self.subtotal()`. However, in `Order.total()`, a discount is applied to the subtotal. Most tax jurisdictions require tax to be calculated on the **net price** (after discounts), not the gross price.
*   **Fix:** Change `tax()` to use the discounted subtotal.

### 2. High: Scalability & Distributed Systems Design

#### **D. Statefulness in a Stateless Microservice**
*   **Issue:** The service is heavily "stateful." `InventoryManager` uses a local `json` file, `PaymentGateway` uses an in-memory `dict`, and `AnalyticsEngine` stores all events in a Python `list`.
    *   **Problem:** If you run two instances of this microservice behind a load balancer, they will not share inventory, they will lose all transaction history on restart, and analytics will be fragmented.
*   **Fix:** Move all state to external persistent storage:
    *   **Inventory/Orders:** Use a relational database (PostgreSQL).
    *   **Analytics:** Use a Time-Series Database (InfluxDB) or a specialized OLAP store.
    *   **Caching/Locks:** Use Redis for distributed locking and reservations.
