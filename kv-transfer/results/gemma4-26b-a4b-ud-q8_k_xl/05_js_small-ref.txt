*   `constructor`: Initializes `events` (array) and `listeners` (object).
*   `schedule(name, time, callback)`: Adds an event and sorts the array by time.
*   `on(eventName, handler)`: Subscribes a handler to an event.
*   `off(eventName, handler)`: Unsubscribes a handler.
*   `run()`: An `async` method that iterates through events, calculates wait time based on `Date.now()` and a start time, waits using `setTimeout`, then executes the callback and triggers listeners.
*   `cancel(name)`: Removes events by name.
*   `reschedule(name, newTime)`: Updates the time of an event.

    *   *Issue 1: `reschedule` breaks the sort order.*
        *   `schedule` sorts the array.
        *   `reschedule` changes a time but doesn't re-sort the array.
        *   If I reschedule a later event to an earlier time, the `run()` loop will process events in the old (wrong) order.

    *   *Issue 2: `run()` logic - `event.time` vs absolute time vs relative time.*
        *   `const elapsed = Date.now() - startTime;`
        *   `const wait = event.time - elapsed;`
        *   This implies `event.time` is a *relative offset from the moment `run()` was called*.
        *   However, `schedule` allows adding events at any time. If I schedule an event for 1000ms, then call `run()` 5 seconds later, `elapsed` starts at 0, so it waits 1000ms. This is okay *if* the intention is relative time.
        *   *Wait, there's a subtle bug:* If I call `run()` once, it processes all events. If I call `run()` again, it starts `startTime` from zero again. This might be intended, but usually, a scheduler handles absolute time or a continuous timeline. Let's look closer at the loop.

    *   *Issue 3: Execution order and Time Drifts in `run()`.*
        *   The loop is: `wait -> execute -> notify listeners`.
        *   If `handler()` or `event.callback()` is slow or asynchronous (though they aren't awaited here), the `elapsed` calculation for the *next* event will include that time. This is actually correct for a sequential scheduler.
        *   *Wait, a bigger bug:* `event.callback()` is not `await`ed. If a callback is async, the loop continues immediately. This might be fine, but the `run` method is `async`, suggesting it should handle async tasks.

    *   *Issue 4: The `reschedule` equality check.*
        *   `if (event.name == name)` uses loose equality. While not strictly a bug in all JS contexts, it's bad practice. More importantly, it updates *all* events with that name. If there are multiple events named "backup", they all change. This might be intended or a bug depending on requirements.

    *   *Issue 5: Concurrency/Race Conditions in `run()`.*
        *   `run()` is an async function. If someone calls `run()` twice, two loops will start. Both will iterate over `this.events`. This will cause events to fire twice. There is no state to check if the scheduler is already running.

    *   *Issue 6: Modifying the array while iterating (Edge Case).*
        *   If a callback inside `run()` calls `cancel()` or `schedule()`, the loop in `run()` might behave unpredictably (though `for...of` on an array handles this reasonably well in JS, it's still risky).

    *   *Issue 7: The "Time Drift" logic in `run()` (The most significant logic bug).*
        *   `const wait = event.time - elapsed;`
        *   Let's trace:
            1. Event A (time 1000). `elapsed` = 0. `wait` = 1000. `setTimeout` 1000ms.
            2. Event A callback runs. It takes 500ms.
            3. Loop moves to Event B (time 2000).
            4. `elapsed` is now 1500ms.
            5. `wait = 2000 - 1500 = 500ms`.
            6. `setTimeout` 500ms.
            7. Event B fires at 1000 + 500 + 500 = 2000ms.
        *   Wait, the math actually works out to keep things on schedule *if* the intention is that `event.time` is the timestamp relative to the start of `run()`.
        *   *However*, if `event.time` is actually 500ms and `elapsed` is 1000ms, `wait` is -500. The code says `if (wait > 0)`. So it executes immediately. That's correct.

    *   *Issue 8: `reschedule` and `schedule` name collisions.*
        *   `cancel(name)` and `reschedule(name)` use the `name` property. If multiple events have the same name, `cancel` deletes all of them. `reschedule` updates all of them. This is a design flaw if names aren't unique IDs.

    *   *Issue 9: The `on`/`off` listener leak/error.*
        *   If `off` is called with a handler that doesn't exist, it handles it fine.
        *   If `on` is called with something that isn't a function, it'll crash later. (Standard JS behavior).

    *   *Bug 1: Sorting in `reschedule`.* (Crucial)
    *   *Bug 2: Multiple `run()` calls.* (Crucial)
    *   *Bug 3: Event name ambiguity.* (Design/Logic)
    *   *Bug 4: Async callbacks not awaited.* (Functional)

    *   *Refining Bug 1 (The Sorting Issue):* `reschedule` changes the `time` property, but the `events` array is only sorted inside `schedule`. If an event's time is moved from 5000 to 100, the `run` loop (which is already iterating or about to iterate) will see the array in the wrong order.

    *   *Refining Bug 2 (The `run()` Re-entrancy):* There's no protection against calling `run()` multiple times. If you call `run()` twice, every event triggers twice.

    *   *Refining Bug 3 (The Callback/Listener issue):* `event.callback()` is called, then listeners are called. If `event.callback` is an `async` function, the loop doesn't wait for it. The "all events done" log might trigger while callbacks are still running.

    *   *Refining Bug 4 (The `cancel`/`reschedule` name issue):* The scheduler uses `name` as a unique identifier for `cancel` and `reschedule`, but `schedule` allows duplicate names. This makes the behavior of `cancel` and `reschedule` unpredictable (it acts on all events with that name).

    I `reschedule('B', 5)`.
    The array is now `[{name: 'A', time: 10}, {name: 'B', time: 5}]`.

    Loop 1: Event 'A'. Wait 10ms.

    Result: 'A' fired at 10ms, 'B' fired at 10ms.
    Wait... 'B' was supposed to be at 5ms. So it fired late. The order was wrong.
