        *   *Logic Check:* Bump allocator. Uses `std::align`.
        *   *Bug 1:* `allocate` logic. If `std::align` fails, it calls `grow()`. Then it recalculates `space` and `ptr`. But it doesn't check if `std::align` succeeds a *second* time. Also, the update logic `offset_ = block_size_ - space + size;` is risky if `std::align` modifies `ptr` and `space`.
        *   *Bug 2:* `grow()` adds a new block but `current_block_` is updated. However, `offset_` is reset to 0 in `grow()`. This is fine, but the `allocate` function needs to handle the case where a single allocation is larger than `block_size_`.
        *   *Bug 3:* `reset()` is dangerous. It deletes all blocks except the last one. It doesn't call destructors of the objects created in the arena. This is a design choice for a bump allocator, but the user needs to be aware.
        *   *Bug 4 (Critical):* Memory Leak / UB in `reset()`. `blocks_ = { blocks_.back() };` replaces the vector. If `blocks_` had multiple elements, it deletes them all except the last. This part is okay.
        *   *Bug 5 (Performance):* `allocate` calls `grow()` which allocates a new large block. But what if the request is larger than `block_size_`? It will fail or cause UB because `std::align` won't work on the existing block.
        *   *Bug 6:* `total_allocated_` doesn't account for alignment padding.

        *   *Logic Check:* Type erasure using a `Concept`/`Model` pattern (similar to `std::function`).
        *   *Observation:* It's a manual implementation of a simplified `std::function`.
        *   *Issue:* No support for objects that aren't copy-constructible? `Model` takes `F&&`. But `Task` doesn't have a copy constructor, only move. This is actually okay.
        *   *Issue:* `Task` is missing a copy constructor, but the `TaskQueue` uses `std::queue<Task>`, which requires move/copy. The implementation provided only has move, so `Task` is move-only. This is fine.

        *   *Logic Check:* Mutex-protected queue.
        *   *Bug 1:* `wait_and_pop` condition: `cv.wait(lock, [this] { return !queue_.empty() || done_; });`. If `done_` is true and the queue is empty, it returns `false`. This is correct for shutdown.
        *   *Performance:* Single mutex for the whole queue. High contention if many threads are pushing/popping.

        *   *Logic Check:* Work-stealing via multiple queues.
        *   *Bug 1 (Critical - Race Condition/Deadlock):* `worker_loop` logic.
            *   Scenario: `try_pop` fails -> goes to `else` -> loops through others. If none found, it calls `queues[idx].wait_and_pop(task)`.
            *   If `idx` is waiting on its CV, but a task is pushed to `idx+1`, the worker for `idx` won't see it until it's woken up or it tries to steal.
            *   Wait... if `wait_and_pop` is called, the thread sleeps. If a task is pushed to `queues[idx]`, it wakes up. But if a task is pushed to `queues[idx+1]`, the thread in `idx` is still sleeping! This defeats the purpose of work stealing for sleeping threads.
        *   *Bug 2 (Deadlock potential/Logic error):* `pending_tasks_` management. The `worker_loop` does: `try_pop` -> `pending_tasks.fetch_add(1)`. This is actually okay, but it's slightly weird to increment *after* popping.
        *   *Bug 3 (Wait_all):* `wait_all` uses a busy-wait loop with `std::this_thread::yield()`. This is inefficient.
        *   *Bug 4 (Performance):* `submit` uses `next_queue_++ % queues_.size()`. This spreads tasks, but not necessarily locally.

        *   *Logic Check:* Parallel merge sort.
        *   *Bug 1 (Algorithmic/Correctness):* The `merge` function logic: `while (i <= mid) dst[k++] = src[i++];`. If `mid` is the boundary, and `i` starts at `left`, the loop should be `i < mid`. If `mid` is included, it will access out of bounds or duplicate elements.
        *   *Bug 2 (Data Race):* `parallel_sort_impl` calls `pool.submit` with `[=, &pool]`. The `data` and `buf` pointers are captured by value, but they point to the same memory. The recursion depth is limited. However, the `merge` step uses `std::memcpy(data + left, buf + left, ...)`. This part is okay, but the merge function itself must be correct.
        *   *Bug 3 (Logical Error):* `merge(data, buf, left, mid, right)`. The parameters are `src`, `dst`, `left`, `mid`, `right`. The `merge` implementation: `while (i < mid && j < right)`. This is standard. But `while (i <= mid)` is wrong. It should be `i < mid`.
        *   *Bug 4 (Performance/Memory):* Creating a new `MemoryArena` for every `parallel_merge_sort` call. Inside the recursion, we don't use the arena, we use the `buf` passed down. That's good.

        *   *Logic Check:* Linear probing hash map with a single mutex.
        *   *Bug 1 (Performance - Major):* It's a "Concurrent" hash map that uses a **single global mutex** for every operation (`insert`, `find`, `erase`). This is not concurrent; it's just thread-safe. It will be a massive bottleneck.
        *   *Bug 2 (Correctness - Erasure):* In linear probing, you cannot simply set `occupied = false` during `erase`. This breaks the probe chain. You must use "tombstones" or rehash the cluster.
        *   *Bug 3 (Correctness - Rehash):* During `rehash`, `size_` is reset to 0 and incremented inside `insert_into`. This is okay, but the loop `for (size_t i = 0; i < capacity_; ++i)` is used. This is fine.
