### 1. Soundness Issues (Correctness & Safety)

#### A. Panic on NaN (Floating Point Comparison)
In the `percentile` method, you use:
```rust
sorted.sort_by(|a, b| a.partial_cmp(b).unwrap());
```
**The Issue:** In Rust, `f64` does not implement `Ord` because `NaN` values cannot be compared. If a `NaN` (Not-a-Number) value is ever recorded in a metric (e.g., due to a division by zero in the calling code), `partial_cmp` will return `None`, and `.unwrap()` will cause the entire thread to **panic**.
**The Fix:** Use `total_cmp` (available since Rust 1.62) which provides a total ordering for floats, or explicitly handle `NaN`.

#### B. Panic in `drop` (Double Panic)
The `Timer` struct records metrics inside its `drop` implementation.
**The Issue:** If the `MetricsRegistry` or the underlying `Mutex` is poisoned (meaning a previous thread panicked while holding the lock), `metric.lock().unwrap()` will panic. If a panic occurs during the unwinding of another panic (which happens when a `drop` panics), the Rust runtime will immediately **abort the process**.
**The Fix:** Avoid using `.unwrap()` inside `drop`. Use `if let Ok(m) = metric.lock() { ... }` to fail gracefully.

### 2. Performance Problems

#### A. Unbounded Memory Growth (Memory Leak)
The `Metric` struct stores every single value and timestamp in `Vec<f64>` and `Vec<Instant>`.
**The Issue:** While you implemented `trim_older_than`, **it is never called anywhere in the code.** In a long-running production service, these vectors will grow indefinitely until the system runs out of RAM and the OOM (Out of Memory) killer terminates the process.
**The Fix:** The `MetricsRegistry` or a background worker thread must periodically call `trim_older_than` on all metrics.
