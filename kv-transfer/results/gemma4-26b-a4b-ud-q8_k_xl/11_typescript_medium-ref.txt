    *   `TypedEventEmitter`: A generic event emitter.
    *   `Task` & `PipelineEvents`: Interfaces/Types for the pipeline.
    *   `Pipeline`: The core orchestrator.

    *   *Class: `TypedEventEmitter`*
        *   `listeners` and `onceListeners` use `Set<Function>`. This loses type safety. `Function` is too broad.
        *   `once` implementation: It adds the listener to `onceListeners`, but `emit` only clears the *whole* set for that event. If there are multiple `once` listeners for the same event, they all fire once, then they are all wiped. This is technically correct for "once", but usually, `once` implies the individual listener is removed after its specific invocation.
        *   `emit`: `fn(...args)` is called. Since `fn` is typed as `Function`, TypeScript doesn't know if `args` matches the signature of `fn`.
        *   `once` bug: If I call `once('event', fn1)` and `once('event', fn2)`, and then `emit('event')`, both fire. Then `onceSet.clear()` clears both. That's actually okay, but if I call `on('event', fn3)` and `once('event', fn1)`, `fn3` will *not* be in `onceListeners`, but `fn1` *is*. The current implementation of `emit` runs `listeners` (permanent) AND `onceListeners` (one-time). This seems okay, *except* `onceListeners` are never integrated into the normal execution flow until `emit` is called. Wait, let's re-read:
            *   `on` adds to `listeners`.
            *   `once` adds to `onceListeners`.
            *   `emit` runs `listeners`, then runs `onceListeners`, then clears `onceListeners`.
            *   *Issue:* If I call `on` for an event, then `once` for the same event, `emit` runs the `on` listeners and then the `once` listeners. This works.
            *   *Bug:* `off` tries to remove from both. This is fine.
            *   *Architectural Issue:* `Function` type is bad. Use `(...args: any[]) => void`.

    *   *Interface: `Task`*
        *   `T = any` is a red flag.
        *   `result?: T` and `error?: Error`.
        *   `dependencies: string[]`. This implies we need to look up tasks by ID.

    *   *Interface: `PipelineEvents`*
        *   Extends `EventMap`.
        *   The signatures look okay, but `task:complete` uses `result: any`. This breaks the "Typed" promise of the emitter.

    *   *Class: `Pipeline`*
        *   `private results: Map<string, any>`. The `any` is problematic.
        *   `addTask`: Returns `this`. Good for chaining.
        *   `getReadyTasks`: Iterates through all tasks every time. $O(N)$ per poll.
        *   `executeTask`:
            *   `Promise.race` with a timeout. This is a common pattern but has a "dangling promise" issue. If the task finishes before the timeout, the `setTimeout` timer is still running in the background.
            *   `task.status = 'running'`.
            *   `this.emit('task:retry', task, attempt)`.
            *   `this.emit('task:complete', task, result)`.
        *   `run`:
            *   `while(true)` loop with `await new Promise(r => setTimeout(r, 100))`. This is a "busy wait" or "polling" approach. It's inefficient. A better way is to trigger a check whenever a task completes.
            *   `const batch = ready.slice(0, this.concurrency); await Promise.all(batch.map(...))`. **Major Logic Bug Alert!** `await Promise.all(batch...)` means the loop waits for the *entire batch* to finish before starting the next batch. If concurrency is 4 and 3 tasks take 1s and 1 takes 10s, the 3 slots sit idle for 9 seconds. This defeats the purpose of concurrency.
            *   `Deadlock` detection: It detects if tasks are pending but nothing is running. Good.
        *   `cancel`: Sets status to `cancelled`. But `executeTask` doesn't check `task.status` between retries or during execution. The `execute` function itself isn't cancelled. It's just "fire and forget" cancellation.

    *   *Usage Example (`main`):*
        *   `const users = pipeline['results'].get('fetch-users');`. **Major Type Safety Violation.** Accessing a private member using string indexing. This is a huge smell. The tasks should probably be able to access results via a typed way or pass them in.

    *   **Issue 1: Concurrency Logic (The "Batching" Problem).**
        *   `await Promise.all(batch.map(...))` blocks the loop.
        *   *Fix:* Use a worker pool pattern or a semaphore. Or, instead of a `while` loop with `setTimeout`, trigger the next task immediately when a task finishes.

    *   **Issue 2: Event Emitter Type Safety.**
        *   `Set<Function>` is bad.
        *   The `once` implementation is slightly unconventional but mostly works; however, the `emit` logic is better handled by wrapping the listener.
        *   *Fix:* Use `(...args: any[]) => void` and wrap `once` listeners in a wrapper that calls `off`.

    *   **Issue 3: The "Dangling Timeout" in `executeTask`.**
        *   `Promise.race` with `setTimeout` leaves a timer running.
        *   *Fix:* Use `AbortController` to cancel the timeout and/or the task if supported.

    *   **Issue 4: Private Access in Usage.**
        *   `pipeline['results']` is a hack.
        *   *Fix:* Provide a `getResult<K>(id: string): T` method or make the `execute` function receive the results of its dependencies.

    *   **Issue 5: Polling in `run()`.**
        *   `while(true)` + `setTimeout(100)` is inefficient.
        *   *Fix:* Event-driven execution. When a task finishes, call `this.processNext()`.

    *   **Issue 6: Weak Typing in `PipelineEvents`.**
        *   `task:complete` uses `any`.
        *   *Fix:* Use generics in `Pipeline` to track task types if possible, or at least more specific types.

    *   **Issue 7: Cancellation Implementation.**
        *   Setting `task.status = 'cancelled'` doesn't stop the `execute()` promise.
        *   *Fix:* Use `AbortSignal`.

    *   *Refactoring `TypedEventEmitter`:*
        ```typescript
        on<K extends keyof T>(event: K, listener: T[K]): () => void {
          if (!this.listeners.has(event)) this.listeners.set(event, new Set());
          this.listeners.get(event)!.add(listener as any);
          return () => this.off(event, listener);
        }
        // ... rewrite once to wrap the function
        ```

    *   *Refactoring `Pipeline.run` (The core engine):*
        Instead of `while(true)`, use a queue and a counter for `runningTasks`.

    *   *Addressing the `results` access:*
        The `execute` function should probably receive a `context` object: `execute: (ctx: PipelineContext) => Promise<T>`.
