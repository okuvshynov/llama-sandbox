### 1. Data Races (Critical)
**The Problem:**
While the `WorkerPool` uses a mutex to protect the `results` map, it does **not** protect the individual `Job` objects stored within that map.
* **Race Condition:** The `processJob` method (running in worker goroutines) modifies `job.Status`, `job.Result`, and `job.Retries`. Simultaneously, the HTTP handler `/status` or `/stats` reads these same fields via `GetJob` or by iterating over the map.
* **Impact:** This will trigger the Go Race Detector and can lead to unpredictable behavior or crashes when reading/writing strings and interfaces.

**The Fix:**
Add a `sync.RWMutex` to the `Job` struct itself to protect its internal fields.

```go
type Job struct {
    mu        sync.RWMutex // Protects the fields below
    ID        string
    Status    string
    // ... other fields
}

// When updating:
job.mu.Lock()
job.Status = "completed"
job.mu.Unlock()

// When reading:
job.mu.RLock()
status := job.Status
job.mu.RUnlock()
```
